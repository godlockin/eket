# PR Review Checklist

**PR**: `${PR_NUMBER}` - ${PR_TITLE}
**提交者**: ${SUBMITTER_ID}
**Reviewer**: ${REVIEWER_ID}
**Review 时间**: ${REVIEW_DATE}

---

## 1. 代码质量检查

### 1.1 代码规范
- [ ] 遵循项目代码风格
- [ ] 变量/函数命名清晰
- [ ] 代码格式正确（通过 linter）
- [ ] 无未使用的导入/变量

### 1.2 代码设计
- [ ] 函数/方法职责单一
- [ ] 代码复用合理
- [ ] 无过度设计
- [ ] 遵循 SOLID 原则

### 1.3 错误处理
- [ ] 适当的错误处理
- [ ] 错误信息清晰有用
- [ ] 无吞掉异常的情况
- [ ] 边界条件处理正确

---

## 2. 功能正确性检查

### 2.1 需求对齐
- [ ] 实现了 ticket 要求的所有功能
- [ ] 符合验收标准
- [ ] 与设计方案一致

### 2.2 逻辑正确
- [ ] 业务逻辑正确
- [ ] 边界条件处理
- [ ] 并发/竞态条件处理
- [ ] 事务/数据一致性

### 2.3 测试覆盖
- [ ] 单元测试已添加
- [ ] 测试覆盖关键路径
- [ ] 测试用例命名清晰
- [ ] 所有测试通过

---

## 3. 安全性检查

### 3.1 输入验证
- [ ] 用户输入已验证
- [ ] SQL 注入防护
- [ ] XSS 防护
- [ ] 命令注入防护

### 3.2 认证授权
- [ ] 认证逻辑正确
- [ ] 授权检查完整
- [ ] 敏感操作有权限验证

### 3.3 数据安全
- [ ] 敏感数据加密存储
- [ ] 日志中无敏感信息
- [ ] API 密钥/密码未硬编码

---

## 4. 性能检查

### 4.1 算法复杂度
- [ ] 无明显性能问题
- [ ] 循环/递归有终止条件
- [ ] 大数据量场景考虑

### 4.2 资源管理
- [ ] 数据库连接正确管理
- [ ] 文件/网络连接及时关闭
- [ ] 内存泄漏风险排查

### 4.3 缓存使用
- [ ] 合理使用缓存
- [ ] 缓存失效策略正确
- [ ] 缓存穿透/雪崩防护

---

## 5. 文档检查

### 5.1 代码注释
- [ ] 复杂逻辑有注释
- [ ] 公共 API 有文档
- [ ] 注释准确及时

### 5.2 更新文档
- [ ] README 已更新（如需要）
- [ ] API 文档已更新
- [ ] 变更日志已更新

---

## 6. Git 规范检查

### 6.1 提交信息
- [ ] 提交信息符合 Conventional Commits
- [ ] 提交信息清晰描述变更
- [ ] 无敏感信息在提交信息中

### 6.2 分支管理
- [ ] 分支命名正确
- [ ] 提交历史清晰
- [ ] 无不必要的合并提交

---

## 7. Review 结果

### 7.1 总体评价
- [ ] **Approved** - 代码质量良好，可以合并
- [ ] **Changes Requested** - 需要修改后重新 Review
- [ ] **Commented** - 有建议但不阻塞合并
