Ticket 职责规范

版本: v2.9.0-alpha
创建日期: 2026-04-10
目的: 明确定义 Master 和 Slaver 在 Ticket 中的职责边界，禁止越权操作

1. 核心原则

Master 只写文档，Slaver 只写代码。

角色 唯一产出物 禁止操作
Master 需求文档、架构文档、Ticket、PR 审查报告、Slaver 团队初始化 ❌ 不得写任何代码、配置、测试
Slaver 代码、测试、PR、分析报告 ❌ 不得创建/修改 Ticket 元数据、不得审查自己的 PR

关键流程规则：

Master 在创建 Ticket 后，必须立即初始化 Slaver 团队，将任务状态设为 ready，让 Slaver 可立即领取。禁止创建任务后不初始化执行团队，导致任务积压在 backlog 或 analysis 状态。

Nyquist Rule（验收标准自动化要求）

每条验收标准必须满足： 1. 可自动化：附带具体的 shell 命令（而非手动运行/人眼确认） 2. 有时限：命令在 60 秒内完成（长时间运行的集成测试需拆分） 3. 客观可重复：相同代码 + 相同命令 = 相同结果（无随机性） 4. 覆盖质量四件套：命令必须涵盖该技术栈的类型检查 + lint + test + 安全扫描（可用 scripts/detect-stack.sh 查看当前栈要求）

违反此规则的 PR 描述（仅贴截图/文字描述而非命令输出）视同未完成验收。

3. 禁止包含的内容

3.1 Ticket 中禁止的内容

内容 原因 正确位置
❌ 完整源代码 Ticket 是任务追踪，不是代码仓库 代码文件
❌ 敏感信息（密码、API Key） 安全风险 环境变量/密钥管理
❌ 人身攻击或负面情绪 保持专业 -
❌ 模糊的验收标准 必须可验证 验收标准字段
❌ 未授权的依赖变更 需 Master 审批 依赖关系字段

3.2 Master 禁止在 Ticket 中写的

内容 原因
❌ 代码实现细节 这是 Slaver 的职责
❌ 测试代码 这是 Slaver 的职责
❌ 配置文件内容 这是 Slaver 的职责
❌ “我来实现这个功能” Master 禁止写代码

3.3 Slaver 禁止在 Ticket 中写的

内容 原因
❌ 修改验收标准 这是 Master 和人类确认的合同
❌ 修改优先级/重要性 这是 Master 的决策
❌ 自行添加依赖 需 Master 审批架构影响
❌ Review 自己的 PR 利益冲突

5. 违规处理

违规行为 处理措施
Master 写代码 立即停止，删除代码，重新分配给 Slaver
Slaver 修改验收标准 恢复原状，通知 Master 重新确认
Slaver 审查自己的 PR Review 无效，重新分配 Master 审查
Slaver 跳过分析报告直接编码 停止编码，补写分析报告，等待批准
Master 代填执行记录 删除记录，由 Slaver 重新填写

Slaver 执行报告 Schema（Artifact Schema v1）

Slaver 在 Ticket 的「执行记录」部分填写时，必须包含以下结构化字段：

分析报告（analysis → ready 前必填）

analysis_report:
  summary: "一句话描述方案"
  approach: "技术方案关键点（1-3 点）"
  risks: "已识别风险（无则填 none）"
  estimated_hours: N  # 预估工时（小时）
  blocked_by: "依赖的 ticket ID 或 none"

实现报告（in_progress → test 前必填）

implementation_report:
  files_changed:
    - "path/to/file1.ts (新增/修改/删除)"
    - "path/to/file2.ts (新增/修改/删除)"
  test_result: "X/Y tests passing"  # 必须是真实命令输出
  pr_link: "https://github.com/..."
  notes: "实现过程中的重要决策（可选）"
