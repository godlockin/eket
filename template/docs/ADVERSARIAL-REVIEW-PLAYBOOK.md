# 对抗式验收 Review 机制

> EPIC 开发完成进入验收测试时执行

## 核心原则

**持续自问**：我有 100% 信心交付和部署上线吗？

如果没有 → 继续检查、测试、修复、迭代升级

## AB 对抗组设置

### A 组：Review 组（验证交付物）

**职责**：
- 验证功能完整性
- 检查代码质量
- 确认测试覆盖
- 评估文档完备性

**输出**：Review 报告（通过/待改进清单）

### B 组：挑刺组（发现漏洞盲区）

**职责**：
- 有理有据地挑战 A 组结论
- 寻找边界条件漏洞
- 质疑隐含假设
- 压力测试异常路径

**原则**：不是为找茬而找茬，每个质疑必须：
1. 有具体证据（代码/日志/复现步骤）
2. 说明潜在风险（影响范围/严重程度）
3. 建议修复方向

**输出**：漏洞盲区清单（问题/证据/风险/建议）

## 执行流程

```
EPIC 开发完成
     ↓
A组 Review（2h）→ Review 报告
     ↓
B组 挑刺（2h）→ 漏洞盲区清单
     ↓
联合评审（1h）
     ↓
┌─────────────────┐
│ 100% 信心上线？ │
└────────┬────────┘
         │
    Yes ─┴─ No
     ↓       ↓
   发布    修复迭代 → 重新 Review
```

## 检查清单

### 功能完整性
- [ ] 所有 ticket AC 满足
- [ ] 关键路径手动验证通过
- [ ] 边界条件覆盖

### 代码质量
- [ ] 无 any 类型泄漏
- [ ] 错误处理完备
- [ ] 无硬编码敏感信息
- [ ] 符合项目规范

### 测试覆盖
- [ ] 核心逻辑有单元测试
- [ ] 集成测试通过
- [ ] 无 flaky test

### 文档完备
- [ ] API 文档更新
- [ ] 变更日志记录
- [ ] 部署说明清晰

### 运维就绪
- [ ] 日志可观测
- [ ] 错误可追溯
- [ ] 回滚方案明确

## 输出模板

### A 组 Review 报告

```markdown
# EPIC-XXX Review 报告

**Review 组**: [成员列表]
**日期**: YYYY-MM-DD

## 通过项
- [ ] 功能完整性 ✓/✗
- [ ] 代码质量 ✓/✗
- [ ] 测试覆盖 ✓/✗
- [ ] 文档完备 ✓/✗

## 待改进清单
| # | 问题 | 严重程度 | 建议 |
|---|------|----------|------|
| 1 | ... | P0/P1/P2 | ... |

## 结论
[ ] 通过 / [ ] 需修改后复审
```

### B 组挑刺报告

```markdown
# EPIC-XXX 漏洞盲区报告

**挑刺组**: [成员列表]
**日期**: YYYY-MM-DD

## 发现清单

### 1. [问题标题]
**证据**: [代码位置/复现步骤/日志]
**风险**: [影响范围/严重程度]
**建议**: [修复方向]

### 2. ...

## 对 A 组结论的质疑
| A组结论 | 质疑点 | 依据 |
|---------|--------|------|
| ... | ... | ... |
```

## 专家组配置建议

### A 组（Review）
- 🏗️ 架构师：整体设计评审
- 🖥️ 后端专家：实现质量
- 🧪 测试专家：覆盖评估

### B 组（挑刺）
- 🔒 安全专家：漏洞挖掘
- 🚀 运维专家：部署风险
- 🎯 产品专家：需求偏差

## 触发时机

在 EPIC ticket 中添加：

```markdown
## 验收阶段

**状态**: 待验收 → 验收中 → 已验收

**A 组 Review**: [链接]
**B 组挑刺**: [链接]
**联合评审结论**: [ ] 通过 / [ ] 迭代

**上线信心**: ___% 
```

---

> 记住：没有 100% 信心，就继续迭代。交付质量 > 交付速度。
