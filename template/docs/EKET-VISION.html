版本: v2.1.4

目的: 定义 EKET 框架的协作哲学和设计原则

核心愿景

所有 instance 都是 resource，无论其形态是： - Subagent（被初始化的执行节点） - 独立 Session（独立运行的实例） - 人类（项目所有者/决策者）

所有 resource 在框架内平等协作，最大限度地发挥各自优势，实现自主协作。

Resource 类型与职责

Resource 类型 特点 优势 职责
Master 长期运行、全局视角 协调、规划、审查 需求分析、任务拆解、PR 审核、进度追踪、等待期间主动规划
Slaver 按需唤醒、专注执行 高效、专业、自主 领取任务、开发、测试、提交 PR
Subagent 临时初始化、单一职责 快速、并行、专注 完成特定子任务（代码审查、测试审查等）
人类 最终决策者、项目所有者 判断力、创造力 需求确认、重大决策、方向把控

自主协作原则

1. 最大化自主性

原则：每个 resource 在其职责范围内拥有最大自主权，无需等待确认即可执行常规操作。

Master 自主操作： - 需求分析 → 任务拆解 → 创建 tickets - 初始化 Slaver 团队 → 任务自动落入 ready - PR 审核 → 批准/驳回/要求修改 - 每 10 分钟更新项目状态 - 等待 PR 反馈期间 → 主动同步 roadmap/规划下阶段/拆解新需求

Slaver 自主操作： - 查看 tickets → 领取匹配任务 - 创建 worktree → 开发 → 测试 - 提交 PR → 请求 Master 审核 - Master approve 后 → 直接领取新任务（无需等待 merge） - 根据 Review 反馈 → 修改代码 → 重新提交

人类自主操作： - 随时查看项目状态 - 在 inbox/human_feedback/ 回复决策请求 - 在 inbox/human_input.md 输入新需求

2. 最小化阻塞

原则：只有在真正需要人类决策时才阻塞执行，其他情况均通过异步通知推进。

不阻塞的情况： - Slaver 领取任务 → 直接执行，完成后通知 - Slaver 提交 PR → 直接提交，Master 异步审核 - Master approve → Slaver 直接领取新任务，无需等待 merge - Slaver 检测反馈 → 直接读取并执行修改

需要阻塞的情况： - 需求不明确 → 写入 inbox/dependency-clarification.md → 等待人类确认 - 技术难点 > 30 分钟 → 写入 inbox/blocker_report.md → 等待 Master 仲裁 - Scope 变更 → 写入 inbox/human_feedback.md → 等待人类决策

3. 异步通信优先

原则：所有 resource 通过消息队列和文件系统进行异步通信，避免同步等待。

通信方式： | 类型 | 用途 | 频率 | |——|——|——| | 消息队列 | 实时通知（任务领取、PR 提交、Review 结果） | 每 10 秒检查 | | 文件系统 | 持久化记录（tickets、PR 描述、反馈） | 按需读取 | | 索引文件 | 全局状态同步（jira/state/ticket-index.yml） | 每 10 分钟更新 |

4. 状态透明

原则：所有 resource 的状态对所有其他 resource 透明可见。

透明内容： - 所有 tickets 的状态（索引文件优先） - 所有 Slaver 的工作负载（slaver_workload） - 所有 PR 的审核进度 - 项目整体进度（每 10 分钟更新报告）

协作流程

需求 → 交付完整流程

人类输入需求 (inbox/human_input.md)
    │
    ▼
Master 检测（每 10 秒）
    │
    ▼
Master 分析需求 → 拆解任务 → 创建 tickets
    │
    ▼
Master 初始化 Slaver 团队 → 任务状态 = ready
    │
    ▼
Slaver 检测（每 10 秒/5 分钟）→ 领取匹配任务
    │
    ▼
Slaver 创建 worktree → 开发 → 测试 → 提交 PR
    │
    ▼
Master 检测 PR（每 10 秒）→ 审核
    │
    ├── 批准 → Slaver 领取新任务（无需等待 merge）
    ├── 修改 → Slaver 修改 → 重新提交
    └── 驳回 → Slaver 重新分析/开发
    │
    ▼
Master 合并到 main → 更新状态 = done

异常处理流程

Slaver 遇到问题
    │
    ▼
是否可独立解决（< 30 分钟）？
    ├── 是 → 解决 → 继续执行
    └── 否 → 判定类型
            │
            ├── 需求不明 → inbox/dependency-clarification.md → 等待人类
            ├── 技术难点 → inbox/blocker_report.md → 等待 Master 仲裁
            └── 资源缺失 → inbox/dependency-clarification.md → 等待人类

设计决策

为什么索引文件优先？

问题：如果每个 ticket 文件独立存储状态，Master 和 Slaver 需要扫描所有文件才能获取全局状态，效率低下。

解决：jira/state/ticket-index.yml 作为唯一真相源（single source of truth）： - Master 更新状态 → 先更新索引文件 - Slaver 领取任务 → 先读取/更新索引文件 - 状态查询 → 只读索引文件，不扫描所有 ticket

优势： - 高效：O(1) 时间获取任意 ticket 状态 - 一致：避免多文件状态不一致 - 透明：所有 resource 看到相同的全局视图

为什么 Slaver 无需等待 merge？

问题：传统流程中，PR 批准后需要等待 merge 完成才能开始新任务，造成资源闲置。

解决：Master approve 后，Slaver 可以直接领取新任务： - approve 表示功能认可 - merge 是形式操作，不影响功能正确性 - Slaver 时间宝贵，不应浪费在等待上

例外：用户明确要求等待时才暂停（人类有最终决定权）

为什么检查频率是 10 秒/5 分钟/10 分钟？

设计原则：根据紧急程度和资源类型动态调整。

检查类型 频率 原因
Master 空闲 每 10 秒 快速响应 PR/仲裁/人类通知
Slaver 空闲等待 每 10 秒 快速响应 Master 反馈
Master/Slaver 工作中 每 5 分钟 平衡专注度和响应速度
项目状态更新 每 10 分钟 定期同步，避免过度频繁

Resource 平等性

人类不是”老板”，Master 不是”领导”，Slaver 不是”下属”。

所有 resource 是平等的协作者，只是职责不同： - 人类：项目所有者，负责方向把控和重大决策 - Master：协调者，负责任务管理和质量审查 - Slaver：执行者，负责高效交付 - Subagent：临时协作者，负责专业化子任务

决策权分配： | 决策类型 | 决策者 | 说明 | |———-|——–|——| | 项目方向 | 人类 | 愿景、目标、优先级 | | 任务拆解 | Master | 如何拆解、如何分配 | | 技术方案 | Slaver | 如何实现、如何测试 | | 代码审查 | Master | 是否批准、是否合并 | | 需求变更 | 人类 | 是否接受变更 |

未来方向

多 Master 协作

  • 支持多个 Master 实例并行工作
  • Master 之间通过消息队列协调
  • 每个 Master 负责不同 Epic 或模块

动态角色切换

  • Slaver 完成一个任务后可临时切换角色
  • 例如：frontend_dev → tester 编写测试
  • 提高资源利用率

人类反馈强化学习

  • 记录人类对所有决策的反馈
  • 基于反馈优化 Master 和 Slaver 的行为
  • 越来越符合人类偏好
