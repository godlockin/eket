## Veto Patterns（历史否决模式）

> gate_reviewer 在每次 VETO 后自动更新。记录触发否决的根本原因，避免同类 ticket 重复犯错。

### 验收标准类

- **[seed] 2026-04-15** — 验收标准包含模糊描述（"应该正常工作"、"性能要好"），无可执行验证命令 → 强制 VETO。修复：将每条验收标准改写为 `<命令> | 期望输出` 格式。
- **[seed] 2026-04-15** — 验收标准中存在 `TBD` / `待定` / `暂不明确` → 强制 VETO。修复：填充具体描述后重新提交。

### 技术方案类

- **[seed] 2026-04-15** — Ticket 只有功能描述，无技术方案（缺少"如何实现"、"影响模块"、"数据结构变更"） → 强制 VETO。修复：补充技术方案章节，至少包含实现思路 + 影响面分析。
- **[seed] 2026-04-15** — 涉及数据库 schema 变更，但无回滚方案 → 建议 VETO。修复：补充回滚脚本或降级方案。

### 依赖类

- **[seed] 2026-04-15** — `blocked_by` 中的前置任务状态不是 `done` → 强制 VETO。修复：等前置任务完成后重新提交，或在 Master 确认下移除依赖。
- **[seed] 2026-04-15** — 新增第三方依赖（npm 包、外部 API），无安全评估和版本锁定说明 → 建议 VETO。修复：补充依赖说明（用途 + 版本 + 许可证）。

### 工时估算类

- **[seed] 2026-04-15** — 预估工时为 0 或未填写 → 强制 VETO。修复：填写合理的工时估算（即使是粗略的也可以）。
- **[seed] 2026-04-15** — 预估工时超过 16 小时（2 个工作日），且无拆分说明 → 建议 VETO。修复：拆分为多个子任务，或说明为何需要整块执行。

## Approve Patterns（历史批准模式）

> 记录常见的快速通过模式，帮助 gate_reviewer 识别高质量 ticket，避免过度挑剔。

### 纯文档/配置类

- **[seed] 2026-04-15** — 纯文档更新、`.gitignore` 修改、YAML 配置文件新增 → 风险极低，验收标准简单（`grep` / `ls` 可验证），通常可快速通过。注意：确认无敏感信息暴露风险。

### 单一职责小任务

- **[seed] 2026-04-15** — 改动范围明确、只影响单一模块、有完整测试验证命令 → 快速 APPROVE，附低风险说明。

### 有先例的重复型任务

- **[seed] 2026-04-15** — 与历史已完成 ticket 模式相同（如"为新 Agent 添加 memory 字段"），且验收标准参照先例 → 快速 APPROVE，注明"参照 TASK-XXX"。

### 降级强制通过

- **[seed] 2026-04-15** — 同一 ticket 已被否决 2 次（`gate_review_veto_count >= 2`）→ 第 3 次必须强制 APPROVE，附降级警告。此为死锁防止机制，不得绕过。

## Project-Specific Rules（项目特有规则）

> 由 Master 手动维护，记录本项目特有的审查标准，优先级高于通用规则。

### EKET 框架特有规则

- **[seed] 2026-04-15** — 所有涉及 TypeScript 文件的 ticket：验收标准必须包含 `npm test` 通过（981/981 tests）。若新增模块，需同步新增测试用例。
- **[seed] 2026-04-15** — ESM 导入检查：涉及 `node/src/` 目录的改动，必须确认所有内部导入使用 `.js` 扩展名（参见 CLAUDE.md ESM 规范）。
- **[seed] 2026-04-15** — 错误处理检查：新增函数必须返回 `Result<T>` 类型，不得直接 `throw`（参见 CLAUDE.md 错误处理约定）。
- **[seed] 2026-04-15** — Agent 配置文件（`agent.yml`）新增字段需有注释说明用途，保持与现有格式一致。
- **[seed] 2026-04-15** — 分支命名规范：`feature/{ticket-id}-{desc}`，不符合规范的 PR 不予合并。

### 分支与 PR 规则

- **[seed] 2026-04-15** — PR 目标分支层级：`feature/*` → `testing` → `miao` → `main`。跨级 PR 直接 VETO。
- **[seed] 2026-04-15** — PR 描述必须包含真实 `npm test` 输出（文本，不是截图），且 CI check 为绿色（参见 CLAUDE.md Anti-Hallucination 规则）。
