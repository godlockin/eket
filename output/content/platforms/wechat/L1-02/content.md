## 🎭 两个角色，一套规则

EKET 的核心就两个角色：

**Master（项目经理）**
- 🧠 负责思考，不写代码
- 📋 拆需求、分任务、审代码
- 🚫 禁止直接改代码文件

**Slaver（开发工程师）**
- 💻 负责执行，专注交付
- ✅ 领任务、写代码、提PR
- 🚫 禁止自己决定做什么

---

## 🤔 为什么要强制分工？

### 单AI的问题

一个AI同时当PM和开发：

```
需求来了
  ↓
边想边写   ← 容易跑偏
  ↓
写到一半改需求  ← 上下文混乱
  ↓
忘了最初要做什么  ← 💥
```

### 分工后

```
Master                    Slaver
───────                   ───────
收到需求
  ↓
拆分任务
  ↓
分发 ──────────────────→ 领取
                           ↓
                         独立开发
                           ↓
                         提交 ──────→ 审核
                                       ↓
                                     合并
```

关键：**各司其职，互不越界**

## 📝 Ticket：任务的载体

每个任务都是一个 Markdown 文件：

```markdown
# TASK-042: 实现用户登录功能

## 状态
- [x] 设计
- [ ] 开发
- [ ] 测试

## 验收标准
1. 支持邮箱+密码登录
2. 返回 JWT token
3. 错误提示友好

## 依赖
- TASK-041（用户表设计）

## 负责人
slaver_1
```

**文件就是协议**，Git 追踪一切变更。

## 🔄 协作流程

### 1. Master 创建任务

```bash
eket task:create "实现用户登录" --priority P1
```

### 2. Slaver 领取任务

```bash
eket task:claim TASK-042
```

领取是**原子操作**：
- 检查依赖是否完成
- 检查是否已被领取
- 更新状态 + 分配负责人
- **<21ms 完成**

### 3. Slaver 开发提交

开发完成后：

```bash
eket task:complete TASK-042
```

自动执行 5 步 Saga：
1. 运行测试
2. 更新 ticket 状态
3. 创建 checkpoint
4. 触发 Gate Review
5. 通知 Master

### 4. Master 审核合并

Gate Review 检查：
- ✅ 测试通过
- ✅ 代码规范
- ✅ 无安全问题
- ✅ 满足验收标准

## 💡 设计哲学

### 约束即自由

Master 不能写代码 → 专注规划
Slaver 不能定需求 → 专注交付

就像流水线：**专业化分工提升整体效率**

### 文件即通信

不用复杂的消息队列：
- Ticket 文件 = 任务状态
- Git commit = 变更记录
- 文件锁 = 并发控制

**简单到极致，可靠到极致**

---

## ⚠️ 常见误区

### ❌ "Slaver就是执行机器"

错。Slaver 在任务范围内有完全自主权：
- 技术选型
- 实现细节
- 测试策略

### ❌ "Master什么都不干"

错。Master 的工作量不小：
- 需求分析
- 任务拆解
- 依赖梳理
- 代码审核
- 冲突协调
