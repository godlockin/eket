> 核心价值：不是告诉你「怎么做」，而是展示「怎么想」

## L3-01: 三仓分离——为什么代码/知识/任务要分开存

### 决策场景

**面临的选择**：
- 方案 A：单仓库（Monorepo）—— 所有内容放一起
- 方案 B：双仓库 —— 代码 + 文档/任务
- 方案 C：三仓库 —— 代码 / 知识 / 任务 完全分离

**触发时机**：
- AI Agent 开始产生大量中间制品（思考记录、对话历史、决策日志）
- 团队规模从 1-2 人扩展到多 Agent 协作
- Git 历史被非代码提交淹没，难以追溯真正的代码变更

### 核心权衡点

| 维度 | 单仓 | 双仓 | 三仓 |
|------|------|------|------|
| 原子性 | 高（一次提交全部同步） | 中 | 低（需额外同步机制） |
| 关注点分离 | 低 | 中 | 高 |
| Git 历史清晰度 | 低（混杂） | 中 | 高（各司其职） |
| 权限粒度 | 低 | 中 | 高（可独立授权） |
| 工具链复杂度 | 低 | 中 | 高 |
| AI 上下文污染 | 严重 | 中等 | 最小化 |

### 结构大纲

#### 1. 问题：单仓的崩溃时刻（500字）
- 案例：当 EKET 的 Git 日志变成这样...
  - 80% 的提交是 "update memory"、"add ticket"
  - `git log --oneline` 找不到上次代码改动
  - Code Review 被知识库更新淹没
- 根因分析：代码 vs 知识 vs 任务的生命周期差异
  - 代码：低频高质量变更
  - 知识：中频累积式更新
  - 任务：高频状态流转

#### 2. 方案演进：从痛点到三仓（800字）
- **阶段一：单仓蜜月期**
  - 一切都在一起，简单直接
  - 适用：个人项目、探索期
- **阶段二：双仓过渡期**
  - 代码 + 文档分离
  - 发现问题：任务和文档混在一起，ticket 状态变更污染知识库历史
- **阶段三：三仓成熟期**
  - 代码仓（code_repo）：纯代码 + 配置
  - 知识仓（confluence）：Memory、Wiki、经验沉淀
  - 任务仓（jira）：Tickets、Sprint、进度追踪

#### 3. 权衡：三仓的代价（600字）
- **同步复杂性**：
  - 问题：代码引用哪个版本的文档？
  - 解法：版本标签 + 引用约定（`confluence@v1.2` 语法）
- **原子性丧失**：
  - 问题：改代码 + 更新文档 + 关闭 ticket 需要三次提交
  - 解法：CI Pipeline 联动 + 脚本自动化
- **上手门槛**：
  - 问题：新人需要 clone 三个仓库
  - 解法：初始化脚本 `eket init` 一键拉取

#### 4. 决策框架：什么时候该分（400字）
- **分离信号**（满足 2 条即考虑分仓）：
  - [ ] 非代码提交占比 > 50%
  - [ ] 团队成员 > 3 人（或 Agent > 2 个）
  - [ ] 需要对不同内容设置不同权限
  - [ ] Git 历史追溯效率明显下降
- **保持单仓的场景**：
  - 探索期项目
  - 纯个人项目
  - 文档极少的工具类项目

#### 5. 验证：EKET 的实践数据（300字）
- 分离前 vs 分离后对比：
  - 代码仓提交噪音：-85%
  - Code Review 聚焦度：+60%
  - Agent 上下文利用率：+40%
- 三个月后的复盘：未出现同步事故

### 决策矩阵

```
                    ┌─────────────────────────────────────┐
                    │         项目复杂度                    │
                    │    低        中        高            │
        ┌───────────┼─────────────────────────────────────┤
   团队  │  1-2人    │  单仓      单仓      双仓           │
   规模  │  3-5人    │  单仓      双仓      三仓           │
        │  >5人     │  双仓      三仓      三仓           │
        └───────────┴─────────────────────────────────────┘
        
        AI Agent 场景：复杂度自动 +1 级（因为产出量大）
```

### 适用读者画像

- **主要读者**：正在设计 AI Agent 工作流的架构师
- **次要读者**：维护大型 Monorepo 感到痛苦的 Tech Lead
- **前置知识**：了解 Git 基本操作、有过多人协作经验
- **阅读收益**：获得「何时分仓」的决策框架，避免踩坑

## L3-02: Rust vs Node.js——性能与效率的平衡术

### 决策场景

**面临的选择**：
- 全 Node.js：开发快，但性能有天花板
- 全 Rust：性能极致，但开发成本高
- 混合架构：核心用 Rust，胶水用 Node.js

**触发时机**：
- 某个模块成为性能瓶颈
- 需要处理高并发/低延迟场景
- 团队有 Rust 经验想发挥

### 核心权衡点

| 维度 | Node.js | Rust | 混合 |
|------|---------|------|------|
| 开发速度 | 快（1x） | 慢（3-5x） | 中（1.5x） |
| 运行性能 | 中 | 极高（10-100x） | 按需 |
| 内存安全 | GC 管理 | 编译期保证 | 混合 |
| 生态成熟度 | 极高 | 高速增长 | 需要桥接 |
| 招聘难度 | 低 | 高 | 中 |
| 维护成本 | 低 | 中 | 中高（两套） |

### 结构大纲

#### 1. 问题：Node.js 的性能天花板（500字）
- 场景还原：EKET 选举算法的响应延迟问题
  - Node.js 版本：P99 延迟 50ms
  - 在多 Agent 场景下，累积延迟不可接受
- 根因分析：
  - 单线程事件循环的局限
  - JSON 序列化开销
  - GC 停顿的不可预测性

#### 2. 方案：三条路的探索（800字）
- **路线一：优化 Node.js**
  - 尝试：Worker Threads、消息池化
  - 结果：提升 30%，仍不够
  - 教训：架构问题无法靠优化解决
- **路线二：全量重写 Rust**
  - 评估：开发周期 3 个月，风险高
  - 放弃原因：团队 Rust 经验不足
- **路线三：关键路径 Rust 化**
  - 决策：只迁移选举、消息总线两个模块
  - 方案：Rust 编译成 Native Addon，Node.js 调用

#### 3. 权衡：混合架构的代价（600字）
- **FFI 边界成本**：
  - 问题：Node.js ↔ Rust 数据传递有开销
  - 解法：批量传输 + 二进制协议（MessagePack）
  - 经验：跨边界调用粒度要大
- **调试复杂性**：
  - 问题：堆栈跨两个运行时
  - 解法：Rust 侧完善 Error 类型 + tracing
- **构建流程**：
  - 问题：需要 Rust 工具链 + Node.js 工具链
  - 解法：Docker 统一构建环境

#### 4. 决策框架：什么场景选 Rust（400字）
- **Rust 适用场景**（满足任意一条）：
  - [ ] 延迟敏感（P99 < 10ms）
  - [ ] CPU 密集计算
  - [ ] 需要精确内存控制
  - [ ] 安全关键路径
- **Node.js 适用场景**：
  - [ ] I/O 密集型
  - [ ] 快速迭代需求
  - [ ] 需要丰富 NPM 生态
  - [ ] 团队 JS/TS 经验丰富
- **混合架构适用场景**：
  - [ ] 有明确的性能热点
  - [ ] 团队有（或愿意培养）Rust 能力
  - [ ] 长期维护的核心系统

#### 5. 验证：EKET 的性能提升（300字）
- Rust 化前后对比：
  - 选举延迟：50ms → 2ms（25x 提升）
  - 消息吞吐：1k/s → 50k/s（50x 提升）
  - 内存占用：200MB → 50MB（4x 降低）
- 开发成本：
  - Rust 模块：6 周（2 人）
  - Node.js 胶水层：1 周
- ROI 评估：值得

### 决策矩阵

```
选择流程：

[性能需求] ─── 不敏感 ───> Node.js（结束）
     │
   敏感
     │
     ▼
[热点明确？] ─── 否 ───> 先用 Node.js + Profile（结束）
     │
    是
     │
     ▼
[团队有 Rust 经验？] ─── 是 ───> 热点 Rust 化
     │
    否
     │
     ▼
[能接受学习成本？] ─── 是 ───> 热点 Rust 化 + 培训
     │
    否

     ▼
Node.js 优化 + 考虑其他方案（Go、C++）
```

### 适用读者画像

- **主要读者**：在 Node.js 项目中遇到性能瓶颈的架构师
- **次要读者**：想引入 Rust 但不知如何落地的技术负责人
- **前置知识**：熟悉 Node.js，对 Rust 有基本了解
- **阅读收益**：获得「何时引入 Rust」的判断框架，避免过早优化或优化不足

## L3-03: 文件 vs 数据库——分布式协作的取舍

### 决策场景

**面临的选择**：
- 纯文件系统：JSON/YAML 文件作为状态存储
- 嵌入式数据库：SQLite
- 分布式存储：Redis、etcd、PostgreSQL

**触发时机**：
- 多 Agent 同时读写同一状态
- 状态查询变复杂（需要过滤、排序、聚合）
- 数据一致性出现问题

### 核心权衡点

| 维度 | 文件 | SQLite | Redis | PostgreSQL |
|------|------|--------|-------|------------|
| 部署复杂度 | 无 | 极低 | 中 | 高 |
| 并发写入 | 差（需锁） | 中（WAL） | 优 | 优 |
| 查询能力 | 无 | 完整 SQL | 有限 | 完整 SQL |
| 可观测性 | 低 | 中 | 高 | 高 |
| 分布式支持 | 无 | 无 | 是 | 是（需配置） |
| 人类可读 | 高 | 需工具 | 需工具 | 需工具 |

### 结构大纲

#### 1. 问题：文件系统的并发噩梦（500字）
- 场景还原：两个 Agent 同时更新 `tickets.json`
  - Agent A 读取 → 修改 → 写入
  - Agent B 读取 → 修改 → 写入
  - 结果：Agent A 的修改被覆盖（Last Write Wins）
- 根因分析：
  - 文件系统无事务支持
  - 读-改-写不是原子操作
  - 缺乏并发控制机制

#### 2. 方案演进：从文件到混合存储（800字）
- **阶段一：纯文件 + 文件锁**
  - 方案：`flock` / `.lock` 文件
  - 问题：锁粒度太大，性能差
  - 适用：单 Agent 场景
- **阶段二：SQLite 过渡**
  - 方案：结构化数据迁移到 SQLite
  - 优势：ACID 事务、SQL 查询
  - 问题：仍是单机，多节点需要共享存储
- **阶段三：分层存储架构**
  - 热数据：Redis（状态、锁、缓存）
  - 温数据：SQLite（任务、日志）
  - 冷数据：文件（知识库、文档）

#### 3. 权衡：每层的取舍（600字）
- **文件层保留的内容**：
  - Memory 文档（Markdown，需要人类可读）
  - 配置文件（YAML，需要版本控制）
  - 原则：「需要 Git 追踪」或「需要人类直接编辑」
- **SQLite 承载的内容**：
  - Ticket 状态机
  - 操作日志
  - 原则：「需要复杂查询」但「单机足够」
- **Redis 承载的内容**：
  - 分布式锁
  - Agent 心跳
  - 实时状态
  - 原则：「需要多节点共享」且「可以丢失」

#### 4. 决策框架：数据放哪层（400字）
```
数据分层决策树：

[需要 Git 版本控制？] ─── 是 ───> 文件
     │
    否
     │
     ▼
[需要人类直接编辑？] ─── 是 ───> 文件
     │
    否
     │
     ▼
[需要多节点共享？] ─── 是 ───> Redis
     │
    否
     │
     ▼
[需要复杂查询？] ─── 是 ───> SQLite
     │
    否
     │
     ▼
文件（简单 JSON）
```

- **反模式警示**：
  - 把所有东西都放 Redis（成本高、数据不持久）
  - 把所有东西都放文件（并发灾难）
  - 过早引入 PostgreSQL（运维成本）

#### 5. 验证：EKET 的分层效果（300字）
- 并发冲突：100% 消除
- 查询性能：
  - 文件遍历：O(n) → SQLite 索引：O(log n)
  - 大型项目 Ticket 查询：10s → 50ms
- 部署复杂度变化：
  - 单机场景：仅增加 SQLite（零配置）
  - 多机场景：需要 Redis（标准组件）
- 一致性事故：0 起

### 决策矩阵

```
┌────────────────┬──────────┬──────────┬──────────┬────────────┐
│ 数据特征        │ 文件     │ SQLite   │ Redis    │ PostgreSQL │
├────────────────┼──────────┼──────────┼──────────┼────────────┤
│ 人类可读性要求   │ ✅ 首选  │ ❌       │ ❌       │ ❌         │
│ Git 版本控制    │ ✅ 首选  │ ❌       │ ❌       │ ❌         │
│ 高并发写入      │ ❌       │ ⚠️ 有限  │ ✅ 首选  │ ✅         │
│ 复杂查询        │ ❌       │ ✅ 首选  │ ⚠️ 有限  │ ✅         │
│ 多节点共享      │ ❌       │ ❌       │ ✅ 首选  │ ✅         │
│ 零配置部署      │ ✅       │ ✅       │ ❌       │ ❌         │
│ 数据可丢失      │ -        │ -        │ ✅ 适合  │ ❌         │
└────────────────┴──────────┴──────────┴──────────┴────────────┘

EKET 的选择：三层混合（文件 + SQLite + Redis 可选）
```
