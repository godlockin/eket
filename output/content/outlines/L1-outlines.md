## L1-01: EKET是什么

**小红书标题**: "3个AI打架怎么办？"

### Hook（开篇痛点）
同时开3个AI窗口，互相覆盖代码，项目彻底乱套。

### 核心论点（3个）

**1. 多AI协作 = 团队管理问题**
- 比喻：3个实习生同时改同一个文件
- 痛点可视化：代码冲突、重复工作、互相覆盖
- 转折：AI不是工具问题，是协作问题

**2. EKET = 给AI装上组织架构**
- 核心概念：Master（项目经理）+ Slaver（执行者）
- 类比：麦当劳厨房流水线 vs 家庭小作坊
- 关键差异：谁能改什么、谁批准谁

**3. 5分钟理解框架全貌**
- 三仓分离：代码/任务/知识（confluence/jira/code）
- 工作流可视化：需求 → 拆解 → 分配 → 执行 → 审核
- 类比：Git分支策略 + 企业OA审批

### 案例/故事

**场景**：小明让3个Claude同时开发登录功能
- **Before**：A写前端、B写后端、C写测试，各自为战，最后合不到一起
- **After**：Master分配任务、约定接口、Slaver各司其职、PR统一审核

### 金句收尾
"AI不需要更聪明，需要更有组织。"

## L1-02: Master-Slaver分工

**小红书标题**: "让AI学会分工协作"

### Hook（开篇痛点）
AI做完就跑，下次问它"上次做到哪了"，一脸懵逼。

### 核心论点（3个）

**1. Master = 项目经理（不动手写代码）**
- 职责清单：需求分析、任务拆解、进度监控、PR审核
- 红线：禁止自己写代码、禁止自审自批
- 比喻：交响乐指挥 vs 乐手

**2. Slaver = 执行专家（只管自己的票）**
- 职责清单：领取任务、分析设计、编码测试、提交PR
- 红线：禁止跨票协助、禁止绕过审核
- 关键机制：Worktree隔离（每人一个工作目录）

**3. 信息交换协议（不靠记忆靠文件）**
- 五阶段生命周期：CLAIM → ANALYSIS → IN_PROGRESS → TEST → REVIEW
- 消息队列：`shared/message_queue/inbox/`
- 类比：微信工作群的@和回复

### 案例/故事

**场景**：用EKET开发一个Todo应用
- Master收到需求 → 拆成前端/后端/测试3个ticket
- Slaver-A领取前端票 → 写分析报告 → Master批准 → 开发 → PR
- Slaver-B领取后端票 → 同样流程，但在独立worktree
- Master审核两个PR → 合并 → 发布

### 金句收尾
"好的分工让AI从'单兵作战'变成'特种部队'。"

## L1-03: 上下文问题

**小红书标题**: "AI总忘事怎么办？"

### Hook（开篇痛点）
聊到一半AI说"对不起，我不记得我们之前说了什么"。

### 核心论点（3个）

**1. 上下文窗口 = AI的"工作记忆"**
- 科普：什么是context window（128K tokens ≈ 10万字）
- 痛点可视化：长对话 → 信息丢失 → 重复劳动
- 比喻：金鱼的7秒记忆 vs 人的短期记忆

**2. EKET的三层记忆系统**
- **Session Cache**：当前对话的临时记忆（内存）
- **Ticket文件**：任务级持久记忆（Git跟踪）
- **Confluence知识库**：项目级长期记忆（可检索）
- 类比：便签纸 → 工作日志 → 公司Wiki

**3. 文件系统 = AI的"外脑"**
- 核心设计：所有状态写入文件，不依赖AI记忆
- 恢复机制：`.eket/ACTIVE_CONTEXT.md`
- 知识沉淀：复盘 → 写入memory → 未来可检索

### 案例/故事

**场景**：开发中断后恢复
- **Before**：第二天打开Claude，"我们昨天做到哪了？" → AI懵逼
- **After**：Slaver启动 → 读取ACTIVE_CONTEXT.md → 自动恢复到昨天的worktree和进度

### 金句收尾
"AI的记忆力不行，但文件系统的记忆力无限。"
