# EKET - AI 智能体协作框架

## 系统目标

本框架是一个**多智能体协作系统**，用于 AI 智能体（如 Claude Code）进行研发任务的自动化处理。系统通过规范化的流程、清晰的职责划分和可靠的通信机制，实现多个智能体实例的高效协作。

## 核心架构

### 三大 Git 仓库

| 仓库 | 用途 | 内容 |
|------|------|------|
| `shared/confluence` | 文档中心 | 需求文档、架构设计、技术规范、会议记录 |
| `shared/code_repo` | 代码仓库 | 源代码、测试、部署配置、CI/CD |
| `shared/jira` | 任务管理 | 需求票、缺陷票、任务票、状态追踪 |

### 智能体分类

| 类型 | 生命周期 | 职责 | 示例 |
|------|---------|------|------|

| **执行智能体** | 按需 | 任务领取、分支开发、PR 提交 | 设计师、开发、测试、运维 |

## 工作流程

### 标准任务流转

```
1. 人类输入需求 → 需求分析师接收
2. 需求分析师 → 拆解为 Jira tickets
3. 执行智能体 → 轮询并领取任务
4. 执行智能体 → 创建 git 分支 → 开发 → 提交 PR
5. 执行智能体 → 通过消息队列唤醒协调智能体
6. 协调智能体 → review PR → 合并或要求修改
7. 任务完成 → 更新 Jira 状态 → 通知相关方
```

### Ticket 状态机

```
backlog ──→ analysis ──→ design ──→ dev ──→ test ──→ review ──→ done
   │            │           │         │       │         │

```

## 通信协议

### 三种通信方式

1. **消息队列**（实时通信）
   - 位置：`shared/message_queue/`
   - 格式：JSON 文件
   - 用于：任务通知、状态更新、紧急消息

2. **Git Commit**（状态持久化）
   - 位置：三大 Git 仓库
   - 格式：Conventional Commits
   - 用于：任务状态、代码变更、文档更新

3. **共享状态文件**（跨实例同步）
   - 位置：`shared/.state/`
   - 格式：JSON
   - 用于：智能体状态、锁信息、任务分配表

### 消息格式

```json
{
  "id": "msg_20240115_001",
  "timestamp": "2024-01-15T10:30:00Z",
  "from": "agent_frontend_dev_001",
  "to": "agent_tech_manager",
  "type": "pr_review_request",
  "priority": "normal",
  "payload": {
    "ticket_id": "FEAT-123",
    "pr_number": 42,
    "branch": "feature/feat-123-user-auth",
    "summary": "实现用户登录功能"
  }
}
```

## 决策机制

### 混合决策模式

| 决策类型 | 决策者 | 流程 |
|---------|-------|------|
| **常规任务** | 执行智能体自主 | 领取 → 执行 → 提交 PR |
| **技术方案** | 技术经理 | 执行智能体提议 → 技术经理审批 |
| **需求变更** | 需求分析师 + 项目经理 | 分析影响 → 评估优先级 → 决策 |
| **架构变更** | 技术经理 + 人类确认 | 提案 → Review → 人类批准 |
| **跨模块协调** | 项目经理 | 协调资源 → 排期 → 分配 |

### 上报规则

执行智能体在以下情况需上报协调智能体：
- 任务复杂度超出预估（>4 小时工作量）
- 技术选型存在争议
- 需要跨模块协调
- 发现需求不明确或矛盾
- 遇到阻塞性问题超过 30 分钟

## 分支管理

### 分支命名规范

```
feature/{ticket-id}-{short-desc}    # 功能开发
bugfix/{ticket-id}-{short-desc}     # 缺陷修复
hotfix/{ticket-id}-{short-desc}     # 紧急修复
docs/{ticket-id}-{short-desc}       # 文档更新
```

### 合并流程

```
1. 执行智能体完成开发
2. 执行智能体创建 PR，指定协调智能体为 reviewer
3. 执行智能体发送 review 请求消息
4. 协调智能体接收消息 → 执行 review
5. review 通过 → 合并到主分支
6. review 不通过 → 评论反馈 → 执行智能体修改
```

## 冲突避免

### 权责分离

| 模块 | 负责智能体 |
|------|----------|
| 需求文档 | 需求分析师 |
| 架构设计 | 技术经理 |
| 前端代码 | 前端开发 |
| 后端代码 | 后端开发 |
| 测试代码 | 测试员 |
| 部署配置 | 运维 |

### 文件锁机制

当智能体需要修改共享文件时：
1. 创建 `{filename}.lock` 文件，内容为智能体 ID
2. 完成修改后删除 lock 文件
3. 其他智能体检测到 lock 文件时等待或通知

## 智能体生命周期

### 常驻智能体（协调层）
```
启动 → 初始化 → 轮询循环 → 处理消息 → 更新状态 → (持续运行)
```

### 按需智能体（执行层）
```
休眠 → 被唤醒 → 领取任务 → 创建分支 → 开发 → 提交 PR → 等待 review → 修改/合并 → 休眠
```

### 动态加载
- 新智能体定义：在 `agents/executor/` 下创建配置
- 注册能力：更新 `agents/executor/executor_registry.yml`
- 唤醒方式：通过消息队列或脚本触发

## 记忆系统

### 三级记忆

| 类型 | 存储位置 | 内容 | 保留策略 |
|------|---------|------|---------|
| **短期记忆** | `runtime/memory/short_term/` | 当前任务上下文、临时变量 | 任务结束清除 |
| **长期记忆** | `runtime/memory/long_term/` | 项目知识、技术决策、经验 | 永久保留 |
| **外部记忆** | `shared/confluence/memory/` | 文档化知识、最佳实践 | 永久保留，可检索 |

## 快速开始

```bash
# 1. 初始化项目
./scripts/init.sh

# 2. 启动顶层协调智能体
./scripts/start.sh coordinator

# 3. 查看智能体状态
./scripts/manage.sh status

# 4. 手动触发智能体
./scripts/manage.sh wake <agent-name> --ticket <ticket-id>
```

## 给 AI 智能体的说明

当你（AI 智能体）被加载到本系统中时：

1. **首先读取本文件**，理解系统架构和协作规则
2. **读取 AGENTS.md**，了解你的角色定义和职责
3. **读取 SKILLS.md**，了解可用的技能
4. **检查消息队列**，查看是否有待处理的消息
5. **检查 Jira tickets**，查看是否有可领取的任务
6. **按照工作流程**执行任务并提交结果
