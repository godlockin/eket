# Performance AC Ambiguity — "Startup Time"

**场景/症状**: 验收标准说 "启动 < Xms" 但基线实现耗时远超目标

**根因**: 规格未区分：
- 进程启动开销（常量，~10-20ms）
- I/O 初始化（变量，取决于文件数量）
- 首次结果总耗时（进程 + I/O + 计算）

**解法**:
1. **先对基线实现进行 benchmark**
2. **向 Master 确认**: "startup 指进程启动，还是包含 I/O 的总耗时？"
3. **在实现前更新 AC**，补充明确定义

**示例**: 
- **Ticket**: TASK-636
- **AC 原文**: "二进制启动 < 10ms（vs Node 200ms）"
- **基线实测**: Node.js 总耗时 184ms
- **澄清后**:
  - 进程启动 < 15ms（✅ 实测 12ms）
  - 总耗时 < 10ms（❌ 对 284 个文件 stat 不可能）
- **解决方案**: 采纳第一种解读（进程开销），放弃第二种（总耗时）

**来源**: TASK-636

**相关**:
- 若 AC 要求 "<Xms 响应时间"，同样需区分 "冷启动" vs "热路径" vs "P99 延迟"
- 性能指标必须附带测量方法（wall time / CPU time / user+system）
