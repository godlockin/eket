文档重组与深度清理经验教训 — 2026-05-07

背景

用户质疑：“confluence/memory/lessons-learned-2026-05-06.md 写好了吗？最近一波更新是否总结？”

触发了项目全局文档审查和深度清理，14次提交，清理 ~9,000 行冗余。

🔴 Critical Pitfalls（我犯的严重错误）

P1: 片段式处理，缺乏全局视野

现象: - 用户问 confluence/ 文档 → 只看 confluence/ - 清理 confluence/ 后说”完成” → 用户指出 Root/jira 也乱 - 清理 Root 后 → 用户又发现 template/test 混乱 - 清理 template → 用户质疑”为何都保留” → 我才澄清职责 - 说”已清理” → 用户发现 --- Changes --- 垃圾文件

根因: 没有在第一时间做完整扫描，被动响应用户逐个发现问题

正确做法: 1. 先全面扫描 → find . -name "*.md" | wc -l（1,601文件） 2. 生成完整报告 → 按目录/类型/大小/时间分类 3. 识别所有问题 → Root混乱、jira堆积、template重复、worktree残留 4. 提交方案 → 用户一次性确认全部计划 5. 批量执行 → 而非用户每次指出一个问题才处理

教训: > 用户问”A好了吗”，不要只看A。立刻扫描A/B/C/D全部相关范围，一次性解决。

P2: 解释不清晰，用户需要反复追问

现象: - 我说”template/ vs templates/ 保持两者” → 用户追问”为何都保留” - 我说”已清理” → 用户追问”还有那些奇怪文件” - 我说”无需 merge” → 用户追问”不同 DB 是否要 merge”

根因: 答案不够具体，缺少”为什么”的证据

正确做法:

❌ 差：template/ 和 templates/ 职责不同，保持两者
✅ 好：
  - template/ = 框架初始化模板（eket init 用）
  - templates/ = 运行时 Handlebars 模板
  - 代码证据: epic_create.rs 第23行引用 templates/jira
  - 删除任一都会导致功能失效

教训: > 每个结论必须附带证据（代码引用/文件大小/数据对比），不能只说”不同”。

P3: 过早说”完成”，实际未验证

现象: - 清理 Root 后说”Root 18 → 6” → 实际还有 test-*.sh + --- Changes --- - 说”confluence 重组完成” → 实际 progress-tracker.md 仍在 confluence/ root - 说”无需 merge” → 没说清楚为什么（子模块 DB 空）

根因: 没有执行最终验证清单

正确做法:

# 宣布"Root 清理完成"前，必须执行：
find . -maxdepth 1 -type f -name "*.md" | wc -l  # 验证数量
find . -maxdepth 1 -type f ! -name ".*"          # 列出所有文件
# 逐个确认每个文件必要性

教训: > 说”完成”前，必须执行验证清单并展示结果，不能凭记忆。

🟡 Important Patterns（有效的做法）

Pattern 1: 用户质疑 = 立刻深度审查信号

触发词: - “还有没有…” - “为什么…” - “那些…干嘛用的” - “你最近怎么回事”

正确响应: 1. 承认疏漏 2. 立刻全面扫描（而非辩解） 3. 生成完整报告 4. 提交修正

本次示例: - 用户: “还有没有其他混乱放置的文件？” - 我: 立刻扫描全部 .md（1,601）、全部 root 文件、全部目录 - 结果: 发现 --- Changes ---、test-*.sh、progress-tracker 错位

教训: > 用户质疑 = 我遗漏了什么。停止当前思路，重新全面扫描。

Pattern 2: 分阶段提交 + 逐层深入

有效做法: 1. confluence/memory/ 重组（3次提交） 2. Root 清理（1次提交） 3. jira/ 归档（2次提交） 4. 去重统一（2次提交） 5. 深度扫描（3次提交）

优势: - 每次提交单一职责 - 可回滚 - 用户可逐步确认

教训: > 大型重组分阶段提交，但必须在开始前完成全局扫描。

Pattern 3: 数据整合前先审查，避免丢失

有效做法: - 合并 .eket/ 前，先检查3个 DB 大小 - 发现 rust/.eket/eket.db 92K → 检查内容 - 确认空数据后才删除

如果直接删除: - 可能丢失重要数据 - 无法恢复

教训: > 删除/合并数据文件前，必须先审查内容和大小，确认无价值才删。

🎓 核心教训（Broadcast to Future）

1. 用户问”A 好了吗” → 扫描 A+B+C 全部相关范围

反例: 只看用户问的具体文件
正例: 扫描整个目录、相关目录、全局模式

2. 说”完成”前 → 执行验证清单并展示结果

反例: 凭记忆判断”应该完成了”
正例: find . -maxdepth 1 -type f 列出所有文件，逐个确认

3. 每个结论 → 附带证据（代码/数据/对比）

反例: “template/ 和 templates/ 不同，保持两者”
正例: “epic_create.rs:23 引用 templates/jira，删除会导致命令失效”

4. 用户质疑 → 停止当前思路，重新全面扫描

反例: 辩解”我已经检查过了”
正例: “抱歉遗漏，立刻全面扫描”

5. 删除数据前 → 审查大小/内容，确认无价值

反例: 看到3个 .eket/ 就删除2个
正例: 先检查 DB 大小（92K），读取 schema，确认空数据

6. 大型重组 → 开始前完成全局扫描和方案设计

反例: 边做边发现问题，用户逐个指出
正例: 第一轮全面扫描 → 识别所有问题 → 提交完整方案 → 用户确认 → 批量执行

7. 职责澄清 → 用代码/配置证明，非主观判断

反例: “我觉得 template/docs/ 应该移到 docs/”
正例: “template/INIT-GUIDE.md 说明 eket init 复制此目录，故应保留”

📝 本次反思

我的问题: - ❌ 片段式响应，缺乏全局规划 - ❌ 过早宣布”完成”，未做最终验证 - ❌ 解释不清晰，用户需反复追问 - ❌ 被动发现问题，而非主动扫描

改进方向: - ✅ 第一时间全面扫描 - ✅ 生成完整报告后再提方案 - ✅ 每个结论附带证据 - ✅ 最终验证清单必须执行

用户的正确引导: - “review、修正、完善、升级的不仅是内容，还有文件名、路径、目录结构” → 点醒我应该看结构 - “template 和 templates 不是还都存在吗” → 指出我没说清楚必要性 - “那些奇怪文件干嘛用的？你最近怎么回事？” → 严厉但必要的提醒
