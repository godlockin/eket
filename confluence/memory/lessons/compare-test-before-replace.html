修改在用逻辑：对比测试后再替换

核心原则

修改正在生产/运行中的逻辑（API、服务、CLI 命令、heartbeat 循环等），不能改完直接替换。必须先用隔离环境跑新旧两个版本做对比，确认所有交互行为一致（或符合改动目标）后，才能更新、替换和部署。

隔离方式（按场景选择）

场景 隔离手段
HTTP 服务 不同端口启动新旧版本（如旧 :9877，新 :9878），用 curl/测试脚本对比响应
CLI 命令 不同 binary 路径（target/debug/eket-new vs ~/.local/bin/eket），同一数据集跑对比
heartbeat / 长驻进程 不同 DB 文件 + 不同 mailbox 目录，跑新版本不影响生产状态
SQLite 操作 复制一份 DB 文件给新版本，旧版本继续用原文件
文件队列 不同 project root / .eket/ 目录，互不干扰

对比测试要检查的内容

  1. 正常路径：相同输入 → 输出结构一致（JSON schema、字段名、状态迁移）
  2. 边界条件：空队列、无 slaver、依赖未解除等 → 行为与旧版本一致或符合改动说明
  3. 错误路径：文件缺失、DB 锁、网络超时 → 新版本不引入新的 panic / silent failure
  4. 性能：关键路径延迟不显著劣化

替换时机

所有对比测试通过后，按以下顺序操作： 1. 停旧版本进程 2. 替换 binary / 配置 3. 用生产数据启动新版本 4. 监控首个完整周期（心跳、派送、完成）无异常

大 release 前：独立专家组评审门卫

在任何大 release 合并/部署之前，必须召唤一个由多个专家组成的独立评审专家组，进行刻薄的、吹毛求疵的 review 和测试。有任何问题毫不留情地指出，要求修改，不通过不得合并。

专家组构成（建议）

专家 职责
🏗️ 架构师 接口设计、模块边界、依赖合理性、是否引入循环依赖
🔒 安全专家 文件写入权限、路径穿越、敏感信息泄漏、竞态条件
🧪 QA 工程师 测试覆盖盲区、边界条件、回归风险
🖥️ 后端工程师 错误处理完整性、性能回归、并发安全、数据一致性
⚙️ DevOps 部署影响、配置变更、降级路径、监控可观测性

评审清单（每个专家独立完成）

触发时机

  • 影响超过 2 个命令的改动

  • 任何状态机（Saga、heartbeat、dispatch 流程）变更

  • 新增持久化文件格式

  • 依赖链变更（新 crate、新外部服务）

  • 改动后直接替换，若有行为差异，生产 tickets 状态机会走错路径（如 waiting 条目无法清理、unblocked 队列重复处理）

  • 多智能体环境中，一个节点的行为变化会级联影响其他节点的状态判断

  • 单测通过 ≠ 集成行为一致；对比测试是单测之外的必要验证层

适用场景

  • heartbeat 循环逻辑变更
  • task:claim / task:complete saga 步骤变更
  • 任何读写 .eket/state/ 文件的逻辑变更
  • DB schema 或查询逻辑变更
  • slaver 匹配 / 派送逻辑变更（本次 TASK-246~247 即属此类）
