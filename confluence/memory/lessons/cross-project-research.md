# 借鉴外部项目的经验教训

## 背景

Round 20~22 研究了 Archon 等外部 AI 智能体框架，提炼 ACI 白名单、DAG 可视化、Artifact Schema、工作流模板等概念，整合进 EKET。

## 经验教训

### ✅ 有效做法

**目标导向阅读，不读全代码**
- 先看 README + 目录结构（5 分钟），判断是否值得深入
- 只读与当前需求直接相关的模块（如只需要理解 DAG 调度，就只读调度核心）
- 用 grep/symbol search 定位核心抽象，不逐行阅读

**提炼模式而非复制实现**
- 记录"这个项目解决了什么问题、用什么模式"，而非具体代码
- 借鉴点写成 research/ 文档（概念 + 适配说明），与实现解耦
- 好的借鉴结果：一段 EKET-适配的伪代码 + 动机说明

**设定借鉴上限**
- 每次研究会话最多 2~3 个借鉴点，超出放入 backlog
- 借鉴点必须对应已存在的 EKET ticket 或立即创建 ticket
- 无 ticket 的借鉴点 = 未落地的想法，不算研究成果

### ❌ 踩坑记录

**研究瘫痪**
- 症状：读了 5+ 个外部项目，写了大量笔记，没有任何 EKET 代码变化
- 根因：没有设定"研究结束条件"（何时停止研究，何时开始实现）
- 代价：浪费 context，借鉴内容在会话结束后遗失

**过度借鉴导致设计漂移**
- 症状：为了"对齐外部最佳实践"，引入 EKET 当前不需要的复杂度
- 根因：混淆了"学习"和"照搬"
- 案例：Archon 的 Redis Streams 事件总线对当前规模是过度设计

**借鉴内容没有适配说明**
- 症状：research/ 文档直接粘贴外部代码，无 EKET 上下文
- 后果：下一个 instance 看到这段内容不知道是否已实现、是否适用

### 💡 下次建议

1. **研究前写"研究问题"**：明确"我想从这个项目学什么"，研究完只需回答这个问题
2. **研究结束条件**：找到 1 个可落地的借鉴点 OR 确认"无可借鉴"，立即停止
3. **research/ 文件模板**：
   ```
   ## 来源
   ## 解决的问题
   ## 核心模式
   ## EKET 适配方案
   ## 实现状态（未开始/已开始/已完成）
   ```
4. **连续读 3 个外部文件无产出 = 触发研究瘫痪警报**，强制切换到实现模式
