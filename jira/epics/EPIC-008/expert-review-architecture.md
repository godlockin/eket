## 🏗️ 架构师评审

### 观察
1. **核心矛盾**: 当前 "一次性 PR 提交" 设计与 "增量容错" 需求冲突
2. **技术债**: 依赖 .output transcript JSONL（41831 tokens）反映 Slaver 状态 → 不可扩展
3. **现有机制**: 已有 `jira/tickets/<id>/analysis-report.md`（仅覆盖分析阶段），缺乏实现阶段的 checkpointing

### 担忧
1. **Git history 膨胀**: 若每个函数提交一次，100 函数 = 100 commits，污染 PR diff
2. **状态一致性**: Progress log 写成功但 git commit 失败 → 状态分裂
3. **过度设计**: 引入复杂状态机（10+ 状态）会增加维护成本

### 建议

**方案 A：阶段式 Checkpoint（推荐）**
```
├── jira/tickets/TASK-NNN/
│   ├── analysis-report.md          # 已有
│   ├── design-decisions.md         # 新增：技术选型 + 模块拆解
│   ├── progress.md                 # 新增：实时进度
│   └── checkpoints/                # 新增：阶段产物
│       ├── phase-1-analysis.done   # 空文件标记
│       ├── phase-2-design.done
│       ├── phase-3-impl-core.done
│       └── phase-4-tests.done
```

**Progress.md 格式**（轻量 Markdown，非 JSON）:
```markdown
# Task Progress: TASK-NNN

**Last Update**: 2026-05-14 14:45:30 +0800  
**Slaver**: slaver-005  
**Current Phase**: `implementation`

## Completed
- [x] Analysis (2h) — commit: `abc123`
- [x] Design (1h) — doc: `design-decisions.md`
- [x] Implement AC-1 — files: `src/auth.rs`, test: ✅
- [x] Implement AC-2 (50%) — files: `src/db.rs` (WIP)

## Next Steps
- [ ] Complete AC-2 validation logic
- [ ] Implement AC-3 error handling
- [ ] Write integration test

## Blockers
- None

---
Generated by `eket-slaver-checkpoint`
```

**Git 策略**：
- **不在 feature 分支频繁 commit**（保持 PR 简洁）
- **使用独立 `checkpoint/<task-id>` 分支**：
  ```bash
  # Slaver 每完成一个 AC 后
  git checkout -b checkpoint/TASK-640 || git checkout checkpoint/TASK-640
  git add jira/tickets/TASK-640/progress.md src/new_file.rs
  git commit --amend -m "Checkpoint: TASK-640 AC-2 50%"
  git push -f origin checkpoint/TASK-640
  ```
- **最终 PR 时**：仅合并 feature 分支（不包含 checkpoint commits）

**恢复流程**:
```bash
# 新 Slaver 接手
eket task:claim TASK-640 --resume
→ 读 progress.md + 检出 checkpoint/TASK-640 分支
→ 自动显示：已完成文件 + 剩余 AC + 下一步
```

## 🔧 DevOps 评审

### 观察
1. **CI/CD 影响**: 每次 checkpoint commit 会触发 CI 吗？（成本/时间）
2. **分支管理**: `checkpoint/*` 分支生命周期？何时清理？
3. **Git 仓库膨胀**: checkpoint 分支 force-push 会压缩 history，但仍占 reflog 空间

### 担忧
1. **CI 风暴**: 若 Slaver 每 5 分钟 checkpoint 一次，10 个 Slaver = 每分钟 2 次 CI 触发
2. **Orphan 分支**: Slaver 崩溃后 `checkpoint/TASK-XXX` 分支可能永远残留
3. **权限问题**: Slaver 需要 push 权限到 checkpoint 分支（当前仅 Master 有 main 权限）

### 建议

**CI 优化**:
```yaml
# .github/workflows/ci.yml
on:
  push:
    branches-ignore:
      - 'checkpoint/**'   # checkpoint 分支不触发 CI
```

**Checkpoint 分支生命周期**:
```bash
# 自动清理规则（Master 执行）
eket gc:checkpoints --older-than 7d --merged-only
→ 删除 PR 已合并 + 创建时间 > 7 天的 checkpoint 分支
```

**权限方案**（最小权限原则）:
- Slaver 仅能 push `checkpoint/<task-id>`（通过 GitHub branch protection rule）
- Master 能 push `feature/*`, `testing`, `main`
- 所有人只读 `main`

**替代方案**（若 git 方案过重）:
使用 **文件系统 + rsync** 而非 git checkpoint：
```bash
# Slaver 每 30s 异步写
echo "$(date -Iseconds),impl_ac2,src/db.rs" >> jira/tickets/TASK-640/progress.log

# Master 读进度
tail -f jira/tickets/TASK-640/progress.log
```
**优点**: 零 git 开销；**缺点**: 文件未版本化，丢失无法恢复

## 🧪 QA 评审

### 观察
1. **验收可测性**: AC-2 "新 Slaver 快速恢复" 如何自动化测试？
2. **边界 case**:
   - Progress 文件损坏（写一半 Slaver 崩溃）
   - Progress 显示 100% 但代码未 commit（Slaver 伪造）
   - 两个 Slaver 同时 claim 同一 task（竞态）
3. **现有测试覆盖**: `node/tests/` 中无 checkpoint 相关测试

### 担忧
1. **脏数据验证**: Progress 声称完成 AC-1，但 test 未通过 → 谁来验证？
2. **恢复正确性**: 新 Slaver resume 后可能基于错误的 checkpoint 继续，累积错误
3. **手动验证成本**: 若每次恢复需 Master 人工检查 progress.md 真实性，未真正省时间

### 建议

**Progress 验证机制**（防伪造）:
```markdown
## Completed
- [x] Implement AC-1
  - files: `src/auth.rs` (sha256: e3b0c44298fc1c149afbf)
  - test: `npm test -- auth.test.ts` ✅ (exit 0, 2026-05-14 14:30)
  - commit: `abc123def` (verifiable via `git show`)
```

**自动验证命令**:
```bash
eket task:verify TASK-640
→ 检查:
  1. progress.md 列出的文件是否真实存在
  2. 列出的 commit hash 是否可 git show
  3. 列出的 test 是否可重跑并通过
→ 输出: ✅ verified / ⚠️ inconsistent (显示哪里不匹配)
```

**测试策略**:

**边界 case 处理**:
| Case | 检测方式 | 处理 |
|------|---------|------|
| Progress 文件损坏 | 解析失败 / 无 timestamp | 回退到 `.bak` 备份（每次写前备份） |
| Progress 与 git 不一致 | `eket task:verify` 交叉验证 | 标记为 `verification_failed`，Master 介入 |
| 并发写冲突 | 检测 Slaver ID 不匹配 | 后写者失败并报错（不覆盖） |

## 💻 后端工程师评审

### 观察
1. **Slaver 代码侵入性**: 需要在现有 Slaver 逻辑中**多处插入 checkpoint 调用**
2. **当前 Slaver 流程**:
   ```
   claim → analyze → implement → test → submit PR
   ```
   **需插入 checkpoint 的位置**:
   ```
   claim → [checkpoint: analysis_start]
        → analyze → [checkpoint: analysis_done]
        → implement AC-1 → [checkpoint: ac1_done]
        → implement AC-2 → [checkpoint: ac2_done]
        → test → [checkpoint: test_done]
        → submit PR
   ```
3. **异步写性能**: 若同步写 progress.md（每次 `fs.writeFileSync`），会阻塞 Slaver 执行

### 担忧
1. **代码耦合**: 在业务逻辑中混杂 checkpoint 调用 → 难维护
2. **错误处理**: Checkpoint 写失败（磁盘满/权限）如何处理？继续执行 vs 中止？
3. **幂等性**: Slaver 重试同一 AC 时，checkpoint 应覆盖还是追加？

### 建议

**最小侵入式设计**（装饰器模式）:
```javascript
// node/src/core/slaver.js
class Slaver {
  async executeTask(taskId) {
    const tracker = new ProgressTracker(taskId);  // 包装器

    await tracker.checkpoint('analysis_start');
    const analysis = await this.analyze();
    await tracker.checkpoint('analysis_done', { artifact: 'analysis-report.md' });

    for (const ac of analysis.acceptanceCriteria) {
      await tracker.checkpoint(`ac_${ac.id}_start`);
      const files = await this.implementAC(ac);
      await tracker.checkpoint(`ac_${ac.id}_done`, { files, tests: ac.testResults });
    }

    await tracker.checkpoint('ready_for_pr');
  }
}

// node/src/core/progress-tracker.js
class ProgressTracker {
  constructor(taskId) {
    this.taskId = taskId;
    this.buffer = [];  // 内存缓存
    this.flushInterval = setInterval(() => this.flush(), 30000);  // 每 30s flush
  }

  async checkpoint(phase, metadata = {}) {
    this.buffer.push({
      timestamp: new Date().toISOString(),
      phase,
      ...metadata
    });
    // 关键里程碑同步写（防崩溃丢失）
    if (['analysis_done', 'ready_for_pr'].includes(phase)) {
      await this.flush();
    }
  }

  async flush() {
    if (this.buffer.length === 0) return;
    const progressPath = `jira/tickets/${this.taskId}/progress.md`;
    const content = this.renderMarkdown();
    await atomicWrite(progressPath, content);  // 原子写（先 .tmp 再 mv）
    this.buffer = [];
  }
}
```

**错误处理策略**:
```javascript
async checkpoint(phase, metadata) {
  try {
    // ... 正常逻辑
  } catch (error) {
    console.warn(`Checkpoint failed (non-critical): ${error.message}`);
    // 继续执行，不阻塞主流程（checkpoint 是辅助功能）
    // 但记录到 .eket/logs/checkpoint-failures.log 供 Master 监控
  }
}
```

**幂等性**（覆盖模式）:
```markdown
# Progress 文件结构
## Completed (最新状态)
- [x] AC-1 (updated: 2026-05-14 15:00)  ← 若重复 checkpoint，覆盖 timestamp
- [x] AC-2 (updated: 2026-05-14 15:30)

## History (可选，供 debug)
- 2026-05-14 14:30 - AC-1 started
- 2026-05-14 15:00 - AC-1 completed
- 2026-05-14 15:10 - AC-1 completed (retry)  ← 追加记录
```

## 🔍 Master 决策汇总

### 最终技术方案（综合专家意见）

**采纳决策**:
1. ✅ **架构师方案 A（阶段式 Checkpoint）** — 使用 `progress.md` + `checkpoints/` 目录
2. ✅ **DevOps 建议（独立 checkpoint 分支）** — 避免污染 feature 分支 PR diff
3. ✅ **QA 验证机制** — 实现 `eket task:verify` 防伪造
4. ✅ **后端装饰器模式** — 最小侵入，30s 异步 flush

**驳回/延后决策**:
- ❌ **DevOps 替代方案（纯文件系统）** — 驳回理由：无版本化，无法回滚到历史 checkpoint
- ⏳ **QA History 追加模式** — 延后到 phase 2（当前版本仅保留最新状态，降低复杂度）

**修改决策**:
- 🔄 **架构师 Git 策略修正**：不使用 `git commit --amend`（force-push 有风险），改用**正常 commit + squash on merge**：
  ```bash
  # Slaver 每完成一个 AC
  git add progress.md src/new_file.rs
  git commit -m "Checkpoint: AC-2 done"
  git push origin checkpoint/TASK-640
  
  # PR 合并时 Master 执行
  git merge --squash checkpoint/TASK-640
  ```

## 技术栈选型

| 模块 | 技术选择 | 理由 |
|------|---------|------|
| **进度文件格式** | Markdown | 人类可读、Git diff 友好、无需解析库 |
| **Checkpoint 存储** | Git 分支 `checkpoint/<task-id>` | 版本化、可回滚、与现有 git workflow 一致 |
| **进度更新机制** | 30s 异步 flush + 关键节点同步写 | 平衡性能与数据安全 |
| **验证工具** | Node.js CLI `eket task:verify` | 复用现有 eket 命令框架 |
| **原子写** | `fs.writeFile` → `fs.rename` | 防止写一半崩溃导致损坏 |
