## 1. 需求概述

### 1.1 背景与动机

**借鉴来源**：OpenClaw "AI 自己创建 Skill" 机制 + EKET 现有"复盘后经验沉淀到 `confluence/memory/`" 规定。

**当前痛点**：Slaver 复盘写的经验是 Markdown 文档，人类可读，但下一个 Slaver 实例不会主动检索它，经验沉淀是单向的、被动的。

**目标**：让复盘产出的通用解法变成**可执行 Skill**，注册到 `node/src/skills/`，下次类似任务自动调用，实现经验的主动复用。

### 1.2 功能描述

Slaver 完成 ticket 后，在复盘阶段：

1. 调用 `skill:extract` 命令，传入本次 ticket 的关键步骤 + 解法描述
2. 框架用 LLM（或规则）提炼出 Skill 元数据（name/category/trigger/steps）
3. 自动在 `node/src/skills/{category}/` 生成 `.ts` 文件并调用 `SkillsRegistry.register()`
4. Master 下次分配同类任务时，`task:claim` 输出"检测到相关 Skill: {name}"

### 1.3 验收标准

- [ ] `node dist/index.js skill:extract --ticket TASK-XXX --summary "解法描述"` 生成 skill 文件
- [ ] 生成的 skill 通过 `SkillsRegistry.register()` 不报错
- [ ] `node dist/index.js skill:list` 能列出自动生成的 skill
- [ ] `node dist/index.js task:claim TASK-YYY` 若存在相关 skill，输出提示
- [ ] 生成的 skill 文件通过 TypeScript 编译（`npm run build` 0 error）
- [ ] 新增 5 个测试（skill 提炼、注册、检索、task:claim 提示）
- [ ] 验收命令：
  ```bash

  node dist/index.js skill:list | grep zod
  cd node && npm run build && npm test 2>&1 | tail -3
  ```

### 2.2 Skill 提炼流程

```
Slaver 输入：
  --ticket TASK-042
  --summary "Zod schema 驱动消息格式"
  --steps "1.定义 schema 2.safeParse 3.写入 JSON"
  --category "requirements"  (可选，默认推断)
        ↓
skill-generator.ts:
  1. 读取 ticket 文件获取 type/tags
  2. 根据 tags 推断 category（schema→requirements, hook→testing 等）
  3. 生成 Skill 骨架代码（静态模板填充）
  4. 写入 node/src/skills/{category}/auto-{name}.ts
  5. 调用 SkillsRegistry.register(skill)
  6. 写入 confluence/memory/skills/{name}.md（人类可读版本）
```

### 2.3 生成的 Skill 格式（静态模板）

```typescript
// node/src/skills/requirements/auto-zod-message-schema.ts
// AUTO-GENERATED by skill:extract — TASK-042 — 2026-04-16
// DO NOT EDIT manually; regenerate with skill:extract

import type { Skill } from '../types.js';

export const zodMessageSchemaSkill: Skill = {
  id: 'zod-message-schema',
  name: 'Zod Message Schema Validation',
  category: 'requirements',
  description: 'Zod schema 驱动消息格式，双向 validate，Fail Fast',
  trigger: ['schema', 'message', 'validate', 'zod'],
  steps: [
    '1. 用 z.object() 定义消息结构',
    '2. safeParse() 在发送/读取时双向检查',
    '3. 失败返回 INVALID_MESSAGE Result 而非 throw',
  ],
  relatedTicket: 'TASK-042',
  createdAt: '2026-04-16',
};

export default zodMessageSchemaSkill;
```

### 2.4 task:claim 集成

```typescript
// task:claim 时，对 ticket 的 tags 做 skill 匹配
const relatedSkills = registry.findByTriggers(ticket.tags);
if (relatedSkills.length > 0) {
  console.log(`[Skills] 检测到相关 Skill: ${relatedSkills.map(s => s.name).join(', ')}`);
  console.log(`[Skills] 参考: node dist/index.js skill:show ${relatedSkills[0].id}`);
}
```

### 2.5 设计决策

1. **静态模板而非 LLM 生成代码**：避免 LLM 幻觉写出无法编译的 TS 代码；模板保证类型安全
2. **auto- 前缀**：区分人工 skill 和自动生成 skill，方便清理和审查
3. **不自动 import 到 registry**：生成文件，但需要下次 `npm run build` + 重启才生效；避免运行时热加载带来的不稳定
