### 2.2 各专家借口候选（≥5 条，按职责高频出错点设计）

#### 🏗️ architect.md（架构师）

| 借口 | 反驳 |
|------|------|
| "这个模块先耦合一下，后面再拆" | 耦合边界一旦上线就有调用方，"后面再拆"的实际概率 < 5%；边界设计必须 Day 1 |
| "先跑起来再考虑架构" | 没有架构的代码是技术债的温床；方向错跑得越快越难回头 |
| "这个技术选型大家都在用，应该没问题" | 流行 ≠ 匹配场景；选型必须对应本项目的规模、团队和约束 |
| "架构图可以等实现稳定后再补" | 没有架构图的系统靠口口相传；下一个接手的人 = 你三个月后的自己 |
| "这是业务逻辑，不是架构问题" | 业务逻辑持续变化是架构稳定性的最大压力源；隔离变化点就是架构核心职责 |
| "重构太贵了，维持现状吧" | 架构债的利息是指数增长的；当下的"太贵"换来的是未来更贵的全量重写 |

#### 🖥️ backend.md（后端工程师）

| 借口 | 反驳 |
|------|------|
| "SQL 注入不会有问题，我们内部系统" | 内部系统不是免于被攻击的理由；最危险的攻击往往来自内部或已渗透的账户 |
| "这个接口暂时不做鉴权，后面加" | 无鉴权接口一旦上线就是敞开的门；「后面加」意味着所有调用方都要改，成本翻倍 |
| "先不加索引，数据量小时没问题" | 数据量增长是早晚的事；数据上涨后加索引要锁表或迁移，风险远高于现在 |
| "事务太复杂，业务兜底就好" | 业务兜底是人工 + 易漏；数据库事务是机器级保证，不在同一量级 |
| "N+1 查询先上线，下个版本优化" | N+1 在规模上是 O(N) 的数据库请求；上线后找到并改动的成本是设计期的 10 倍 |
| "密码明文存没事，内网安全" | 一旦数据库被拖库，明文密码直接暴露用户所有账号；散列存储是基本线 |

#### 🎨 frontend.md（前端工程师）

| 借口 | 反驳 |
|------|------|
| "这个列表先全量渲染，数据不会太多" | 用户数据量你控制不了；10 条 → 10000 条是常见演进，虚拟滚动要在设计阶段考虑 |
| "状态放全局 store 更方便，不用传 props" | 全局状态是隐式依赖；组件间耦合无声无息，调试时没人知道谁改了什么 |
| "可访问性不是这个版本的需求" | 可访问性是基本人权，不是 feature toggle；事后改动需要全面重构 HTML 结构 |
| "打包体积大点没关系，带宽便宜" | 首屏加载直接影响用户留存；3G/弱网用户体验 ≠ 办公室 WiFi 体验 |
| "这段逻辑先 copy 一份，下次再抽象" | 每次「先 copy」都在制造未来的蝴蝶效应；一个 bug 要改两个地方且必有一处被遗忘 |
| "没有 memo/useMemo 也 OK，重渲染没那么频繁" | 重渲染成本在复杂组件树下是累积的；profiling 成本远高于一开始加 memo |

#### 📋 product.md（产品经理）

| 借口 | 反驳 |
|------|------|
| "这个需求先做，需求文档写完再补" | 没有文档的需求在开发阶段会产生歧义；每次口头澄清都是被稀释的需求精度 |
| "用户要什么我们就做什么" | 用户说的是「症状」，不是「解法」；产品的价值是找到真正的 Jobs-to-be-Done |
| "这个功能先上再看数据" | 没有验收指标的功能无法判断成败；上线前必须定义「成功」的度量标准 |
| "优先级可以都是 P0，都很重要" | 所有事都重要 = 所有事都不重要；优先级的核心就是做取舍，拒绝排序是逃避决策 |
| "范围蔓延一点点无所谓" | 每一个「小」需求都消耗开发窗口；EPIC 范围蔓延 20% 通常延期 50% |
| "技术实现细节是工程师的事，产品不用懂" | 不了解技术约束的产品会制造无法兑现的承诺；至少需要理解边界和成本数量级 |

#### 🖌️ ux.md（UX 设计师）

| 借口 | 反驳 |
|------|------|
| "这个交互看起来很直观，不需要用研" | 「对设计师直观」≠「对用户直观」；没有数据的直觉是最危险的假设 |
| "设计稿只要视觉好看就行，细节开发自己处理" | 开发阶段的「自行处理」= 交互随意降级；边缘 case 和错误状态必须在设计稿中明确 |
| "移动端适配之后再做，先做桌面端" | 超过 60% 的用户在移动端；后适配意味着重构组件布局，成本是设计期的 3-5 倍 |
| "这个错误提示随便写个英文就行" | 错误提示是用户压力最高的时刻；模糊的错误提示直接损害信任度和转化率 |
| "设计系统先跳过，做完功能再统一" | 没有设计系统的界面是视觉债；每次新页面都在重复造轮子 + 引入不一致 |
| "用户会适应这个交互的" | 用户不会主动学习难用的产品；他们会直接离开或永不再来 |

## 5. 风险评估

| 风险项 | 可能性 | 影响 | 缓解措施 |
|--------|--------|------|----------|
| **跨平台 grep 差异**：macOS `grep` vs Linux `grep` 计数行为不一致（尤其 `-c "^\| "`）| M | M | 使用 `grep -cP` 或 `grep -c "^\|"` POSIX 写法；CI 在 Linux 跑，本地 macOS 预验证时注意 `\|` 转义 |
| **借口过度本地化**：写的借口太针对 EKET 框架，换场景无法识别 | M | L | 借口表顶部加注"非穷举，LLM 借口措辞可变"；专家组审查时关注通用性 |
| **PR 行数超标**：5 个文件各追加 ~40 行 = ~200 行，加上 frontmatter 可选字段可能超 300 | L | M | 不加 frontmatter 字段（单独 ticket），仅追加表格，预估 5×40=200 行，安全 |
| **YAML frontmatter 结构破坏**：Edit 工具定位时误伤 frontmatter | L | H | 锚点选 closing ` ``` ` 行，不在 YAML 内部操作；修改后立即用 `grep "^id:"` 验证 frontmatter 完整 |
| **借口内容重复**：AC-3 要求各专家借口不可复用同一模板，评审可能主观 | M | M | 本报告已为每个专家设计独立借口集，无跨文件重复；PR 描述中附差异说明 |
