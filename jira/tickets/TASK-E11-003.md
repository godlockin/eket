## 背景

Understand-Anything 的增量更新策略：

1. **git diff** — 检测文件级变更
2. **fingerprint** — 基于 AST 结构的哈希，判断是否需要重新分析
3. **变更分类** — STRUCTURAL（需重分析）vs COSMETIC（跳过）

这比纯 git diff 更精准：
- 只改注释/空格 → COSMETIC → 跳过
- 改了函数签名 → STRUCTURAL → 重分析

## 目标

在 EKET 中实现 fingerprint 机制，支持增量分析。

## 技术方案

### 1. Fingerprint 计算

```rust
use sha2::{Sha256, Digest};

#[derive(Serialize, Deserialize)]
pub struct FileFingerprint {
    pub path: String,
    pub content_hash: String,      // 文件内容 SHA256
    pub structure_hash: String,    // AST 结构 SHA256
    pub last_analyzed: String,     // ISO timestamp
}

pub fn compute_fingerprint(path: &Path, analysis: &StructuralAnalysis) -> FileFingerprint {
    let content = std::fs::read_to_string(path).unwrap();
    let content_hash = hex::encode(Sha256::digest(content.as_bytes()));
    
    // 结构哈希：函数签名 + 类签名 + 导入列表
    let structure_str = format!(
        "{}|{}|{}",
        analysis.functions.iter().map(|f| format!("{}({})", f.name, f.params.join(","))).collect::<Vec<_>>().join(";"),
        analysis.classes.iter().map(|c| format!("{}[{}]", c.name, c.methods.join(","))).collect::<Vec<_>>().join(";"),
        analysis.imports.iter().map(|i| &i.source).collect::<Vec<_>>().join(";"),
    );
    let structure_hash = hex::encode(Sha256::digest(structure_str.as_bytes()));
    
    FileFingerprint {
        path: path.to_string_lossy().to_string(),
        content_hash,
        structure_hash,
        last_analyzed: chrono::Utc::now().to_rfc3339(),
    }
}
```

### 2. 变更分类

```rust
#[derive(Debug, PartialEq)]
pub enum ChangeType {
    None,       // 无变化
    Cosmetic,   // 只有注释/空格变化
    Structural, // 结构变化，需重分析
    New,        // 新文件
    Deleted,    // 删除的文件
}

pub fn classify_change(
    path: &Path,
    old_fp: Option<&FileFingerprint>,
    new_analysis: &StructuralAnalysis,
) -> ChangeType {
    let new_fp = compute_fingerprint(path, new_analysis);
    
    match old_fp {
        None => ChangeType::New,
        Some(old) => {
            if old.content_hash == new_fp.content_hash {
                ChangeType::None
            } else if old.structure_hash == new_fp.structure_hash {
                ChangeType::Cosmetic
            } else {
                ChangeType::Structural
            }
        }
    }
}
```

### 3. 增量分析流程

```rust
pub async fn incremental_analyze(
    project_root: &Path,
    baseline_fp: &HashMap<String, FileFingerprint>,
) -> IncrementalResult {
    // 1. git diff 获取变更文件
    let changed_files = git_diff_files(project_root)?;
    
    // 2. 分类每个文件
    let mut to_analyze = vec![];
    let mut skipped = vec![];
    
    for path in changed_files {
        let analysis = analyze_structure(&path)?;
        let change_type = classify_change(&path, baseline_fp.get(&path.to_string_lossy().to_string()), &analysis);
        
        match change_type {
            ChangeType::Structural | ChangeType::New => {
                to_analyze.push((path, analysis));
            }
            ChangeType::Cosmetic | ChangeType::None => {
                skipped.push(path);
            }
            ChangeType::Deleted => {
                // 标记删除
            }
        }
    }
    
    // 3. 只分析 STRUCTURAL 变更
    IncrementalResult {
        to_analyze,
        skipped,
        total_changed: changed_files.len(),
    }
}
```

## 专家分析

### 数据张 (Data Zhang) — 数据工程视角

**分析日期**: 2026-05-24

#### 数据流特征

fingerprint 本质是 **增量 CDC (Change Data Capture)** 系统：
- **Source**: 文件系统 + git diff
- **Transform**: AST 解析 → 哈希计算 → 变更分类  
- **Sink**: 持久化索引 + 增量分析触发

核心特征：
1. **写少读多**: fingerprint 只在 commit/build 时生成，每次 diff 都要查询
2. **时间序列性**: 每个 commit 一个 baseline，支持历史回溯
3. **空间局部性**: 同目录文件往往一起变更，可优化批量读取

#### 存储策略 (三层架构)

```
Layer 1: 热层 (moka LRU Cache)
├── 最近 1000 个文件的 fingerprint
├── content_hash → structure_hash 映射
└── TTL: 30min，命中率预期 85%+

Layer 2: 温层 (SQLite + B-Tree Index)
├── fingerprints 表
├── B-Tree 索引: (path), (content_hash), (commit_sha, path)
└── 分区键: commit_sha (支持历史版本查询)

Layer 3: 冷层 (JSON 归档)
├── .eket/fingerprints/{commit_sha}.json.gz
├── 压缩后 ~10KB / 1000 files
└── 用于跨机器共享 (git LFS / artifact)
```

**推荐 SQLite Schema** (加入现有 `eket.db`):

```sql
CREATE TABLE file_fingerprints (
    id INTEGER PRIMARY KEY,
    path TEXT NOT NULL,
    content_hash CHAR(64) NOT NULL,  -- SHA256 hex
    structure_hash CHAR(64) NOT NULL,
    commit_sha CHAR(40),
    analyzed_at INTEGER NOT NULL,    -- Unix timestamp
    file_size INTEGER,
    line_count INTEGER,
    UNIQUE(path, commit_sha)
);

CREATE INDEX idx_fp_path ON file_fingerprints(path);
CREATE INDEX idx_fp_content ON file_fingerprints(content_hash);
CREATE INDEX idx_fp_commit ON file_fingerprints(commit_sha, path);
```

#### 计算成本分析

| 操作 | 成本 | 优化策略 |
|------|------|----------|
| content_hash | O(文件大小), ~1ms/100KB | 流式 SHA256，内存映射大文件 |
| structure_hash | O(AST 节点数), ~5-50ms | **瓶颈!** tree-sitter 解析 |
| classify_change | O(1) 哈希比较 | 极快，可忽略 |

#### 性能优化建议

**1. 两阶段过滤** (关键优化，节省 60-70% AST 解析):

```
git diff → content_hash 变化? → 否 → NONE (跳过 AST)
                 ↓是
           structure_hash 变化? → 否 → COSMETIC
                 ↓是
              STRUCTURAL
```

**2. 批量并行计算**:

```rust
// 并行计算 content_hash (I/O bound)
let content_hashes: Vec<_> = rayon::iter(files)
    .map(|f| (f, sha256_file(f)))
    .collect();

// 只对 content 变化的文件做 AST (CPU bound)
let changed: Vec<_> = content_hashes
    .into_par_iter()
    .filter(|(f, h)| baseline.get(f).map_or(true, |old| old != h))
    .map(|(f, _)| compute_structure_hash(f))
    .collect();
```

**3. Structure Hash 稳定化** (避免重构误判):

```rust
// 用 Merkle Tree 结构，顺序无关化
let mut sigs: Vec<_> = functions.iter()
    .map(|f| format!("{}({})", f.name, f.params.join(",")))
    .collect();
sigs.sort();  // 顺序无关化
let func_hash = sha256(sigs.join("|"));

// structure_hash = sha256(func_hash + class_hash + import_hash)
```

**4. 命中率监控指标**:

```
hit_rate = COSMETIC / (COSMETIC + STRUCTURAL)

- < 20%: structure_hash 粒度太细，需调整
- > 80%: 可能漏检语义变化
- 目标: 30-50% (正常开发模式)
```

#### 补充的 CLI 接口

```bash

eket fingerprint:build src/ --commit HEAD

# 增量 diff（两阶段过滤）
eket fingerprint:diff --baseline HEAD~1 --target HEAD

# 统计命中率
eket fingerprint:stats --since "2026-05-01"
# 输出: total=1234, structural=156, cosmetic=478, skipped=600, hit_rate=75.4%
```

#### 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| tree-sitter 解析失败 | 无法生成 structure_hash | Fallback 到 content_hash only, 标记 `degraded` |
| 大文件阻塞 | 单文件 >100MB 导致 OOM | 流式哈希 + 跳过阈值 (--max-file-size 10MB) |
| SQLite 并发写入 | 多 slaver 同时 build | WAL 模式 + 写入队列 (channel) |
| 历史 baseline 膨胀 | 每 commit 存全量 | 增量存储: 只存 diff + 定期 compaction |
| Structure hash 不稳定 | 重构误判 STRUCTURAL | Merkle tree + 配置化忽略项 |

#### 与现有系统集成点

- 复用 `eket.db` SQLite 实例 (WAL 模式已启用)
- 复用 `eket-engine` 的 `moka` 缓存基础设施
- 复用 `KnowledgeBase` 的 schema pattern
- Git Hook 集成: `.git/hooks/post-commit` 自动 build

- **三层存储**: DashMap (L1 热) → SQLite (L2 温) → JSON (L3 冷)
- **支持语言**: Rust, TypeScript/JavaScript, Python, Go (via tree-sitter)
