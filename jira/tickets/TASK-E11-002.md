## 背景

Understand-Anything 处理大代码库时的核心策略：

1. **语义批次分割** — 按依赖关系聚类，相关文件放同一批
2. **neighborMap** — 记录跨批次依赖，让 agent 能生成跨批次边
3. **并发控制** — 最多 5 个 file-analyzer 并发

这解决了两个问题：
- 单次上下文不爆炸（每批 ≤60 节点）
- 跨批次边不丢失（neighborMap 提供上下文）

## 目标

在 EKET 中实现类似机制，用于大 ticket 拆解和并行 Slaver 执行。

## 技术方案

### 1. 批次分割算法

```rust
#[derive(Serialize, Deserialize)]
pub struct Batch {
    pub batch_index: u32,
    pub files: Vec<FileEntry>,
    pub batch_import_data: HashMap<String, Vec<String>>,  // file -> imports
    pub neighbor_map: HashMap<String, NeighborInfo>,      // 跨批次邻居
}

#[derive(Serialize, Deserialize)]
pub struct NeighborInfo {
    pub path: String,
    pub batch_index: u32,
    pub symbols: Vec<String>,  // 导出的符号
}

pub fn compute_batches(
    files: Vec<FileEntry>,
    import_map: HashMap<String, Vec<String>>,
    max_nodes_per_batch: usize,  // 默认 60
) -> Vec<Batch> {
    // 1. 构建依赖图
    let dep_graph = build_dependency_graph(&files, &import_map);
    
    // 2. 社区检测（Louvain 或简单连通分量）
    let communities = detect_communities(&dep_graph);
    
    // 3. 按社区分批，超大社区再拆
    let mut batches = vec![];
    for community in communities {
        if community.len() <= max_nodes_per_batch {
            batches.push(community);
        } else {
            // 超大社区按文件路径前缀再分
            batches.extend(split_by_path_prefix(community, max_nodes_per_batch));
        }
    }
    
    // 4. 计算 neighborMap
    for batch in &mut batches {
        batch.neighbor_map = compute_neighbor_map(batch, &batches, &import_map);
    }
    
    batches
}
```

### 2. neighborMap 计算

```rust
fn compute_neighbor_map(
    current_batch: &Batch,
    all_batches: &[Batch],
    import_map: &HashMap<String, Vec<String>>,
) -> HashMap<String, NeighborInfo> {
    let mut neighbors = HashMap::new();
    
    for file in &current_batch.files {
        if let Some(imports) = import_map.get(&file.path) {
            for imported_path in imports {
                // 检查是否在其他批次
                if let Some((other_batch_idx, symbols)) = find_in_other_batch(imported_path, current_batch.batch_index, all_batches) {
                    neighbors.insert(imported_path.clone(), NeighborInfo {
                        path: imported_path.clone(),
                        batch_index: other_batch_idx,
                        symbols,
                    });
                }
            }
        }
    }
    
    neighbors
}
```

### 4. 与 Master 集成

```rust
// Master 派发任务时使用批次信息
impl Master {
    async fn dispatch_analysis_task(&self, ticket_id: &str) -> Result<()> {
        // 1. 获取 ticket 涉及的文件
        let files = self.get_ticket_files(ticket_id)?;
        
        // 2. 计算批次
        let batches = compute_batches(files, self.import_map.clone(), 60);
        
        // 3. 并发派发（最多 5 个 Slaver）
        let semaphore = Arc::new(Semaphore::new(5));
        let handles: Vec<_> = batches.iter().map(|batch| {
            let permit = semaphore.clone().acquire_owned();
            let batch = batch.clone();
            tokio::spawn(async move {
                let _permit = permit.await;
                self.dispatch_to_slaver(&batch).await
            })
        }).collect();
        
        // 4. 等待所有批次完成
        for handle in handles {
            handle.await??;
        }
        
        // 5. 合并结果
        self.merge_batch_results(ticket_id).await
    }
}
```

### 陈架构 (Alex Chen) — 系统架构视角

**分析时间**: 2026-05-24

#### 1. 架构视角

本任务本质是**图分区问题**的工程实现，核心约束：

| 约束 | 目标 |
|------|------|
| 规模约束 | 每批 ≤60 节点，避免 LLM 上下文溢出 |
| 语义聚合 | 紧耦合文件同批，降低跨批通信成本 |
| 信息补偿 | neighborMap 弥补分割造成的信息丢失 |

**与现有模块的关系**：
- 复用 `dag.rs` 图结构定义（DagNode/DagEdge）
- batch 作为独立模块 `batch.rs`，避免功能耦合
- 依赖 TASK-E11-001 的 `StructuralAnalysis` 输出

#### 2. 算法选型建议

**Louvain vs Union-Find 连通分量对比：**

| 维度 | Louvain | Union-Find |
|------|---------|------------|
| 时间复杂度 | O(n log n) 迭代 | O(n α(n)) ≈ O(n) |
| 实现复杂度 | 高（模块度优化） | 低（20 行代码） |
| 分割质量 | 最优模块度 | 仅按连通性 |
| Rust 生态 | 需自实现 | std 足够 |

**建议：两阶段渐进策略**

```
Phase 1 (MVP): Union-Find + 路径前缀分割
  - 连通分量快速划分
  - 超大分量按 path prefix 二次切割
  - 覆盖 80% 场景，实现成本低

Phase 2 (优化): 引入 Louvain
  - 当连通分量内部耦合度差异大时启用
  - 如 monorepo 多子包形成大连通分量场景
```

**理由**：
1. EKET 场景多为中小项目（100-500 文件），连通分量足够
2. 依赖图本身是**稀疏图**，社区结构不明显
3. Louvain 价值在于「同一连通分量内再分层」，MVP 可 defer

#### 4. 核心算法伪代码

```rust
pub fn compute_batches(files: Vec<FileEntry>, max_nodes: usize) -> Vec<Batch> {
    // 1. Union-Find 按导入关系合并
    let mut uf = UnionFind::new(files.len());
    for (i, f) in files.iter().enumerate() {
        for imp in &f.imports {
            if let Some(j) = find_file_index(imp, &files) {
                uf.union(i, j);
            }
        }
    }
    
    // 2. 按连通分量分组
    let components = uf.into_components();
    
    // 3. 超大分量二次分割（按路径前缀）
    let mut batches = vec![];
    for indices in components {
        if indices.len() <= max_nodes {
            batches.push(indices);
        } else {
            batches.extend(split_by_path_prefix(&files, indices, max_nodes));
        }
    }
    
    // 4. 计算 neighborMap
    finalize_batches_with_neighbor_map(files, batches)
}
```

impl Master {
    pub async fn dispatch_analysis(&self, ticket_id: &str) -> Result<AnalysisResult> {
        let files = self.resolve_ticket_files(ticket_id).await?;
        let entries = self.analyze_structure(&files).await?;  // TASK-E11-001

#### 6. 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 连通分量过大无法有效分割 | 中 | 高 | 路径前缀分割 + 后备随机分割 |
| neighborMap 过大撑爆上下文 | 低 | 中 | 符号数量上限 (20) + 按引用频次截断 |
| 跨批次边生成质量下降 | 中 | 中 | neighborMap 提供符号类型增强推理 |
| 循环依赖导致分批不均 | 低 | 低 | 依赖图预处理：检测环并合并 |

#### 7. 实现建议

1. **新增模块**: `rust/crates/eket-core/src/batch.rs`
2. **复用结构**: 从 `dag.rs` 复用图定义，避免重复
3. **测试策略**: 
   - 小项目 (<60 文件) → 单批次
   - 中项目 (60-200 文件) → 多批次 + neighborMap 验证
   - 大项目 (>200 文件) → 压力测试分割质量
4. **度量指标**:
   - 批次间依赖比例 (越低越好)
   - neighborMap 平均大小
   - 分批耗时 (目标 <100ms for 500 files)
