# 缺陷修复任务 SOP

**版本**: 1.0.0
**最后更新**: 2026-03-23
**适用任务类型**: Bugfix

---

## 概述

缺陷修复任务用于修复已发现的 bug 和问题。

### 任务特点

| 特点 | 说明 |
|------|------|
| **复杂度** | 低 - 中 |
| **预估工时** | 1-4 小时 |
| **参与智能体** | 后端开发/前端开发 |
| **产出物** | 修复代码、回归测试 |

---

## 流程概览

```
任务认领 → 问题复现 → 根因分析 → 修复实现 → 回归测试 → PR 提交 → Review → 合并
```

---

## 详细步骤

### 1. 任务认领

**命令**:
```bash
/eket-claim BUGFIX-XXX
```

**检查项**:
- [ ] 任务状态为 `ready` 或 `backlog`
- [ ] Bug 描述清晰
- [ ] 复现步骤明确
- [ ] 优先级合理

**输出**:
- 任务状态：`in_progress`
- 计时器启动
- Checkpoint: `task_start`

---

### 2. 问题复现

**目的**: 确认 bug 存在，理解问题现象

**步骤**:
1. 阅读 bug 描述
2. 按照复现步骤操作
3. 确认 bug 现象
4. 记录环境信息

**复现记录模板**:
```markdown
# Bug 复现记录：BUGFIX-XXX

## Bug 描述

{简述 bug 现象}

## 复现步骤

1. {步骤 1}
2. {步骤 2}
3. {步骤 3}

## 预期结果

{预期的正确行为}

## 实际结果

{实际的错误行为}

## 环境信息

- OS: {操作系统}
- Browser: {浏览器版本}
- App Version: {应用版本}

## 截图/日志

{相关截图或日志}
```

---

### 3. 根因分析

**目的**: 找到 bug 的根本原因

**分析方法**:

**5 Why 分析法**:
```
问题：用户登录失败
1. 为什么登录失败？ → API 返回 500 错误
2. 为什么 API 返回 500？ → 数据库查询失败
3. 为什么数据库查询失败？ → 字段名拼写错误
4. 为什么字段名拼写错误？ → 迁移时手动修改
5. 为什么要手动修改？ → 缺少自动化迁移脚本
```

**鱼骨图分析**:
```
              人员          流程          技术
               │             │             │
        ┌──────┴──────┬──────┴──────┬──────┴──────┐
        │             │             │             │
      培训不足     规范缺失     工具不完善    代码审查不严
                        │
                        ▼
                  用户登录失败
```

**根因记录模板**:
```markdown
# 根因分析：BUGFIX-XXX

## 问题现象

{描述问题}

## 分析过程

{分析步骤和发现}

## 根本原因

{根本原因}

## 影响范围

- 影响功能：{功能列表}
- 影响用户：{用户群体}
- 影响时间：{时间范围}
```

---

### 4. 修复实现

**步骤**:
1. 创建修复方案
2. 创建分支
3. 实现修复
4. 编写回归测试

**修复方案模板**:
```markdown
# 修复方案：BUGFIX-XXX

## 修复思路

{如何修复}

## 修改内容

- 文件 1: {修改说明}
- 文件 2: {修改说明}

## 回归测试

- [ ] 测试用例 1
- [ ] 测试用例 2
```

**分支命名**:
```bash
bugfix/{slaver_name}-{ticket_id}-{short_desc}
# 示例：
bugfix/slave-004-BUGFIX-001-login-error
```

---

### 5. 回归测试

**目的**: 确保修复有效且未引入新问题

**测试类型**:
| 类型 | 说明 |
|------|------|
| **复现测试** | 验证原 bug 已修复 |
| **边界测试** | 验证边界情况 |
| **影响测试** | 验证相关功能未受影响 |

**回归测试模板**:
```markdown
# 回归测试：BUGFIX-XXX

## 复现测试

- [ ] 原 bug 已修复
- [ ] 复现步骤不再触发问题

## 边界测试

- [ ] 边界情况 1
- [ ] 边界情况 2

## 影响测试

- [ ] 相关功能 1 正常
- [ ] 相关功能 2 正常
```

---

### 6. PR 提交

**PR 描述模板**:
```markdown
## 关联任务

BUGFIX-XXX

## Bug 描述

{简述 bug 现象}

## 修复内容

- 修复了 {问题}
- 修改了 {文件}

## 根因

{根本原因}

## 回归测试

- [ ] 复现测试通过
- [ ] 边界测试通过
- [ ] 影响测试通过

## 测试步骤

1. {测试步骤 1}
2. {测试步骤 2}
```

---

### 7. Review 和合并

**Review 重点**:
- [ ] 根因分析正确
- [ ] 修复方案合理
- [ ] 回归测试完整
- [ ] 无新引入问题

**合并后操作**:
1. 验证修复效果
2. 更新任务状态为 `done`
3. 记录经验教训

## 常见问题

### Q1: 无法复现 bug 怎么办？

**A**:
1. 联系报告人获取更多信息
2. 在不同环境尝试复现
3. 检查日志和监控数据
4. 如仍无法复现，记录并上报

### Q2: 修复引入新问题怎么办？

**A**:
1. 回滚修复
2. 重新分析根因
3. 制定新修复方案
4. 重新实现和测试

### Q3: 修复需要较大改动怎么办？

**A**:
1. 评估风险和收益
2. 如改动过大，创建新的 Feature 任务
3. 当前 Bugfix 任务标记为阻塞
