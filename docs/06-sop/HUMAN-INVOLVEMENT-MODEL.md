# 人类参与模式设计

**版本**: 1.0.0
**创建日期**: 2026-03-25
**状态**: 提案

---

## 核心理念

将人类参与分为**必须参与**和**选择性参与**两类，最大化 Agent 自主性，同时在关键节点保留人类判断。

---

## 人类参与分类

### 第一类：人类必须参与 (Mandatory)

这些环节**必须**有人类参与，Agent 不能自主决定：

| 环节 | 说明 | 原因 |
|------|------|------|
| **M1: 需求提出** | 人类提出原始需求、业务目标 | 只有人类知道真正想要什么 |
| **M2: 背景知识补充** | 提供领域知识、业务规则、约束条件 | Agent 无法获取内部/隐性知识 |
| **M3: 项目方向把握** | 重大方向变更、优先级调整 | 涉及战略决策，需人类判断 |

### 第二类：人类选择性参与 (Optional)

这些环节人类**可以选择性参与**，Agent 可自主执行：

| 环节 | 说明 | Agent 自主模式 | 人类参与模式 |
|------|------|---------------|-------------|
| **O1: 文档审查** | 需求/架构/任务分析文档审查 | Agent 交叉审查 | 人类审查关键文档 |
| **O2: 任务承接** | 领取/分配任务 | Agent 自动领取 | 人类手动分配 |
| **O3: 角色扮演** | 项目中扮演特定角色 (如 PM、测试) | Agent 自主扮演 | 人类亲自扮演 |
| **O4: PR 审查** | 代码审查 | Agent 交叉审查 | 人类审查关键 PR |

## 工作流程设计

### 模式 A：全自主模式 (人类最小参与)

```
┌─────────────────────────────────────────────────────────────────┐
│ Phase 1: 项目初始化 (Agent 主导)                                │
├─────────────────────────────────────────────────────────────────┤
│ M1: 人类提出需求 → inbox/human_input.md                         │
│      ↓                                                          │
│ M2: 人类补充背景知识 (Agent 追问时)                              │
│      ↓                                                          │
│ Agent: 需求分析 → 架构设计 → 任务规划                            │
│      ↓                                                          │
│ M3: 人类确认项目方向 (可选，默认自动通过)                         │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ Phase 2: 任务执行 (Agent 主导)                                  │
├─────────────────────────────────────────────────────────────────┤
│ O2: Agent 自动领取任务 (或人类手动分配)                          │
│      ↓                                                          │
│ Agent: 任务分析 → 实现方案 → 测试策略                            │
│      ↓                                                          │
│ O1: Agent 交叉审查文档 (或人类审查)                              │
│      ↓                                                          │
│ Agent: 开发 → 测试 → 自验证                                      │
│      ↓                                                          │
│ O4: Agent 交叉审查 PR (或人类审查)                               │
│      ↓                                                          │
│ Agent: 合并到 testing → 集成测试 → 合并到 main                   │
└─────────────────────────────────────────────────────────────────┘
```

**适用场景**：
- 人类时间有限
- 任务复杂度低
- Agent 能力强、可信度高

**人类参与时间**：
- 需求提出：30-60 min
- 背景补充：10-20 min（分散在各轮追问）
- 方向确认：5-10 min（可选）
- **总计**: ~1 小时 + 可选 10 min

---

### 模式 B：人类深度参与模式

```
┌─────────────────────────────────────────────────────────────────┐
│ Phase 1: 项目初始化 (人类深度参与)                               │
├─────────────────────────────────────────────────────────────────┤
│ M1: 人类提出需求 → inbox/human_input.md                         │
│      ↓                                                          │
│ M2: 人类补充背景知识 (Agent 追问)                                │
│      ↓                                                          │
│ Agent: 需求分析文档                                              │
│      ↓                                                          │
│ O1: 人类审查需求文档 ← 审查点 A1                                 │
│      ↓ (批准后)                                                  │
│ Agent: 架构设计文档                                              │
│      ↓                                                          │
│ O1: 人类审查架构文档 ← 审查点 A2                                 │
│      ↓ (批准后)                                                  │
│ Agent: 任务规划文档                                              │
│      ↓                                                          │
│ O1: 人类审查任务规划 ← 审查点 A3                                 │
│      ↓ (批准后)                                                  │
│ M3: 人类确认项目方向 ← 审查点 A4 (综合审查)                       │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ Phase 2: 任务执行 (人类选择性参与)                               │
├─────────────────────────────────────────────────────────────────┤
│ O2: 人类选择要承接的任务 (或 Agent 自动领取)                      │
│      ↓                                                          │
│ Agent: 任务分析文档                                              │
│      ↓                                                          │
│ O1: 人类审查任务分析文档 ← 审查点 B1                             │
│      ↓ (批准后)                                                  │
│ Agent: 开发 → 测试                                               │
│      ↓                                                          │
│ O4: 人类审查 PR ← 审查点 B2                                      │
│      ↓ (批准后)                                                  │
│ Agent: 合并                                                      │
└─────────────────────────────────────────────────────────────────┘
```

**适用场景**：
- 项目复杂度高
- 人类对质量要求高
- 人类希望深度参与过程
- Agent 能力尚需验证

**人类参与时间**：
- Phase 1: 2-4 小时
- Phase 2 (每个任务): 15-30 min
- **总计**: ~4 小时 + 任务数 × 20 min

---

### 模式 C：混合模式 (推荐)

```
┌─────────────────────────────────────────────────────────────────┐
│ Phase 1: 项目初始化 (人类深度参与)                               │
├─────────────────────────────────────────────────────────────────┤
│ M1, M2, M3: 人类深度参与                                         │
│ O1: 人类审查关键文档 (需求、架构)                                 │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ Phase 2: 任务执行 (Agent 主导，人类选择性介入)                    │
├─────────────────────────────────────────────────────────────────┤
│ 常规任务: Agent 自主执行                                         │
│ 关键任务: 人类审查任务分析文档 + PR                               │
│                                                                  │
│ 判断规则:                                                        │
│ - 工作量 > 8h → 人类审查任务分析                                 │
│ - 核心功能 → 人类审查 PR                                          │
│ - 边缘功能 → Agent 自主执行                                      │
└─────────────────────────────────────────────────────────────────┘
```

**适用场景**：
- 大部分项目
- 平衡效率与质量

**人类参与时间**：
- Phase 1: 2-4 小时
- Phase 2 (关键任务): 10-20% 的任务需要人类审查
- **总计**: ~4 小时 + 关键任务数 × 20 min

## 配置方式

在 `.eket/config.yml` 中配置人类参与模式：

```yaml
# 人类参与模式配置
human_involvement:
  # 模式选择：autonomous | deep_involvement | hybrid
  mode: hybrid

  # 必须参与环节 (不可关闭)
  mandatory:
    - requirement_input       # M1: 需求提出
    - context_supplement      # M2: 背景知识补充
    - direction_setting       # M3: 项目方向把握

  # 选择性参与环节 (可配置)
  optional:
    document_review:
      enabled: true           # O1: 文档审查
      threshold: "critical"   # 审查门槛：all | critical | none
      # all=全部审查，critical=只审查关键文档，none=Agent 审查

    task_claim:
      enabled: false          # O2: 任务承接
      # true=人类分配任务，false=Agent 自动领取

    role_playing:
      enabled: false          # O3: 角色扮演
      roles: []               # 人类要扮演的角色列表

    pr_review:
      enabled: true           # O4: PR 审查
      threshold: "critical"   # 审查门槛：all | critical | none
      # 判断规则：
      # - 工作量 > 8h → 人类审查
      # - 核心功能 → 人类审查
      # - 边缘功能 → Agent 审查
```

## 人类参与点详细说明

### M1: 需求提出

**内容**:
- 业务目标描述
- 目标用户定义
- 核心功能期望
- 约束条件（时间、预算、技术）

**Agent 行为**:
- 追问澄清（通过 `/eket-ask`）
- 整理需求文档
- 确认理解正确

---

### M2: 背景知识补充

**触发时机**:
- Agent 检测到知识盲区
- 涉及内部系统/业务流程
- 需要特定领域知识

**补充方式**:
- 回答 Agent 追问
- 提供文档链接
- 邀请领域专家参与

---

### M3: 项目方向把握

**决策内容**:
- 技术方案重大变更
- 优先级调整
- 资源投入决策
- 是否进入下一阶段

**Agent 上报规则**:
- 任务复杂度 > 8 小时
- 技术选型存在争议
- 需求不明确或矛盾

---

### O1: 文档审查（选择性）

**审查对象**:
- 需求分析文档
- 架构设计文档
- 任务分析文档

**人类审查价值**:
- 确保需求理解正确
- 技术选型符合预期
- 实现方案合理

**Agent 自主模式**:
- Agent 交叉审查（需求分析师 → 技术经理 → 项目经理）
- 文档监控员最终审核

---

### O2: 任务承接（选择性）

**人类参与方式**:
- 手动分配任务给特定 Agent
- 调整任务优先级
- 决定任务并发数

**Agent 自主模式**:
- 基于角色匹配自动领取
- 基于优先级自动排序
- 基于负载自动调节

---

### O3: 角色扮演（选择性）

**人类可扮演角色**:
- 产品经理（需求优先级决策）
- 技术经理（技术方案审批）
- 测试员（验收测试）
- 用户体验师（UI/UX 审查）

**Agent 自主模式**:
- Agent 扮演所有角色
- 人类只参与 M1/M2/M3

---

### O4: PR 审查（选择性）

**人类审查价值**:
- 核心功能质量把控
- 代码规范检查
- 安全性审查

**Agent 自主模式**:
- Agent 交叉审查（开发者 → 测试员 → 技术经理）
- 自动化测试 + 静态分析

---

## 推荐配置

### 新项目启动（前 2 周）

**推荐模式**: 模式 B（人类深度参与）

**原因**:
- 建立对 Agent 能力的信任
- 确保项目方向正确
- 早期发现问题成本低

**配置**:
```yaml
human_involvement:
  mode: deep_involvement
  optional:
    document_review:
      enabled: true
      threshold: "all"
    pr_review:
      enabled: true
      threshold: "all"
```

---

### 成熟项目（2 周后）

**推荐模式**: 模式 C（混合模式）

**原因**:
- Agent 已证明能力
- 人类时间更宝贵
- 平衡效率与质量

**配置**:
```yaml
human_involvement:
  mode: hybrid
  optional:
    document_review:
      enabled: true
      threshold: "critical"
    pr_review:
      enabled: true
      threshold: "critical"
```

---

### 高信任项目（成熟团队）

**推荐模式**: 模式 A（全自主）

**原因**:
- Agent 能力经过验证
- 人类时间极其有限
- 项目复杂度低

**配置**:
```yaml
human_involvement:
  mode: autonomous
  optional:
    document_review:
      enabled: false
    pr_review:
      enabled: false
```

## 成功指标

| 指标 | 模式 A | 模式 B | 模式 C |
|------|--------|--------|--------|
| 人类时间投入 | 低 | 高 | 中 |
| 项目可控感 | 低 | 高 | 中高 |
| 返工率 | 中 | 低 | 低中 |
| Agent 成长速度 | 快 | 中 | 快 |
| 适用项目复杂度 | 低 - 中 | 高 | 中 - 高 |

## 总结

**你的分类是合理的**，但建议进一步优化：

1. **人类必须参与** - 这是**信息输入**环节，只有人类能提供：
   - 需求提出（M1）
   - 背景知识补充（M2）
   - 项目方向把握（M3）

2. **人类选择性参与** - 这是**审查/执行**环节，Agent 可自主：
   - 文档审查（O1）- 可选审查深度
   - 任务承接（O2）- 可选分配方式
   - 角色扮演（O3）- 可选参与程度
   - PR 审查（O4）- 可选审查门槛

**关键设计原则**：
- M1/M2/M3 是**必须**的，无法省略
- O1/O2/O3/O4 是**可选**的，根据项目阶段和信任度配置
- 新项目建议从模式 B 开始，逐步过渡到模式 C
