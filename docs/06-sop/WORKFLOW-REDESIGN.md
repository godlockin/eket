# 工作流程重新设计：文档优先审查模式

**版本**: 1.0.0
**创建日期**: 2026-03-25
**状态**: 提案

---

## 1. 当前工作流程分析

### 1.1 当前流程概览

```
Phase 1: 项目初始化
├── Step 1: 需求收集 (人类提供原始需求)
├── Step 2: 功能整理 (Agent 自动)
├── Step 3: 系统设计 (Agent 自动)
├── Step 4: 审查通过 (← 人类审查点 1)
└── Step 5: 任务规划 (Agent 自动)

Phase 2: 开发阶段 (每个任务)
├── Step 1: 任务认领 (Agent 自动)
├── Step 2: 任务分析 (Agent 自动)
├── Step 3: 任务拆解 (Agent 自动)
├── Step 4: 依赖上报 (Agent 自动)
├── Step 5: 任务实现 (Agent 自动)
├── Step 6: Review & UT (Agent 自动)
├── Step 7: 任务验收 (Agent 自动)
└── Step 8: 任务完成

Phase 3: Review & Merge
├── Step 1: PR Review (← 人类审查点 2)
├── Step 2: 问题修复 (Agent)
├── Step 3: 合并验证 (Agent)
└── Step 4: 合并到 Main
```

### 1.2 当前人类参与点分析

| 审查点 | 阶段 | 审查内容 | 问题 |
|--------|------|----------|------|
| **审查点 1** | Phase 1 结束 | 需求文档、架构设计、任务列表 | 审查通过后，任务执行过程中仍可能发现需求理解偏差 |
| **审查点 2** | Phase 3 开始 | 代码实现、PR 描述 | 此时发现问题需要大量返工 |

### 1.3 当前流程存在的问题

1. **需求偏差发现过晚**
   - Agent 在任务实现阶段可能发现需求理解有误
   - 此时已投入开发时间，造成浪费

2. **设计决策缺乏人类确认**
   - 技术方案由 Agent 自主决定
   - 可能与人类期望不一致

3. **代码审查负担重**
   - 人类需要在 PR 阶段审查代码质量、设计合理性、需求对齐
   - 审查维度太多，容易遗漏

4. **返工成本高**
   - 代码写完后再发现设计问题，需要重写

## 2. 文档优先审查模式（提案）

### 2.1 核心理念

> **人类审查重心前移**：在文档阶段投入更多审查精力，代码阶段只做验证性审查

```
做计划 → 更新文档 → 审查 + 批准 → 代码开发和测试 → 验证审查 → 合并
                            ↑                        ↑
                      主要审查点 (人类)        次要审查点 (验证性)
```

### 2.2 新工作流程

```
┌─────────────────────────────────────────────────────────────────┐
│ Phase 1: 项目初始化 (人类深度参与)                               │
├─────────────────────────────────────────────────────────────────┤
│ Step 1: 需求收集                                                 │
│   - 人类：提供原始需求、业务目标、约束条件                        │
│   - Agent: 整理需求、追问澄清                                    │
│                                                                  │
│ Step 2: 需求分析                                                 │
│   - Agent: 需求分析文档 (用户故事、验收标准)                      │
│   - 人类：← 审查点 A1 - 需求理解是否正确                          │
│                                                                  │
│ Step 3: 系统设计                                                 │
│   - Agent: 架构设计文档、技术方案                                │
│   - 人类：← 审查点 A2 - 技术选型、架构是否合理                    │
│                                                                  │
│ Step 4: 任务规划                                                 │
│   - Agent: 任务拆解 (Jira Tickets)                               │
│   - 人类：← 审查点 A3 - 任务优先级、依赖是否合理                  │
│                                                                  │
│ Step 5: Phase 1 审查通过                                         │
│   - 人类：← 审查点 A4 - 综合审查 (需求 + 架构 + 任务)              │
│   - Agent: 标记 Phase 1 完成，解锁 Phase 2                        │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ Phase 2: 任务执行 (Agent 主导，人类只做文档审查)                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│ 对每个任务 Ticket:                                               │
│                                                                  │
│ Step 1: 任务认领                                                 │
│   - Agent: 根据角色匹配领取任务                                   │
│                                                                  │
│ Step 2: 任务分析计划 (← 人类审查点 B1)                           │
│   - Agent: 创建《任务分析文档》包含：                            │
│          • 需求理解 (链接到 Phase 1 需求文档)                      │
│          • 实现方案 (技术细节、接口设计)                          │
│          • 测试策略 (单元测试、集成测试)                          │
│          • 工作量预估                                            │
│   - 人类：审查并批准任务分析文档                                 │
│          → 批准后 Agent 才能进入实现阶段                          │
│                                                                  │
│ Step 3: 代码开发和测试 (Agent 自动)                              │
│   - Agent: 基于已批准的任务分析文档进行开发                       │
│   - Agent: 编写测试并确保通过                                    │
│                                                                  │
│ Step 4: 验证审查 (← 人类审查点 B2)                               │
│   - Agent: 提交 PR，附带验证清单：                               │
│          • [x] 实现与批准的任务分析文档一致                        │
│          • [x] 所有测试通过                                      │
│          • [x] 代码质量检查通过                                  │
│   - 人类：只做验证性审查：                                       │
│          • 实现是否与批准的任务分析文档一致？                      │
│          • 是否有明显的代码质量问题？                             │
│          • 测试是否充分？                                        │
│                                                                  │
│ Step 5: 合并                                                     │
│   - Agent: 自动合并到 testing 分支                               │
│   - 人类：可选审查 Phase 3 的批量合并                             │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ Phase 3: 批量合并到 Main (可选人类审查)                           │
├─────────────────────────────────────────────────────────────────┤
│ Step 1: 批量 Review                                              │
│   - Agent: 生成批量合并报告 (包含所有已验证的 PR)                 │
│   - 人类：← 审查点 C1 - 批量批准 (可选)                           │
│                                                                  │
│ Step 2: 合并到 Main                                              │
│   - Agent: 合并通过审查的 PR 到 main 分支                         │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2.3 审查点总结

| 审查点 | 阶段 | 审查对象 | 审查重点 | 预计耗时 |
|--------|------|----------|----------|----------|
| **A1** | Phase 1 | 需求分析文档 | 需求理解是否正确、完整 | 30-60 min |
| **A2** | Phase 1 | 架构设计文档 | 技术选型、架构合理性 | 30-60 min |
| **A3** | Phase 1 | 任务规划 | 优先级、依赖关系 | 15-30 min |
| **A4** | Phase 1 | 综合审查 | 整体一致性、可行性 | 60-90 min |
| **B1** | Phase 2 | 任务分析文档 | 实现方案、测试策略 | 10-20 min/任务 |
| **B2** | Phase 2 | PR 验证审查 | 与批准文档的一致性 | 5-10 min/任务 |
| **C1** | Phase 3 | 批量合并报告 | 整体质量把控 (可选) | 30-60 min/批次 |

### 2.4 审查重心对比

| 维度 | 当前流程 | 文档优先流程 | 改进 |
|------|----------|--------------|------|
| **需求理解** | Phase 1 审查一次 | Phase 1 + 每个任务分析文档 | 问题更早发现 |
| **技术设计** | Agent 自主决定 | 任务分析文档中审查 | 人类把控关键决策 |
| **代码审查** | 全面审查 (设计 + 质量) | 验证性审查 (一致性) | 审查负担减少 60% |
| **返工率** | 较高 (设计问题发现晚) | 较低 (设计先批准) | 预计减少 50% |

## 6. 预期收益

| 指标 | 当前 | 预期改进 | 说明 |
|------|------|----------|------|
| 需求偏差发现时间 | 开发阶段 | 任务分析阶段 | 提前 2-3 天 |
| 代码审查时间 | 60 min/PR | 15 min/PR | 减少 75% |
| 返工率 | ~30% | ~10% | 减少 67% |

## 7. 风险缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 文档审查增加前期时间 | 项目启动延迟 | 长期看总时间减少，因返工减少 |
| Agent 不遵守批准的文档 | 实现偏差 | PR 验证阶段严格检查一致性 |
| 人类审查负担过重 | 流程瓶颈 | 限制每个审查点的时间，超时自动批准 |
