# 蓝队审查机制 v1.0

**创建者**: Master (协调实例)
**启用时间**: 2026-04-08
**目的**: 代码审查和质量把关，确保架构一致性和代码安全

---

## 蓝队组成

**蓝队成员**: 3 人 (可轮流担任)
- **蓝队队长**: 资深架构师
- **蓝队成员 A**: 后端专家
- **蓝队成员 B**: 前端/全栈专家

**职责**:
- 审查 Slaver 提交的 PR
- 确保代码符合架构规范
- 识别安全漏洞和潜在问题
- 提供改进建议

---

## 审查流程

```
Slaver 完成开发
       ↓
提交 PR 到 outbox/review_requests/
       ↓
蓝队接收审查请求
       ↓
执行三级审查 (见下方)
       ↓
通过？──是──→ Master 合并到 testing/main
       │
      否
       │
       ↓
返回 Slaver 修改
       ↓
重新提交 PR
```

---

## 三级审查清单

### Level 1: 基础审查 (必做)

**代码质量**:
- [ ] 代码可运行，无语法错误
- [ ] 通过 `npm run build`
- [ ] 通过 `npm run lint`
- [ ] 测试通过 (`npm test`)

**提交规范**:
- [ ] Git commit message 符合 Conventional Commits
- [ ] 原子提交，逻辑清晰
- [ ] 分支命名正确 (`feature/TASK-XXX-desc`)

**文档**:
- [ ] 有更新相关文档
- [ ] 有测试报告或完成报告

---

### Level 2: 架构审查 (必做)

**架构一致性**:
- [ ] 符合 EKET 架构规范
- [ ] 正确使用 SQLiteManager (非 SQLiteClient)
- [ ] 正确使用 Redis 客户端
- [ ] 遵循降级策略 (Level 3 → Level 2 → Level 1)

**代码复用**:
- [ ] 无明显重复代码 (jscpd 检查)
- [ ] 复用现有工具和函数
- [ ] 避免重复造轮子

**错误处理**:
- [ ] 使用 `Result<T>` 模式
- [ ] 正确返回 `EketErrorCode`
- [ ] 无未处理的异常

---

### Level 3: 深度审查 (条件触发)

**触发条件** (满足任一即触发):
- 修改核心模块 (core/)
- 修改 API 接口 (api/)
- 性能敏感代码
- 安全相关代码 (认证、授权、加密)

**审查内容**:
- [ ] 性能影响分析
- [ ] 安全漏洞检查 (OWASP Top 10)
- [ ] 并发/竞态条件
- [ ] 内存泄漏风险
- [ ] 边界条件处理

## 审查反馈模板

```markdown
## 蓝队审查报告

**PR**: feature/TASK-XXX
**Slaver**: [姓名/角色]
**审查人**: 蓝队 [姓名]
**审查时间**: YYYY-MM-DD

### 审查结果

- Level 1: ✅ 通过 / ❌ 问题
- Level 2: ✅ 通过 / ❌ 问题
- Level 3: ✅ 通过 / ❌ 问题 / ⏭️ 未触发

### 发现问题

#### P0: 阻塞性问题 (必须修复)
1. [问题描述]
   - 文件：`path/to/file.ts`
   - 建议：[修复建议]

#### P1: 重要问题 (建议修复)
1. [问题描述]

#### P2: 优化建议 (可选)
1. [建议描述]

### 积极反馈
- [做得好的地方]
- [值得肯定的设计]

### 总结
- [ ] 批准合并
- [ ] 需要修改后重新审查
- [ ] 需要进一步讨论

---
```

## 审查响应时间

| 优先级 | 响应时间 | 说明 |
|--------|---------|------|
| P0 (紧急) | <1 小时 | 阻塞性 Bug 修复 |
| P1 (高) | <4 小时 | 核心功能开发 |
| P2 (中) | <24 小时 | 一般功能开发 |
| P3 (低) | <48 小时 | 文档/重构 |

npm audit

## 审查通过标准

**必须满足所有条件**:
1. Level 1 全部通过 ✅
2. Level 2 全部通过 ✅
3. Level 3 (如触发) 全部通过 ✅
4. 无 P0 阻塞性问题
5. P1 问题已修复或有明确修复计划

---

## 审查不通过处理

**流程**:
1. 蓝队填写审查报告
2. 标注 P0/P1/P2 问题
3. 返回 Slaver 修改
4. Slaver 修复后重新提交
5. 蓝队重新审查 (可仅审查修改部分)

**注意**:
- 避免吹毛求疵，关注实质性问题
- 提供建设性意见，而非单纯批评
- 鼓励创新，避免过度约束

蓝队应定期 (每 Sprint) 回顾审查流程：

**回顾问题**:
1. 审查是否发现关键问题？
2. 审查流程是否高效？
3. 是否有误判或漏判？
4. 如何改进审查清单？

**改进措施**:
- 更新审查清单
- 调整审查流程
- 分享审查经验
