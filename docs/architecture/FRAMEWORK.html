初衷与愿景

初衷

在 AI 智能体技术快速发展的今天，我们发现：

  1. 单点智能体能力强大但协作困难 - 单个 Claude Code 实例可以完成复杂任务，但多个实例之间缺乏可靠的协作机制
  2. 研发流程缺乏标准化 - 每次新项目都需要重新定义流程、沟通方式和工作规范
  3. 知识无法沉淀 - 项目经验、技术决策、最佳实践散落在对话历史中，无法有效传承
  4. 规模化困难 - 随着项目复杂度增加，智能体之间的协调成本呈指数级增长

EKET 框架的诞生就是为了解决这些问题：将多智能体协作研发变成一套可复用、可演进、可审计的工程化底座。

愿景

让智能体从零散的提示词执行升级为系统化协作，持续产出稳定的设计、代码与测试结果。

我们期望通过 EKET 框架：

  • 降低协作成本 - 把”口头约定”固化为可检索的文档与结构化流程
  • 提升可控性 - 让决策与实现可追溯、可复盘、可调整
  • 放大生产力 - 系统化协作而非单点执行
  • 支持规模化 - 适配不同规模与不同技术栈的研发项目

系统目标

EKET 框架是一个多智能体协作系统，用于 AI 智能体（如 Claude Code）进行研发任务的自动化处理。系统通过：

  • 规范化的流程
  • 清晰的职责划分
  • 可靠的通信机制

实现多个智能体实例的高效协作。

核心能力

能力 说明
任务自动化 从需求输入到代码提交的端到端自动化
多智能体协作 支持多个智能体实例并行工作、协调配合
状态持久化 通过 Git 实现任务状态、代码变更的持久化存储
人类在环 关键决策点引入人类确认，确保方向正确
知识沉淀 项目经验、技术决策自动归档，形成组织记忆

架构变更 技术经理 + 人类确认 提案 → Review → 人类批准

智能体定义

每个智能体由 YAML 配置文件定义：

# agents/agent_base_template.yml
name: <agent_name>
type: coordinator|executor
version: 1.0.0
description: <简短描述>

capabilities:
  - <capability_tag_1>
  - <capability_tag_2>

inputs:
  - type: jira_ticket
    fields: [title, description, acceptance_criteria]

outputs:
  - type: jira_transition
    action: transition_ticket
  - type: pull_request
    format: github_pr

decision_policy:
  auto_decide: true|false
  auto_decide_scope:
    - <允许自主决策的场景>
  escalation_rules:
    - condition: <条件>
      escalate_to: <目标智能体>

lifecycle:
  mode: persistent|on_demand
  spawn_method: process|thread

Workflow 设计

标准任务流转

1. 人类输入需求 → 需求分析师接收
       ↓
2. 需求分析师 → 拆解为 Jira tickets
       ↓
3. 执行智能体 → 轮询并领取任务
       ↓
4. 执行智能体 → 创建 git 分支 → 开发 → 提交 PR
       ↓
5. 执行智能体 → 通过消息队列唤醒协调智能体
       ↓
6. 协调智能体 → review PR → 合并或要求修改
       ↓
7. 任务完成 → 更新 Jira 状态 → 通知相关方

分支管理

分支命名规范:
- feature/{ticket-id}-{short-desc}    # 功能开发
- bugfix/{ticket-id}-{short-desc}     # 缺陷修复
- hotfix/{ticket-id}-{short-desc}     # 紧急修复
- docs/{ticket-id}-{short-desc}       # 文档更新

合并流程:
1. 执行智能体完成开发
2. 执行智能体创建 PR，指定协调智能体为 reviewer
3. 执行智能体发送 review 请求消息
4. 协调智能体接收消息 → 执行 review
5. review 通过 → 合并到主分支
6. review 不通过 → 评论反馈 → 执行智能体修改

Agent 宪法与行为方式

第一原则：安全优先

任何操作不得损害项目、数据或系统安全。

  • 删除操作必须经过确认
  • 敏感信息不得提交到 Git
  • 发现安全风险立即上报

第二原则：人类在环

重大决策必须引入人类确认。

必须上报的场景 上报对象
任务复杂度超出预估（>4 小时） 项目经理
技术选型存在争议 技术经理
需要跨模块协调 项目经理
发现需求不明确或矛盾 需求分析师
遇到阻塞性问题超过 30 分钟 协调智能体小组

第三原则：文档即契约

所有决策、变更、经验必须文档化。

  • 需求分析完成 → 更新 Confluence 需求文档
  • 技术方案确定 → 更新 Confluence 架构文档
  • 代码提交 → 关联 Jira ticket
  • PR 合并 → 更新 Release Notes

第四原则：职责分离

各智能体在职责范围内自主决策，不得越权。

模块 负责智能体 决策权限
需求文档 需求分析师 完全自主
架构设计 技术经理 完全自主
前端代码 前端开发 自主实现，PR 审核
后端代码 后端开发 自主实现，PR 审核
测试代码 测试员 完全自主
部署配置 运维 自主实施，事后报告

第五原则：持续改进

每轮任务结束后必须总结经验，优化流程。

  • 任务完成 → 生成本次任务的经验总结
  • 发现问题 → 创建改进建议 ticket
  • 优秀实践 → 归档到长期记忆

记忆系统

三级记忆架构

类型 存储位置 内容 保留策略
短期记忆 runtime/memory/short_term/ 当前任务上下文、临时变量 任务结束清除
长期记忆 runtime/memory/long_term/ 项目知识、技术决策、经验 永久保留
外部记忆 shared/confluence/memory/ 文档化知识、最佳实践 永久保留，可检索

记忆内容示例

短期记忆 (任务执行中):
- 任务描述和验收标准
- 临时变量和中间结果
- 当前分支和文件路径

长期记忆 (永久保存):
- 项目技术栈和架构图
- 编码规范和最佳实践
- 历史决策记录和原因

外部记忆 (共享知识):
- Confluence 文档
- API 接口定义
- 数据库 Schema
