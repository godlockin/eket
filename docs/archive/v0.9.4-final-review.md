#### 4. LRU 算法优化 (cache-layer.ts)

**验证状态**: ✅ **通过 (O(1) 实现)**

**代码审查**:
```typescript
// 使用 Map 的插入顺序特性实现 O(1) LRU
get(key: string): T | undefined {
  const entry = this.cache.get(key);
  if (entry) {
    // O(1) LRU 优化：删除后重新添加到末尾，更新迭代顺序
    this.cache.delete(key);
    this.cache.set(key, entry);
    return entry.value;
  }
  // ...
}

private evictLRU(): void {
  // Map.entries() 返回的迭代器按插入顺序遍历
  // 第一个条目就是最久未使用的（LRU）
  const firstEntry = this.cache.entries().next();
  if (!firstEntry.done) {
    const [lruKey] = firstEntry.value;
    this.cache.delete(lruKey);
    this.stats.evictions++;
  }
}
```

export class OptimizedFileQueueManager {

    try {
      if (!fs.existsSync(this.config.queueDir)) {
        return;

      const files = fs.readdirSync(this.config.queueDir);
      const tempFiles = files.filter(f => f.includes('.tmp.'));

          cleanedCount++;

## 架构级别发现

### 设计优点

1. **四级降级策略设计精良**
   - Remote Redis → Local Redis → SQLite → File
   - 每级都有明确的降级条件和升级路径
   - 统计追踪 (`fallbackCount`, `lastFallbackTime`) 便于运维监控

2. **Master 选举机制健壮**
   - 分布式锁 (SETNX / SQLite 事务 / mkdir)
   - 声明等待期 (2 秒) 防止多 Master
   - 租约续期 (15 秒) 防止假死

3. **缓存层设计合理**
   - LRU 驱逐 + TTL 过期
   - 多层缓存（内存 + Redis）
   - 缓存穿透保护（互斥锁）
   - 连接池复用 + 健康检查

4. **安全实现到位**
   - API Key 哈希存储（非明文）
   - 持久化存储（SQLite）
   - 支持 Key 轮换和吊销

### 深层问题

#### 1. 四级降级可能过度工程化

**分析**:
- 代码审查显示四级降级实现完整
- 但在 80% 的中小规模部署场景中，可能只需要 SQLite 或文件级别
- 远程 Redis 和本地 Redis 两级差异不大

**建议**:
- 添加配置开关，允许项目按需启用
- 在生产环境统计实际降级次数和使用率

**实验验证 (H6)**:
```

四级降级实现：✅ 完整
升级机制：✅ 存在
统计追踪：✅ 存在
H6 假设验证：⚠️ 需在生产环境统计实际使用率
```

---

#### 2. Master 单点故障风险

**分析**:
- Master 选举机制健全，但存在 2-5 秒空白期
- 选举超时 (5 秒) + 声明等待期 (2 秒) = 最多 7 秒
- 在此期间系统处于无主状态

**建议**:
- 实现热备份 Master（Pre-warmed Standby）
- 缩短选举超时时间（从 5 秒降至 2 秒）
- 优化声明等待期逻辑（可配置）

---

#### 3. 水平扩展能力限制

**分析**:
- 当前设计为单实例架构
- Message Queue 基于文件轮询（5 秒延迟）
- SQLite 同步阻塞（单连接）

**建议**:
- 引入 Redis Streams 替代文件轮询
- SQLite 异步化（Worker 线程）
- 支持多实例状态共享
