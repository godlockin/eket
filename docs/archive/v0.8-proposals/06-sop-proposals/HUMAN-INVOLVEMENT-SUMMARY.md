# 人类参与模式设计 - 总结

**版本**: 1.0.0
**创建日期**: 2026-03-25
**关联 issue**: 人类和系统的交互分类

---

## 用户提议

用户提出人类和系统的交互可以分为两类：

### 1. 人类必须参与
- 需求提出
- 背景知识补充
- 项目方向的把握

### 2. 人类选择性参与
- 文档 review
- 承接任务
- 项目进行中各个角色的扮演

---

## 分析与响应

### 提议评估：**合理且清晰**

用户的分类抓住了核心：**信息输入**vs**审查/执行**。

在此基础上，我们进一步细化为：

| 分类 | 编号 | 环节 | 是否可省略 | 说明 |
|------|------|------|-----------|------|
| **必须参与** | M1 | 需求提出 | ❌ 否 | 只有人类知道真正想要什么 |
| | M2 | 背景知识补充 | ❌ 否 | Agent 无法获取内部/隐性知识 |
| | M3 | 项目方向把握 | ❌ 否 | 涉及战略决策，需人类判断 |
| **选择性参与** | O1 | 文档审查 | ✅ 是 | Agent 可自主审查 |
| | O2 | 任务承接 | ✅ 是 | Agent 可自动领取 |
| | O3 | 角色扮演 | ✅ 是 | Agent 可自主扮演 |
| | O4 | PR 审查 | ✅ 是 | Agent 可交叉审查 |

---

## 设计成果

### 1. 三种参与模式

| 模式 | 人类参与环节 | 适用场景 |
|------|-------------|----------|
| **A: 全自主** | M1+M2+M3 | 低复杂度、高信任 |
| **B: 深度参与** | M1+M2+M3+O1+O2+O3+O4 | 高复杂度、低信任 |
| **C: 混合 (推荐)** | M1+M2+M3+ 关键 O1/O4 | 中 - 高复杂度 |

### 2. 配置文件

```yaml
human_involvement:
  mode: "hybrid"  # autonomous | deep_involvement | hybrid

  mandatory:
    # M1/M2/M3 配置

  optional:
    document_review:
      enabled: true
      threshold: "critical"  # all | critical | none
    task_claim:
      enabled: false
    role_playing:
      enabled: false
    pr_review:
      enabled: true
      threshold: "critical"
```

## 核心设计原则

1. **M1/M2/M3 是必须的** - 这些是信息输入环节，只有人类能提供
2. **O1/O2/O3/O4 是可选的** - 这些是审查/执行环节，Agent 可自主
3. **模式可配置** - 根据项目阶段和信任度调整
4. **渐进式放权** - 新项目从深度参与开始，逐步过渡到混合/自主

## 人类时间投入估算

| 模式 | Phase 1 | Phase 2 (每任务) | 总计 (10 任务项目) |
|------|---------|-----------------|-------------------|
| 全自主 | ~1 小时 | ~2 min | ~1.5 小时 |
| 深度参与 | ~4 小时 | ~20 min | ~7 小时 |
| 混合 | ~3 小时 | ~5 min (关键任务) | ~4 小时 |

---

## 推荐采用策略

### 新项目启动（前 2 周）
**模式**: 深度参与 (B)
**原因**: 建立信任、确保方向正确

### 成熟项目（2 周后）
**模式**: 混合 (C)
**原因**: 平衡效率与质量

### 高信任项目
**模式**: 全自主 (A)
**原因**: 人类时间宝贵、Agent 已证明能力
