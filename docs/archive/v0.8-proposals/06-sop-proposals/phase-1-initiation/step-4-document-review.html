Step 4: 文档审查 SOP

版本: 1.0.0 最后更新: 2026-03-23 负责智能体: 文档监控员

目标

审查 Phase 1 产生的所有文档，确保需求清晰、架构合理、任务完整，无 P0/P1 级别问题。

输入

输入项 位置 说明
需求文档 confluence/projects/{project}/requirements/requirements.md 需求规格说明
架构文档 confluence/projects/{project}/architecture/ 系统架构设计
Jira tickets jira/epics/, jira/tickets/ 任务定义

输出

输出项 位置 说明
审查报告 confluence/memory/reviews/phase1-review.md Phase 1 审查结果
问题清单 inbox/human_feedback/phase1-issues.md 需要修复的问题
Checkpoint .eket/state/checkpoints/{project}-phase1-review.md Phase 1 审查通过证明

审查维度

维度 权重 检查内容
完整性 高 文档是否覆盖所有必要内容
一致性 高 需求、架构、任务是否一致
清晰性 中 文档是否清晰易读
可执行性 高 任务是否可以独立执行
可测试性 中 验收标准是否可测试

详细步骤

4.1 需求文档审查

审查清单:

检查项 权重 评分标准
项目愿景清晰 高 1-5 分
目标用户定义 高 1-5 分
核心功能覆盖 高 1-5 分
验收标准可测试 高 1-5 分
非功能需求定义 中 1-5 分
术语表完整 低 1-5 分

问题分级: | 级别 | 说明 | 处理要求 | |——|——|———| | P0 | 严重缺失，无法继续 | 必须修复 | | P1 | 重要缺失，影响理解 | 强烈建议修复 | | P2 | 一般问题，可后续完善 | 建议修复 | | P3 | 轻微问题，可选修复 | 可选修复 |

审查记录模板:

## 需求文档审查

### 评分
| 维度 | 得分 | 说明 |
|------|------|------|
| 完整性 | 4/5 | 缺少非功能需求 |
| 一致性 | 5/5 | 需求内部一致 |
| 清晰性 | 4/5 | 部分术语需澄清 |
| 可执行性 | 5/5 | 需求可转化为任务 |

4.2 架构文档审查

审查清单:

检查项 权重 评分标准
架构模式选择 高 1-5 分
技术选型合理 高 1-5 分
API 设计规范 高 1-5 分
数据库设计合理 高 1-5 分
架构决策记录 中 1-5 分
部署方案完整 中 1-5 分

审查要点:

架构设计: - [ ] 架构满足功能需求 - [ ] 架构满足非功能需求（性能、扩展性、安全性） - [ ] 组件职责清晰 - [ ] 组件间解耦良好 - [ ] 架构决策有记录

技术选型: - [ ] 技术栈一致性好 - [ ] 技术选型有评估 - [ ] 团队能够胜任

API 设计: - [ ] 使用 RESTful 规范 - [ ] 命名一致 - [ ] 错误处理统一 - [ ] 认证机制完善

数据库设计: - [ ] 范式化合理 - [ ] 索引设计合理 - [ ] 扩展性考虑

4.3 Jira Tickets 审查

审查清单:

检查项 权重 评分标准

验收标准可测试 高 1-5 分
依赖关系明确 高 1-5 分
优先级合理 高 1-5 分
估算准确 中 1-5 分
标签使用规范 低 1-5 分

审查要点:

任务粒度: - [ ] 任务可在 4-8 小时内完成 - [ ] 任务可独立开发和测试 - [ ] 任务职责单一

依赖关系: - [ ] 依赖关系已识别 - [ ] 依赖关系记录正确 - [ ] 无循环依赖

优先级: - [ ] 优先级评估一致 - [ ] 关键路径任务优先级高 - [ ] 阻塞任务优先

4.4 一致性审查

目的: 确保需求、架构、任务之间的一致性

审查要点:

需求→架构: - [ ] 每个功能需求在架构中有对应设计 - [ ] 非功能需求在架构中有对应方案

架构→任务: - [ ] 架构中的组件在任务中有开发计划 - [ ] API 设计在任务中有实现

需求→任务: - [ ] 每个需求有对应的任务 - [ ] 验收标准在任务中可验证
