# Phase 1 SOP - 项目初始化阶段

**版本**: 1.1.0
**最后更新**: 2026-03-25
**负责智能体**: 需求分析师、技术经理、项目经理

---

## 概述

Phase 1 是项目的初始化阶段，目标是完成从需求收集到任务规划的全流程。此阶段由**协调智能体小组**负责执行。

### 文档优先审查模式 (v1.1.0)

Phase 1 采用**分阶段审查**模式，每个关键产出都需要人类审查批准：

| 审查点 | 阶段 | 审查对象 | 审查重点 | 审查者 |
|--------|------|----------|----------|--------|
| **A1** | Step 1 后 | 需求分析文档 | 需求理解是否正确、完整 | 产品负责人 |
| **A2** | Step 2 后 | 架构设计文档 | 技术选型、架构合理性 | 技术负责人 |
| **A3** | Step 3 后 | 任务规划文档 | 优先级、依赖关系 | 项目经理 |
| **A4** | Step 4 后 | 综合审查 | 整体一致性、可行性 | 全体 + 人类 |

> **重要**: 每个审查点通过后，才能进入下一个阶段。

### Phase 1 的子阶段

Phase 1 包含 6 个子阶段，每个子阶段都经历类似的分析→设计→验证流程，只是交付粒度和详细程度不同：

| 子阶段 | 输入 | 输出 | 负责智能体 | 粒度 |
|--------|------|------|-----------|------|
| **1. 需求收集** | 人类原始需求 | 需求清单 | 需求分析师 | 粗（业务语言） |
| **2. 功能整理** | 需求清单 | 功能列表 | 需求分析师 | 中（功能模块） |
| **3. 系统设计** | 功能列表 | 系统架构图 | 技术经理 | 粗（组件级） |
| **4. 技术设计** | 系统架构 | 技术栈/API/DB | 技术经理 | 细（技术细节） |
| **5. 任务规划** | 技术设计 | Epics/Tickets | 项目经理 | 细（可执行任务） |
| **6. 审查通过** | 所有产出 | 审查报告 | 文档监控员 | N/A |

### 阶段目标

- [ ] 完成需求分析和文档化
- [ ] 完成系统架构设计
- [ ] 完成技术栈和 API 设计
- [ ] 拆解为可执行的 Jira tickets
- [ ] 设定任务优先级和依赖关系
- [ ] 通过审查（无 P0/P1 问题）

### 参与智能体

| 智能体 | 职责 | 产出 |
|--------|------|------|
| **需求分析师** | 需求分析、用户故事拆解 | 需求文档、用户故事地图 |
| **技术经理** | 架构设计、技术选型 | 架构文档、API 设计 |
| **项目经理** | 任务拆解、优先级设定 | Jira tickets、里程碑 |
| **文档监控员** | 文档质量检查 | 文档审查报告 |

### 预计时长

- 小型项目（1-2 个核心功能）：1-2 小时
- 中型项目（3-5 个核心功能）：2-4 小时
- 大型项目（5+ 核心功能）：4-8 小时

## 详细步骤

### Step 1: 需求分析

**负责智能体**: 需求分析师

**输入**: `inbox/human_input.md`

**输出**: `confluence/projects/{project}/requirements/`

**使用 Skills**:
- `user_interview`
- `requirement_decomposition`
- `acceptance_criteria_definition`

**详细流程**: 见 [step-1-requirements.md](step-1-requirements.md)

---

### Step 2: 架构设计

**负责智能体**: 技术经理

**输入**: 需求文档

**输出**: `confluence/projects/{project}/architecture/`

**使用 Skills**:
- `architecture_design`
- `api_design`
- `database_design`

**详细流程**: 见 [step-2-architecture.md](step-2-architecture.md)

---

### Step 3: 任务拆解

**负责智能体**: 项目经理

**输入**: 需求文档、架构文档

**输出**: `jira/epics/`, `jira/tickets/`

**使用 Skills**:
- `task_decomposition`
- `priority_setting`

**详细流程**: 见 [step-3-task-breakdown.md](step-3-task-breakdown.md)

---

### Step 4: 综合审查（人类 + 文档监控员）

**负责智能体**: 文档监控员 + 人类审查者

**输入**: 所有 Phase 1 产出文档

**输出**: `confluence/memory/reviews/phase1-review.md` + 人类审查批准

**审查流程**:
1. 文档监控员创建审查报告
2. 人类审查者审查并批准（审查点 A4）
3. 批准后 Phase 1 完成

**检查清单**:
- [ ] 需求文档覆盖所有用户痛点
- [ ] 架构设计与需求一致
- [ ] 任务拆解完整且可执行
- [ ] 优先级设定合理
- [ ] 依赖关系明确
- [ ] 人类审查者批准（审查点 A1-A3 已通过）

**详细流程**: 见 [step-4-document-review.md](step-4-document-review.md)

## Checkpoint 定义

在 Phase 1 执行过程中，需要在以下节点创建 Checkpoint：

| Checkpoint | 触发条件 | 负责智能体 | 检查项 |
|-----------|---------|-----------|--------|
| `phase1_requirements_complete` | 需求文档完成 | 需求分析师 | 需求完整性、验收标准 |
| `phase1_architecture_complete` | 架构文档完成 | 技术经理 | 架构一致性、技术选型 |
| `phase1_tasks_created` | 任务拆解完成 | 项目经理 | 任务完整性、优先级 |
| `phase1_review_passed` | 文档审查通过 | 文档监控员 | 审查报告、问题修复 |

## 质量检查清单

### 需求文档质量

- [ ] 目标用户定义清晰
- [ ] 用户痛点描述具体
- [ ] 核心功能列表完整
- [ ] 每个功能有验收标准
- [ ] 非功能性需求明确（性能、安全等）

### 架构文档质量

- [ ] 系统架构图清晰
- [ ] 技术选型有理有据
- [ ] API 设计规范一致
- [ ] 数据库 Schema 完整
- [ ] 安全考虑充分

### 任务拆解质量

- [ ] Epic 覆盖所有功能
- [ ] 任务粒度适中（2-8 小时/任务）
- [ ] 依赖关系明确
- [ ] 优先级合理
- [ ] 验收标准可测试

## 退出标准

Phase 1 完成的标准（Definition of Done）：

1. ✅ 需求文档已创建并审查通过
2. ✅ 架构文档已创建并审查通过
3. ✅ 至少 1 个 Epic 已创建
4. ✅ 至少 5 个 Feature tickets 已创建（状态：ready）
5. ✅ 文档审查报告已创建，无 P0/P1 问题

## 常见问题

### Q1: 需求不明确怎么办？

**A**: 使用 `user_interview` Skill 创建依赖追问任务，要求人类澄清。

### Q2: 技术选型有争议怎么办？

**A**: 创建技术调研任务（research-spike），对比不同方案的优缺点，由技术经理决策。

### Q3: 任务粒度过大怎么办？

**A**: 继续拆解，确保每个任务可以在 2-8 小时内完成。
