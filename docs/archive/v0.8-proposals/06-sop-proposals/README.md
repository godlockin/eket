## 文档优先审查模式 (v1.1.0 新增)

EKET 框架采用**文档优先审查模式**，将人类审查重心前移到文档阶段，代码阶段只做验证性审查。

### 核心理念

> **人类审查重心前移**：在文档阶段投入更多审查精力，代码阶段只做验证性审查

```
做计划 → 更新文档 → 审查 + 批准 → 代码开发和测试 → 验证审查 → 合并
                            ↑                        ↑
                      主要审查点 (人类)        次要审查点 (验证性)
```

### 审查点分布

| 阶段 | 审查点 | 审查对象 | 审查重点 | 预计耗时 |
|------|--------|----------|----------|----------|
| Phase 1 | A1-A4 | 需求/架构/任务规划 | 需求理解、技术选型、任务优先级 | 2-4 小时 |
| Phase 2 | B1 | 任务分析文档 | 实现方案、测试策略 | 10-20 min/任务 |
| Phase 2 | B2 | PR 验证审查 | 与批准文档的一致性 | 5-10 min/任务 |
| Phase 3 | C1 | 批量合并报告 | 整体质量把控 (可选) | 30-60 min/批次 |

### 主要变更 (v1.1.0)

1. **Phase 1 增加人类审查点** - 需求分析、架构设计、任务规划后需人类审查批准
2. **Phase 2 新增任务分析文档** - 执行前创建《任务分析文档》，人类批准后方可实施
3. **Phase 3 简化为验证审查** - PR 审查聚焦于"是否与批准的文档一致"

详见 [WORKFLOW-REDESIGN.md](WORKFLOW-REDESIGN.md)

## 两层流程模型

EKET 框架采用**两层流程模型**：Phase 级流程和 Task 级流程。两者遵循相似的流程模式，但交付的粒度和详细程度不同。

```
┌─────────────────────────────────────────────────────────────────┐
│                        Phase 级流程                              │
│  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐   │
│  │需求  │→ │功能  │  │系统  │→ │技术  │  │任务  │→ │审查  │   │
│  │收集  │  │整理  │  │设计  │  │设计  │  │规划  │  │通过  │   │
│  └──────┘  └──────┘  └──────┘  └──────┘  └──────┘  └──────┘   │
│     ↓          ↓          ↓          ↓          ↓         ↓     │
│  业务语言   功能模块    组件级     技术细节    可执行     N/A    │
│  (粗粒度)   (中粒度)    (粗粒度)   (细粒度)    (细粒度)         │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                        Task 级流程                              │
│  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐   │
│  │任务  │→ │任务  │  │拆解  │  │任务  │  │Review│→ │任务  │   │
│  │认领  │  │分析  │  │(可选)│  │实现  │  │&UT   │  │完成  │   │
│  └──────┘  └──────┘  └──────┘  └──────┘  └──────┘  └──────┘   │
│     ↓          ↓          ↓          ↓          ↓         ↓     │
│  标记状态   理解需求    子任务     代码实现    质量保障   状态更新│
│  (粗粒度)   (中粒度)    (细粒度)   (细粒度)    (细粒度)  (粗粒度)│
└─────────────────────────────────────────────────────────────────┘
```

## Phase 级流程

### Phase 定义

| Phase | 名称 | 目标 | 产出 |
|-------|------|------|------|
| **Phase 1** | 项目初始化 | 从需求到任务规划 | 需求文档、架构设计、Jira tickets |
| **Phase 2** | 开发阶段 | 任务实现和测试 | 功能代码、测试代码、PR |
| **Phase 3** | Review & Merge | 代码审查和合并 | 合并后的代码、测试报告 |
| **Phase 4** | 运维监控 | 线上监控和迭代 | 监控数据、优化建议 |

### Phase 1 子阶段

| 子阶段 | 输入 | 输出 | 详细程度 |
|--------|------|------|---------|
| **1.1 需求收集** | 人类原始需求 | 需求清单 | 业务语言，粗粒度 |
| **1.2 功能整理** | 需求清单 | 功能列表 | 功能模块，中粒度 |
| **1.3 系统设计** | 功能列表 | 系统架构图 | 组件级，粗粒度 |
| **1.4 技术设计** | 系统架构 | API/DB/技术栈 | 技术细节，细粒度 |
| **1.5 任务规划** | 技术设计 | Epics/Tickets | 可执行任务，细粒度 |
| **1.6 审查通过** | 所有产出 | 审查报告 | N/A |

### Phase 2 子阶段

| 子阶段 | 输入 | 输出 | 详细程度 |
|--------|------|------|---------|
| **2.1 任务认领** | 任务列表 | 已领取任务 | 状态标记，粗粒度 |
| **2.2 任务分析** | 任务文档 | 理解记录 | 中粒度 |
| **2.3 任务拆解** | 复杂任务 | 子任务 (可选) | 细粒度 |
| **2.4 依赖上报** | 依赖分析 | 依赖澄清 (可选) | 细粒度 |
| **2.5 任务实现** | 任务设计 | 功能代码 | 细粒度 |
| **2.6 Review & UT** | 完成代码 | 测试报告 | 细粒度 |
| **2.7 任务验收** | 验收结果 | 验收通过 | 中粒度 |
| **2.8 任务完成** | 所有产出 | 状态更新 | 粗粒度 |

### Phase 3 子阶段

| 子阶段 | 输入 | 输出 | 详细程度 |
|--------|------|------|---------|
| **3.1 PR Review** | PR 请求 | Review 意见 | 细粒度 |
| **3.2 问题修复** | Review 意见 | 修复代码 | 细粒度 |
| **3.3 合并验证** | 通过代码 | 验证报告 | 中粒度 |
| **3.4 合并到 Main** | 验证通过 | 合并完成 | 粗粒度 |

### Phase 4 子阶段

| 子阶段 | 输入 | 输出 | 详细程度 |
|--------|------|------|---------|
| **4.1 监控部署** | 部署完成 | 监控配置 | 细粒度 |
| **4.2 数据收集** | 监控配置 | 运行数据 | 中粒度 |
| **4.3 问题分析** | 运行数据 | 问题报告 | 细粒度 |
| **4.4 优化迭代** | 问题报告 | 优化任务 | 中粒度 |

```
backlog → analysis → approved → design → ready → dev → test → review → done
```

```

│                        Task 级流程                               │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  1. 任务认领 (Claim)                                            │
│  - 查看任务列表                                                  │
│  - 检查依赖关系                                                  │
│  - 标记状态：in_progress                                        │
│  - 启动计时器                                                    │

                              │
                              ▼
```

## 两层流程的对应关系

### 粒度对比

| 维度 | Phase 级 | Task 级 |
|------|---------|---------|
| **时间跨度** | 天/周 | 小时/天 |
| **参与智能体** | 多个 | 单个为主 |
| **产出物** | 文档集/代码集 | 单个功能 |
| **审查级别** | 正式审查 | 同行 Review |
| **人类参与** | 关键决策点 | 依赖澄清 |

### 流程映射

| Phase 1 阶段 | 对应 Task 级流程 |
|-------------|-----------------|
| 需求收集 | 任务分析（理解需求） |
| 功能整理 | 任务拆解（分解功能） |
| 系统设计 | 依赖上报（识别约束） |
| 技术设计 | 任务实现（技术细节） |
| 任务规划 | Review & UT（质量保障） |
| 审查通过 | 任务验收/完成（验收） |

## 核心原则

### 1. 流程同构

Phase 级和 Task 级流程遵循相同的模式：
```
输入 → 分析 → 设计 → 实现 → 验证 → 输出
```

### 2. 粒度递进

每个阶段都比前一阶段更详细：
```
业务需求 → 功能规格 → 系统设计 → 技术实现 → 可执行任务
```

### 3. 质量门禁

每个阶段都有审查/验证环节：
- Phase 级：正式审查（文档监控员）
- Task 级：Review & UT（Master/测试员）

### 4. 持续反馈

每个阶段都可能产生反馈循环：
- 审查不通过 → 返回修改
- Review 不通过 → 返回修复
- 依赖缺失 → 创建澄清任务
