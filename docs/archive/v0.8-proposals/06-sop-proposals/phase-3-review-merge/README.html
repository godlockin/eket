Phase 3 SOP - Review & Merge 阶段

版本: 1.1.0 最后更新: 2026-03-25 负责智能体: 技术经理（Master）、执行智能体（Slaver）

概述

Phase 3 是项目的审查和合并阶段，目标是对完成的代码进行 Review、验证、合并，并创建后续任务。

文档优先审查模式 (v1.1.0)

Phase 3 的审查重点从”设计审查”转变为验证性审查：

审查维度 传统模式 文档优先模式
审查焦点 代码质量 + 设计合理性 与批准文档的一致性
审查问题 “这个设计合理吗？” “实现与批准的文档一致吗？”
人类判断 需要判断设计是否正确 只需对比文档确认一致性
审查时间 60+ min/PR 15-20 min/PR

验证性审查清单

核心原则: 设计已在任务分析文档阶段批准，此阶段只做验证性审查。

阶段目标

参与智能体

智能体 职责 产出
技术经理 (Master) 代码审查、合并决策 Review 意见、合并确认
执行智能体 (Slaver) 修复 Review 问题 修复代码、重新提交

预计时长

  • Review: 1-2 小时
  • 修复（如需要）: 2-4 小时
  • 合并：0.5 小时

详细步骤

Step 1: PR Review（验证性审查）

负责智能体: 技术经理

输入: PR 请求（outbox/review_requests/）、批准的任务分析文档

输出: Review 意见

审查重点: 验证实现与批准的任务分析文档一致

检查清单:

文档一致性 (核心审查点): - [ ] 实现与批准的任务分析文档一致 - [ ] 所有变更都在任务范围内 - [ ] 没有未授权的范围蔓延 - [ ] 接口设计与批准文档一致 - [ ] 数据结构与批准文档一致

代码质量: - [ ] 代码符合编码规范 - [ ] 无 Lint 错误 - [ ] 代码可读性好 - [ ] 无重复代码

测试覆盖: - [ ] 单元测试已编写 - [ ] 测试通过 - [ ] 覆盖率达标（>80%）

安全性: - [ ] 无安全漏洞 - [ ] 敏感信息处理正确 - [ ] 输入验证完善

审查记录模板:

# PR Review: {TICKET_ID}

**Reviewer**: 技术经理
**时间**: {timestamp}
**审查类型**: 验证性审查

## 文档一致性验证

- [ ] 实现与批准的任务分析文档一致
- [ ] 所有变更都在任务范围内
- [ ] 没有未授权的范围蔓延

**批准文档链接**: [任务分析文档]()

## Review 结果

- [ ] 代码质量
- [ ] 功能实现
- [ ] 测试覆盖
- [ ] 安全性

## 问题列表

| 严重性 | 问题描述 | 建议修复 |
|--------|---------|---------|
| Major | ... | ... |
| Minor | ... | ... |

## 结论

- [ ] 通过（无需修改）
- [ ] 有条件通过（Minor 问题）
- [ ] 需要修改（Major 问题）
- [ ] 拒绝（严重问题）

注意: 如发现设计与批准文档不一致，应要求执行智能体说明原因或修改。重大设计变更需重新提交任务分析文档审查。

Step 2: 修复 Review 问题

负责智能体: 执行智能体

输入: Review 意见

输出: 修复后的代码

流程: 1. 阅读 Review 意见 2. 逐个修复问题 3. 提交修复代码 4. 重新请求 Review

## 额外修改

{说明额外修改的内容}

Step 3: 重新提交 Review

负责智能体: 执行智能体

输入: 修复后的代码

输出: 重新提交的 PR

流程: 1. 推送修复代码 2. 更新 PR 描述 3. 重新请求 Review 4. 等待再次审查

Step 4: 合并验证

负责智能体: 技术经理

输入: Review 通过的代码

输出: 验证报告

验证脚本:

./scripts/merge-validator.sh validate {TICKET_ID}

验证项: - [ ] 所有功能点完成 - [ ] 单元测试通过 - [ ] Master Review 通过 - [ ] 无 P0/P1 问题

验证报告模板:

# 合并验证报告：{TICKET_ID}

**验证者**: 技术经理
**时间**: {timestamp}

## 验证结果

| 验证项 | 状态 | 说明 |
|--------|------|------|
| 功能点完成 | ✓/✗ | ... |
| UT 通过 | ✓/✗ | ... |
| Review 通过 | ✓/✗ | ... |

## 未完成功能（如有）

| 功能点 | 后续任务 ID | 状态 |
|--------|------------|------|
| ... | FIX-{TICKET_ID}-post | backlog |

## 结论

- [ ] 允许合并
- [ ] 创建后续任务后合并

Step 5: 合并到 Main

负责智能体: 技术经理

输入: 验证通过的代码

输出: 合并完成的代码

流程:

# 1. 合并到 testing 分支
git checkout testing
git merge --no-ff feature/{slaver}-{ticket_id}

# 2. 运行集成测试
./tests/run-integration-tests.sh

# 3. 合并到 main 分支
git checkout main
git merge --no-ff testing

# 4. 推送远程
git push origin main testing

# 5. 清理分支
git branch -d feature/{slaver}-{ticket_id}

Step 6: 创建后续任务

负责智能体: 技术经理

输入: 验证报告、未完成功能列表

输出: 后续 Jira tickets

任务类型: - 测试任务: TEST-{TICKET_ID} - 线上测试验证 - 修复任务: FIX-{TICKET_ID}-post - 修复潜在问题
