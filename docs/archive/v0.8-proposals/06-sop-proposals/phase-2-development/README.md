# Phase 2 SOP - 开发阶段

**版本**: 1.1.0
**最后更新**: 2026-03-25
**负责智能体**: 执行智能体（前端开发、后端开发、设计师、测试员）

---

## 概述

Phase 2 是项目的开发阶段，目标是领取任务、自主开发、测试、提交 PR。此阶段由**执行智能体**负责执行。

### 文档优先审查模式 (v1.1.0)

Phase 2 采用**任务分析文档审查**模式，执行前需人类批准分析文档：

```
领取任务 → 创建任务分析文档 → 人类审查批准 → 开发测试 → PR 验证审查 → 合并
                              ↑                    ↑
                         审查点 B1 (人类)      审查点 B2 (验证性)
```

| 审查点 | 阶段 | 审查对象 | 审查重点 | 预计耗时 |
|--------|------|----------|----------|----------|
| **B1** | Step 2 后 | 任务分析文档 | 实现方案、测试策略 | 10-20 min |
| **B2** | Step 5 后 | PR 验证审查 | 与批准文档的一致性 | 5-10 min |

> **重要**: 任务分析文档批准后，才能进入开发阶段。

### 阶段目标

- [ ] 领取任务并理解需求
- [ ] 创建分支和 worktree
- [ ] 自主开发和测试
- [ ] 提交 PR 请求 Review
- [ ] 根据反馈迭代修改

### 参与智能体

| 智能体 | 职责 | 产出 |
|--------|------|------|
| **前端开发** | 前端功能开发 | 前端代码、单元测试 |
| **后端开发** | 后端功能开发 | 后端代码、API 测试 |
| **设计师** | UI/UX设计 | 设计稿、切图 |
| **测试员** | 测试用例编写 | 测试代码、测试报告 |

### 预计时长

- 小型任务（2-4 小时）：半天内完成
- 中型任务（4-8 小时）：1 天内完成
- 大型任务（8-16 小时）：2-3 天内完成

## 详细步骤

### Step 1: 领取任务

**负责智能体**: 执行智能体

**输入**: `jira/tickets/` 中的 ready 状态任务

**输出**: 任务状态更新为 `in_progress`

**使用脚本**:
- `./scripts/load-agent-profile.sh` - 加载 Agent Profile
- `./scripts/task-time-tracker.sh start` - 启动计时器
- `./scripts/checkpoint-sprint-retro.sh checkpoint task_start` - 创建 Checkpoint

**详细流程**: 见 [step-1-task-claim.md](step-1-task-claim.md)

---

### Step 2: 创建任务分析文档

**负责智能体**: 执行智能体

**输入**: 任务文档、Phase 1 文档（需求、架构）

**输出**: `confluence/tasks/{ticket-id}/task-analysis.md` + 人类审查批准（审查点 B1）

**使用模板**: `template/confluence/templates/task-analysis-template.md`

**文档内容**:
- 需求理解（链接到 Phase 1 需求文档）
- 实现方案（技术方案、接口设计、数据结构）
- 测试策略（单元测试、集成测试）
- 工作量预估
- 风险和问题

**审查流程**:
1. 执行智能体创建任务分析文档
2. 提交人类审查（审查点 B1）
3. 人类审查者批准/修改/拒绝
4. 批准后方可进入开发阶段

**详细流程**: 见 [step-2-task-analysis.md](step-2-task-analysis.md)

---

### Step 3: 创建分支和 Worktree

**负责智能体**: 执行智能体

**输入**: 已批准的任务分析文档

**输出**: Feature 分支和 Worktree

**使用脚本**:
- `git worktree add` - 创建 worktree
- `git checkout -b` - 创建分支

**分支命名规范**:
```bash
feature/{slaver_name}-{ticket_id}-{short_desc}
# 示例：
feature/slave-004-FEAT-001-user-auth
feature/slave-006-FEAT-002-monitoring
```

**详细流程**: 见 [step-3-branch-worktree.md](step-3-branch-worktree.md)

---

### Step 4: 自主开发

**负责智能体**: 执行智能体

**输入**: 已批准的任务分析文档

**输出**: 功能代码、单元测试

**使用 Skills**:
- `frontend_development` - 前端开发

- `unit_test` - 单元测试编写

**详细流程**: 见 [step-4-development.md](step-4-development.md)

### Step 5: 测试验证

**负责智能体**: 执行智能体

**输入**: 开发完成的代码

**使用 Skills**:
- `unit_test` - 单元测试
- `e2e_test` - E2E 测试

**详细流程**: 见 [step-5-testing.md](step-5-testing.md)

### Step 6: 提交 PR

**负责智能体**: 执行智能体

**输出**: PR 请求

- [ ] 更新了相关 API 文档

- [ ] 代码注释充分

## Checkpoint 定义

在 Phase 2 执行过程中，需要在以下节点创建 Checkpoint：

| Checkpoint | 触发条件 | 负责智能体 | 检查项 |
|-----------|---------|-----------|--------|
| `task_start` | 任务领取后 | 执行智能体 | 依赖检查、分支创建、计时器启动 |
| `task_dev_complete` | 开发完成后 | 执行智能体 | UT 通过、Lint 通过、代码覆盖率 |
| `task_test_complete` | 测试完成后 | 执行智能体 | E2E 通过、PR Draft 创建 |
| `task_complete` | 任务完成后 | 执行智能体 | 功能验证、文档更新 |

## 质量检查清单

### 代码质量

- [ ] 代码符合编码规范
- [ ] 无 Lint 错误
- [ ] 代码可测试
- [ ] 无硬编码

### 测试质量

- [ ] 单元测试覆盖核心逻辑
- [ ] 测试通过
- [ ] 代码覆盖率达标（>80%）

### 文档质量

- [ ] 代码注释清晰
- [ ] PR 描述完整
- [ ] 变更已记录

## 退出标准

Phase 2 完成的标准（Definition of Done）：

1. ✅ 功能代码已完成
2. ✅ 单元测试通过
3. ✅ 代码覆盖率达标
4. ✅ PR 已提交到 testing 分支
5. ✅ Review 反馈已处理
