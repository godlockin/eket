1.3 v0.5 引入的新风险

最严重的新问题: 1. 策略集成缺口 - 多个新脚本未集成到主流程 2. 配置复杂度激增 - 11 项策略引入 50+ 新配置参数 3. 监控代理自举问题 - Memory Review Agent 自身无健康检查 4. 权限配置维护负担 - allow/question/reject 列表需手动维护 5. Sprint 机制与任务超时冲突 - 7 天 Sprint vs 3 小时任务超时

二、Critical 风险 (🔴)

C-01: 策略集成缺口

风险描述: v0.5 引入的 11 项策略中，有 3 项未集成到主流程，导致”配置了但无实际作用”。

影响: - Master 监控循环未自动启动 - Memory Review Agent 未自动启动 - Merge Validator 未集成到 merge 流程

复现路径:

1. 运行 /eket-start 启动实例
2. 检查 Master 监控进程 → 未运行
3. 检查 Memory Review Agent → 未运行
4. 执行 merge 操作 → 无验证步骤

当前代码问题:

# eket-start.sh - 无 Master 监控启动
# scripts/memory-review-agent.sh - 独立脚本，无自动启动
# merge-validator.sh - 需要手动调用

修复建议:

# eket-start.sh 添加自动启动
start_master_monitoring() {
    if [ "$AUTO_MODE" = true ]; then
        nohup ./scripts/master-monitor.sh > logs/master-monitor.log 2>&1 &
        echo "✓ Master 监控已启动"
    fi
}

start_memory_review_agent() {
    nohup ./scripts/memory-review-agent.sh > logs/memory-review.log 2>&1 &
    echo "✓ Memory Review Agent 已启动"
}

# merge workflow 集成验证
before_merge() {
    ./scripts/merge-validator.sh validate "$TICKET_ID" || {
        echo "❌ 合并验证失败"
        return 1
    }
}

C-03: Memory Review Agent 自举问题

风险描述: Memory Review Agent 负责监控 memory 健康，但自身无健康检查机制。

影响: - Agent 挂掉后无人知晓 - Memory 问题无法检测 - 配置漂移无法发现

当前代码问题:

# memory-review-agent.sh
while true; do
    monitor_memory
    sleep 3600
done
# ❌ 无心跳
# ❌ 无健康检查
# ❌ 无故障转移

修复建议:

# 添加健康检查
HEALTH_FILE=".eket/state/memory-agent-health"

report_health() {
    cat > "$HEALTH_FILE" << EOF
status: healthy
last_check: $(date -Iseconds)
last_action: $LAST_ACTION
memory_checked: $MEMORY_COUNT
EOF
}

# Master 监控检查 Agent 健康
check_agent_health() {
    local last_check=$(grep "last_check:" "$HEALTH_FILE" | cut -d' ' -f2)
    local elapsed=$(($(date +%s) - $(date -d "$last_check" +%s)))
    if [ "$elapsed" -gt 3600 ]; then
        alert "Memory Review Agent 可能已挂掉"
    fi
}

风险描述: allow/question/reject 列表需手动维护，容易过时或配置错误。

影响: - 配置过慢导致 Slaver 频繁询问 - 配置过快导致安全风险 - 新操作需手动更新配置

当前配置:

  jira:
    allow:
      - "read_ticket"
      - "update_status"
      - "add_comment"
    question:
      - "change_priority"
      - "assign_agent"
    reject:
      - "delete_ticket"
      - "change_type"
  # ... 类似配置 30+ 项

修复建议:

# 基于风险的动态权限
permissions:
  model: "risk_based"
  auto_allow_patterns:
    - "read_*"
    - "list_*"
    - "get_*"
  auto_question_patterns:
    - "update_*"
    - "change_*"
    - "create_*"
  auto_reject_patterns:
    - "delete_*"
    - "drop_*"
    - "purge_*"

C-05: Sprint 机制与任务超时冲突

风险描述: Sprint 周期 7 天，但任务超时 3 小时，存在时间尺度冲突。

影响: - 长周期任务被强制重置 - Sprint 目标无法完成 - Slaver 士气受挫

复现场景:

1. Slaver 领取复杂任务 (预估 8 小时)
2. 配置 timeout_minutes: 180 (3 小时)
3. Slaver 遇到问题，人类 4 小时未响应
4. 任务被强制重置为 ready
5. Sprint 目标缺少关键功能

修复建议:

tasks:
  time:
    default_estimated_minutes: 120
    max_estimated_minutes: 480  # 8 小时上限
    timeout_multiplier: 1.5     # 超时 = 预估 * 1.5
    human_timeout_hours: 24     # 人类响应 24 小时

C-06: 心跳机制实现不完整

风险描述: 心跳机制已定义但无实际监控和告警实现。

影响: - Slaver 停滞无法检测 - 超时后无自动处理 - 配置了但无作用

当前代码问题:

# task-time-tracker.sh - 仅有更新心跳函数
update_heartbeat() {
    echo "last_heartbeat: $(date -Iseconds)" >> "$STATE_FILE"
}

# ❌ 无监控函数
# ❌ 无告警函数
# ❌ 无自动恢复函数

修复建议:

# heartbeat-monitor.sh (新增)
check_heartbeats() {
    for state_file in .eket/state/slaver-*.yml; do
        local last=$(grep "last_heartbeat:" "$state_file" | cut -d' ' -f2)
        local elapsed=$(($(date +%s) - $(date -d "$last" +%s)))
        local timeout=$(grep "timeout_seconds:" "$state_file" | cut -d':' -f2 | tr -d ' ')

        if [ "$elapsed" -gt "$timeout" ]; then
            echo "⚠️ Slaver $(basename $state_file) 心跳超时 ($elapsed > $timeout)"
            alert_slaver_timeout "$state_file"
            reset_task_if_needed "$state_file"
        fi
    done
}

# 每分钟检查
while true; do
    check_heartbeats
    sleep 60
done

C-07: Mock 检测覆盖不全

风险描述: mock-detector.sh 仅检测固定模式，无法覆盖所有 Mock 场景。

影响: - 高级 Mock 实现可绕过检测 - 部分假代码仍能通过检查

当前检测模式:

# mock-detector.sh
detect_mock_patterns() {
    grep -r "return \[\]" src/
    grep -r "return {}" src/
    grep -r "// TODO" src/
    grep -r "NotImplementedError" src/
    # ❌ 无法检测以下模式:
    # - return mock_data (外部文件)
    # - return fake_response() (函数调用)
    # - async def search() -> List: ... (空异步函数)
}

修复建议:

detect_mock_implementations() {
    # 检测空函数体
    for file in $(find src -name "*.ts" -o -name "*.py"); do
        if is_function_empty "$file"; then
            echo "⚠️ 发现空函数体：$file"
            ALERT_FILES+=("$file")
        fi
    done

    # 检测硬编码返回
    grep -rE "return ['\"]mock|fake|dummy" src/
    grep -rE "return \{.*mock.*\}" src/

    # 检测外部 Mock 文件
    find src -name "*mock*" -o -name "*fake*" -o -name "*stub*"
}

H-07: 日志轮转配置无实际作用

风险描述: logging.max_files: 10 配置但无实际轮转实现。

影响: - 日志无限增长 - 磁盘空间耗尽 - 旧日志难以查找

修复建议:

# scripts/log-rotate.sh (新增)
rotate_logs() {
    for log_dir in logs/*/; do
        # 保留最新 10 个日志
        ls -t "$log_dir"*.log | tail -n +11 | xargs -r rm
        # 压缩 3 天前的日志
        find "$log_dir" -name "*.log" -mtime +3 -exec gzip {} \;
        # 删除 30 天前的压缩日志
        find "$log_dir" -name "*.log.gz" -mtime +30 -delete
    done
}

# 每天执行
0 2 * * * ./scripts/log-rotate.sh

H-08: 并发控制无实现

风险描述: max_concurrent_slavers: 3 配置但无实际限制。

影响: - 资源可能过载 - 多 Slaver 冲突 - 性能下降

修复建议:

# scripts/concurrency-controller.sh
check_concurrency_limit() {
    local active=$(count_active_slavers)
    local max=$(grep "max_concurrent_slavers:" .eket/config.yml | cut -d':' -f2 | tr -d ' ')

    if [ "$active" -ge "$max" ]; then
        echo "❌ 已达最大并发 Slaver 数 ($active/$max)"
        return 1
    fi
    return 0
}

before_claim_task() {
    check_concurrency_limit || {
        echo "等待其他 Slaver 完成任务..."
        wait_for_slot
    }
}

8.2 核心问题
