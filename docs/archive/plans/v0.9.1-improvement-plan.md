# EKET v0.9.1 改进方案

**版本**: v0.9.1
**完成日期**: 2026-03-26
**状态**: 实施中

---

## 概述

v0.9.1 针对两个关键问题进行改进：

1. **增强降级策略**：从两级降级（Redis → File）扩展到四级降级（远程 Redis → 本地 Redis → SQLite → File）
2. **Master 选举问题**：解决多个 instance 同时启动时都認為自己是 Master 的竞态条件

---

## 问题 1: 降级策略增强

### 当前状态

当前的 `message-queue.ts` 只支持两级降级：
```
Redis → File
```

### 目标状态

四级降级策略，支持两种驱动模式：

```
┌─────────────────────────────────────────────────────────┐
│                    驱动模式检测                          │
│  ┌─────────────┐     ┌─────────────┐                    │
│  │  JS 驱动    │     │  Shell 驱动 │                    │
│  │  (Node.js)  │     │  (Bash)     │                    │
│  └─────────────┘     └─────────────┘                    │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│                    四级降级链路                          │
│  ┌───────────────────────────────────────────────────┐  │
│  │ Level 1: 远程共享 Redis                            │  │
│  │ - 用于多实例共享状态                               │  │
│  │ - 配置：EKET_REMOTE_REDIS_HOST/PORT               │  │
│  └───────────────────────────────────────────────────┘  │
│                          │ (失败)                        │
│                          ▼                               │
│  ┌───────────────────────────────────────────────────┐  │
│  │ Level 2: 本地 Redis                                │  │
│  │ - 用于单机多实例                                   │  │
│  │ - 配置：EKET_LOCAL_REDIS_HOST/PORT                │  │
│  └───────────────────────────────────────────────────┘  │
│                          │ (失败)                        │
│                          ▼                               │
│  ┌───────────────────────────────────────────────────┐  │
│  │ Level 3: 本地 SQLite                               │  │
│  │ - 用于数据持久化                                   │  │
│  │ - 路径：~/.eket/data/sqlite/eket.db               │  │
│  └───────────────────────────────────────────────────┘  │
│                          │ (失败)                        │
│                          ▼                               │
│  ┌───────────────────────────────────────────────────┐  │
│  │ Level 4: 本地文件系统                              │  │
│  │ - 最终降级模式                                     │  │
│  │ - 路径：.eket/data/queue/                         │  │
│  └───────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

## 问题 2: Master 选举机制

### 问题分析

当前代码在 `start-instance.ts` 中的逻辑：

```typescript
// 旧逻辑 - 存在竞态条件
const masterCheckResult = await checkMasterMarker(projectRoot);
const masterExists = masterCheckResult.data.exists;

if (!masterExists) {
  // 多个 instance 同时检测到这里，都会成为 Master
  return { success: true, data: 'master' };
}
```

**问题场景**：
1. Instance A 启动，检查 Master marker，不存在
2. Instance B 同时启动，检查 Master marker，不存在
3. Instance A 创建 Master marker
4. Instance B 也创建 Master marker（可能失败，但已认为自己是 Master）
5. 结果：两个 instance 都认为自己是 Master

### 解决方案

**`node/src/core/master-election.ts`**

使用分布式锁机制，支持三级降级：

```
┌─────────────────────────────────────────────────────────┐
│                  Master 选举机制                         │
│                                                         │
│  Level 1: Redis SETNX 分布式锁                          │
│  - 原子性：SET key value NX PX timeout                  │
│  - 租约机制：定期续期                                    │
│  - 声明等待期：2 秒检测冲突                              │
│                         │ (失败)                         │
│                         ▼                                │
│  Level 2: SQLite 锁表                                    │
│  - 事务保证原子性                                        │
│  - CREATE TABLE IF NOT EXISTS master_lock               │
│  - INSERT OR REPLACE 实现锁                              │
│                         │ (失败)                         │
│                         ▼                                │
│  Level 3: 文件系统原子操作                               │
│  - mkdir exclusive 创建锁目录                            │
│  - 文件锁信息：JSON 格式                                 │
│  - 声明文件检测冲突                                      │
└─────────────────────────────────────────────────────────┘
```

### 选举流程

```
┌───────────────────────────────────────────────────────────┐
│ Instance 启动                                              │
└───────────────────────────────────────────────────────────┘
                            │
                            ▼
┌───────────────────────────────────────────────────────────┐
│ Step 1: 尝试获取分布式锁                                   │
│ - Redis: SET lock_key instance_id NX PX 5000             │
│ - SQLite: INSERT OR REPLACE INTO master_lock             │
│ - File: mkdir lock_dir (exclusive)                       │
└───────────────────────────────────────────────────────────┘
                            │
              ┌─────────────┴─────────────┐
              │                           │
              ▼                           ▼
      ┌───────────────┐           ┌───────────────┐
      │ 获取成功       │           │ 获取失败       │
      │               │           │               │
      │ 进入声明等待期 │           │ 放弃 Master    │
      └───────────────┘           │ 返回 Slaver   │
              │                   └───────────────┘
              ▼
┌───────────────────────────────────────────────────────────┐
│ Step 2: 声明等待期（2 秒）                                   │
│ - 检测是否有其他 Master 声明                               │
│ - 如有冲突，放弃 Master                                    │
└───────────────────────────────────────────────────────────┘
              │
              ▼
┌───────────────────────────────────────────────────────────┐
│ Step 3: 创建 Master 声明                                    │
│ - Redis: SETEX declaration_key lease_time instance_id    │
│ - SQLite: UPDATE master_lock SET expires_at              │
│ - File: 写入 declaration 文件                              │
└───────────────────────────────────────────────────────────┘
              │
              ▼
┌───────────────────────────────────────────────────────────┐
│ Step 4: 创建 Master marker 文件                             │
│ - 位置：confluence/.eket_master_marker                    │
│ - 内容：instance_id, timestamp                            │
└───────────────────────────────────────────────────────────┘
              │
              ▼
┌───────────────────────────────────────────────────────────┐
│ Step 5: 启动租约续期                                       │
│ - 每 15 秒续期一次（租约 30 秒）                              │
│ - 检测续期失败，自动 relinquish                           │
└───────────────────────────────────────────────────────────┘
```

### 使用示例

```typescript
import { createMasterElection } from './core/master-election.js';

const election = createMasterElection({
  redis: {
    host: process.env.EKET_REDIS_HOST || 'localhost',
    port: parseInt(process.env.EKET_REDIS_PORT || '6379', 10),
  },
  projectRoot: process.cwd(),
  electionTimeout: 5000,    // 选举超时
  declarationPeriod: 2000,  // 声明等待期
  leaseTime: 30000,         // 租约时间
});

const result = await election.elect();

if (result.success && result.data.isMaster) {
  console.log('成为 Master');
  console.log(`Instance ID: ${election.getInstanceId()}`);
} else {
  console.log('成为 Slaver');
  console.log(`Master ID: ${result.data.masterId}`);
}
```

## 向后兼容性

### 降级策略

- 现有代码继续使用 `createMessageQueue()`，行为不变
- 新代码可使用 `createConnectionManager()` 获取更细粒度控制

### Master 选举

- 旧的 `checkMasterMarker()` 函数保留，用于向后兼容
- 新的 `start-instance.ts` 默认使用 `MasterElection`
- 可通过环境变量禁用：`EKET_USE_LEGACY_MASTER_CHECK=true`

   ```bash
   # 启动 Master
   ./eket-start --auto

   # 停止 Master 进程
   # 启动新 Instance
   ./eket-start --auto

   # 预期：新 Instance 检测到 Master 已失效，成为新 Master
   ```

