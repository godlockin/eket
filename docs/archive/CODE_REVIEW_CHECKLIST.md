# EKET 代码审查清单

## PR 审查流程

```
1. 执行智能体提交 PR
2. 自动运行 CI 检查
3. 协调智能体执行审查
4. 审查通过 → 合并
5. 审查不通过 → 反馈修改
```

## 审查清单

### 一、代码质量 [25 分]

| 检查项 | 说明 | 分值 | 得分 |
|--------|------|------|------|
| 代码规范 | 遵循项目代码风格（缩进、命名等） | 5 | |
| 函数复杂度 | 单一职责，函数不超过 50 行 | 5 | |
| 重复代码 | 无明显重复，DRY 原则 | 5 | |
| 可读性 | 变量命名清晰，逻辑易懂 | 5 | |
| 注释质量 | 关键逻辑有注释，无废话注释 | 5 | |

### 二、测试覆盖 [25 分]

| 检查项 | 说明 | 分值 | 得分 |
|--------|------|------|------|
| 单元测试 | 新增代码有对应单元测试 | 10 | |
| 边界条件 | 测试覆盖边界情况和异常 | 5 | |
| 测试通过 | 所有测试通过（CI 绿灯） | 5 | |
| 覆盖率 | 覆盖率不低于项目平均水平 | 5 | |

### 三、安全性 [20 分]

| 检查项 | 说明 | 分值 | 得分 |
|--------|------|------|------|
| 输入验证 | 所有外部输入经过验证 | 5 | |
| 敏感信息 | 无硬编码密码、密钥 | 5 | |
| SQL 注入 | 使用参数化查询 | 5 | |
| XSS 防护 | 输出经过转义 | 5 | |

### 四、功能正确性 [20 分]

| 检查项 | 说明 | 分值 | 得分 |
|--------|------|------|------|
| 需求符合 | 实现与需求一致 | 5 | |
| 验收标准 | 满足所有验收标准 | 10 | |
| 回归测试 | 不影响现有功能 | 5 | |

### 五、文档 [10 分]

| 检查项 | 说明 | 分值 | 得分 |
|--------|------|------|------|
| 变更日志 | PR 描述清晰，包含变更说明 | 5 | |
| API 文档 | 接口变更有文档更新 | 3 | |
| 类型定义 | TypeScript 类型完整 | 2 | |

---

## 审查评分标准

| 总分 | 结果 | 处理方式 |
|------|------|---------|
| 90-100 | ✅ 优秀 | 直接合并，可作为示例 |
| 75-89 | ✅ 通过 | 合并，可后续优化 |
| 60-74 | ⚠️ 需要改进 | 修改后重新审查 |
| <60 | ❌ 不通过 | 重新设计实现 |

## 常见问题检查

### JavaScript/TypeScript

- [ ] 无 `any` 类型滥用
- [ ] Promise 有 error handling
- [ ] 无内存泄漏（定时器、事件监听清理）
- [ ] 异步操作有 timeout

### Python

- [ ] 类型注解完整
- [ ] 异常处理恰当
- [ ] 无循环导入
- [ ] 使用 context manager 处理资源

### API 设计

- [ ] RESTful 路径设计
- [ ] 错误码规范
- [ ] 响应格式一致
- [ ] 有版本控制

### 数据库

- [ ] 索引优化
- [ ] 无 N+1 查询
- [ ] 迁移脚本完整
- [ ] 事务处理正确

## 审查自动化

以下检查应集成到 CI 流程：

```yaml
# .github/workflows/pr-check.yml
name: PR Checks

on: [pull_request]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run linter
        run: npm run lint

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run tests
        run: npm test

  security:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run security scan
        run: npm audit

  typecheck:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Type check
        run: npm run typecheck
```

## 审查者责任

1. **及时响应**: 24 小时内完成审查
2. **建设性反馈**: 指出问题的同时提供改进建议
3. **关注重点**: 优先关注架构、安全、功能正确性
4. **尊重作者**: 对事不对人，使用客观语言
5. **跟踪到底**: 确保反馈问题解决后再批准

## 审查示例

### 好的审查意见

✅ "这个函数的复杂度较高，建议拆分为 `validateInput()`、`processData()`、`formatOutput()` 三个函数，每个函数职责更清晰。"

✅ "这里缺少错误处理，如果 API 调用失败会导致页面卡住。建议添加 try-catch 并显示错误提示。"

### 不好的审查意见

❌ "这代码太乱了" （不具体，无建设性）

❌ "为什么要这样写？" （质问语气，应改为"能否解释一下这样设计的考虑？"）
