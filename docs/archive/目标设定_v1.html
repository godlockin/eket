目标和愿景

构建一套行为规范和执行框架给agent，让他们可以配合人类用户自主独立又能相互协作地推进任务

基础设定

系统中所有的行为都是task，从需求整理、分析到设计、研发… 系统运转主要靠不同专家设定的agent配合相应的skills进行处理 系统中有几个共享的空间： 0. agent profile设定库和skills库：随着系统运行、项目迭代而不断补充和完善 1. confluence：用来记录所有文字/文档类内容的文件树，如原始需求、整理和翻译之后的prd等等 2. jira：用来记录任务，每个任务都会包含状态（如：是否开始/进度等），需要的agent（如：技术经理、产品经理），优先级等等 3. codebase：实际放置代码、系统配置等

整个流程

  1. init：
    • 输入源：用户
      • 载体：命令行 / 文件 / IM channel
    • 行为：通过AI节点加载产品经理专家人设进行整理，对当前系统状况（如有）分析，并通过追问来完善需求
    • review：先Agent后用户
    • 输出：系统需求文档，记录系统能支持和实现的功能、交互方式
      • confluence 里的 PRD.md 文件
      • jira 里后续设计的 ticket
  2. system design：
    • 输入源：用户 + agent
      • 载体：命令行 / 文件 / IM channel + agent分析结果、系统记忆、confluence
    • 行为：根据用户补充的信息和之前的系统设定进行分析、拆解和细化需求
    • review：先Agent后用户
    • 输出：用户交互定义、系统功能模块的定义、功能模块协作方式定义
      • confluence 里的 system_design.md 文件
      • jira 里后续设计的 ticket
  3. tech defination:
    • 输入源：用户 + agent
      • 载体：命令行 / 文件 / IM channel + confluence
    • 行为：根据用户补充的信息和之前的系统设定进行分析、定义技术设计，包含数据模块、系统拓扑、数据链路、第三方系统依赖
    • review：先Agent后用户
    • 输出：用户交互定义、系统功能模块的定义、功能模块协作方式定义
      • confluence 里的 tech_defination.md 文件
      • jira 里后续设计的 ticket
  4. stage defination:
    • 输入源：用户 + agent
      • 载体：命令行 / 文件 / IM channel + confluence
    • 行为：根据用户补充的信息和之前的系统设定进行分析，确定当前stage需要完成的功能以及未来发展的roadmap，以及所需要的能力和依赖内容，如：
      • 功能：识别图片中的沙发种类的能力
      • 能力：
        • 图片上传
        • 图片预处理
        • 图片存储
        • AI模型部署
        • AI模型识别
        • 结果校验
      • 依赖内容：
        • 本地存储
        • 云存储
        • AI推理服务
    • review：先Agent后用户
    • 输出：当前stage需要完成的功能以及未来发展的roadmap、系统中的基础能力、系统依赖的内容及用法
      • confluence 里的 stage_defination_xxx.md 文件
      • confluence 里的 meta_function.md 文件
      • confluence 里的 dependencies.md 文件
      • jira 里后续设计的 ticket
  5. implement plan:
    • 输入源：agent
      • 载体：confluence
    • 行为：根据系统设定以及stage设置进行任务的拆分
    • review：主要agent，遇到需要决策的block问题则系统暂停，创建人类/master节点依赖的任务，然后等待反馈
    • 输出：具体任务的拆分
      • jira 里具体实现的 ticket
  6. implement：
    • 输入源：agent
      • 载体：confluence + jira
    • 行为：
      • 初始化多个AI instance
      • 根据任务优先级的需要加载对应的agent profile 和 skills
      • 承接任务
      • 处理和实现任务要求的内容
      • 汇报并标记任务状态
    • review：主要agent，遇到需要决策的block问题则系统暂停，创建人类/master节点依赖的任务，然后等待反馈
    • 输出：如果任务已经足够原子化无需继续拆分，则直接执行；如果任务还需要进一步拆分，则拆分并创建子任务，并标记任务所需要的profile
      • jira 里具体实现的 ticket
  7. iteration：
    • 输入源：agent
      • 载体：confluence + jira
    • 行为：
      • 根据当前项目状态、进度进行下一阶段的stage defination -> implement plan -> implement
    • review：主要agent，结果需要用户进行确认
    • 输出：
      • confluence 里的 stage_defination_xxx.md 文件
      • jira 里后续设计的 ticket

agent的加载和行为

  1. 新的 agent 在创建之后会读取基础设定，了解整体框架的基础设定
  2. 如果系统中没有confluence、jira、codebase等，则认为系统处于 init 状态，加载需求整理、分析等的profile开始监听和引导用户给出需求
  3. 如果系统中已经存在confluence、jira、codebase等，则认为系统已经完成 init 状态，查看 jira 里的任务列表，根据优先级查看 ticket，然后根据ticket的需要加载对应“专家”的profile和skills并按照流程设置进行推进

任务处理

  1. 获取ticket之后将状态标记为assigned和当前agent的标记
  2. 对任务进行分析、推演
    • 如果任务可以进行拆解和协同，则将当前任务拆解成子任务，创建在 jira 里
    • 如果任务不可被拆分，则开始做任务实施计划
  3. 对任务进行实现
  4. 对任务结果进行分析和unit test
  5. 提交pr并创建merge 任务
  6. 对于高级agent收到merge任务之后，对任务结果进行review、分析和打分
    • 如果review通过则进行merge
    • 如果review结果有争议，则提示用户做决策
    • 如果review不通过，则退回agent继续升级和优化

agent定义和分层

  1. 顶级agent：
    • product manager、project manager、scrum master、architecture、HR等管理层agent
    • 对用户需求进行收集、分析，规划功能、stage
    • 做系统设定、review和merge
    • 监控整个项目进度以及决策需求
    • 根据项目、任务的具体进度、状态需求，分析静态定义agent存在能力短板时，动态搜索、定义、创建和加载新的专家profile/skills
  2. 静态agent：
    • 前端、后端、设计、算法、AI、测试、运营、运维、安全等执行层agent
    • 根据系统具体任务需要预设一些能具体做事的专家profile，每个都是某个领域的专家，可以相互之间协同工作
  3. 动态agent：
    • 除了静态agent之外，还需要根据项目实际情况及能力的欠缺动态设置和加载某些领域专家
    • 这些领域专家更多的是补充某些业务领域知识，如operation、marketing、branding等等
  4. skills：
    • 有个共享的树状的skills库，里面记录着某个专家在执行某个任务时需要的能力和workflow

agent profile库和skills库

  • 这个agent profile会是个树状结构、具有专家定义的库，它会被全局共享不断升级和迭代，里面的profile都是领域专家的完整定义
  • 这个skills会是个树状结构、具有专家定义的库，它会被全局共享不断升级和迭代，里面的skills都是原子能力，如python开发、pdf读取等等

agent profile结构

  1. 由 yml 文件进行定义，在系统初始化的时候进行加载
  2. 包含属性：
    • 名称
    • 种类
    • 描述
    • 职责范围
    • 性格特征
      • MBTI
      • 能力矩阵
      • 工作风格
      • 沟通风格
      • 决策风格
      • 压力反应
    • 能力矩阵
      • 专业背景
      • skills组合（随着任务的推进动态搜索和加载）
    • 输入定义和约束
    • 输出格式和内容
    • 决策策略
    • 生命周期

skills 结构

  1. 由 yml 文件定义，在agent进行任务推进的时候动态加载
  2. 包含属性：
    • 适用场景
    • 依赖资源
    • 执行方法
    • SOP
    • 预期结果
    • 失败应对策略

confluence

所有系统内文档类信息，包括项目背景、需求、架构、设计、代码规范、测试规范等 - 项目背景：项目的背景介绍，包括项目的目标、受众、业务场景、价值和意义等 - 需求：项目的需求规格说明书，包括功能需求、非功能需求、接口需求、项目不同阶段的状态及需求变更记录等 - 架构：项目的架构设计，包括系统架构、组件架构、模块架构、系统拓扑、数据链路、系统组件等 - 设计：项目的详细设计，包括数据库设计、接口设计、前端设计、后端设计等 - 交互：项目的行为、项目与用户、项目模块之间的关系 - 代码规范：项目的代码规范，包括代码风格、命名规范、注释规范等 - 测试规范：项目的测试规范，包括测试用例设计、测试环境搭建、测试执行等 - release信息：项目的发布信息，包括发布版本、发布日期、发布内容、当前的状态、下阶段的目标等

jira

负责管理项目的问题和任务，包括需求管理、缺陷管理、任务分配、进度跟踪等。 - 项目功能管理：按功能模块进行组织，包含每个功能模块的需求规格、实现功能、交互接口、使用方式等 - 项目缺陷管理：按缺陷类型进行组织，包含每个缺陷类型的缺陷描述、影响范围、修复优先级、修复状态等 - 项目任务管理：作为功能ticket的子模块，会按照功能模块进行拆分，包含具体的、无法被进一步细分的任务，每个任务都有一个独立的ticket，包含任务描述、任务分配、任务进度、任务完成状态、任务依赖关系等 - ticket索引：为了方便智能体在jira里查询和管理任务，会为每个功能模块创建一个独立的ticket索引，索引里包含该功能模块下所有的任务ticket - 每个任务ticket都有一个独立的编号，编号格式为功能模块名称-任务序号，如user_management-001、user_management-002等 - 每个任务ticket的编号都是唯一的，不会重复 - 每个任务ticket都会注明当前状态、优先级、依赖关系方便智能体读取

codebase:

负责管理项目的代码，包括项目的代码库、版本控制、代码质量检查等。 - 项目说明文件：项目的基础说明，包含项目的名称、目标、启动方式、代码仓库地址、项目文档地址等 - 代码目录：项目的代码目录，包含项目的所有代码文件，按照功能模块进行组织，每个模块有一个独立的目录，目录名就是模块的名称，目录下包含该模块的所有代码文件 - 配置目录：项目的配置目录，包含项目的所有配置文件，包含系统内所有的基础配置、环境变量、数据库连接配置、日志配置、安全配置等 - 测试目录：项目的测试目录，包含项目的所有测试代码文件，按测试种类进行组织，如单元测试、集成测试、端到端测试等，每个组织中根据测试对象进行组织，如根据功能模块、根据接口等 - 部署文件：项目的部署文件，包含项目的所有部署配置文件，如Dockerfile、Kubernetes配置文件、环境依赖、部署脚本等
