我要构建一个agent group的CLAUDE.md、AGENTS.md、SKILLS.md，我需要你帮我一起脑爆和完善整个文件结构树。

CLAUDE.md中会包含系统的目标、愿景、流程、框架、制约等

AGENTS.md会包含智能体的定义、写作方式等

SKILLS.md会包含每个智能体所需要的具体的unique的技能

## 愿景

把「多级智能体协作研发」变成一套可复用、可演进、可审计的工程化底座：用清晰的文档契约（CLAUDE/AGENTS/SKILLS）统一目标与边界，用标准化的任务与版本流程驱动稳定交付，让任何新项目都能在最短时间内进入高质量迭代。

## 意义

1. 降低协作成本：把“口头约定”固化为可检索的文档与结构化流程，减少重复沟通与误解。
2. 提升可控性：通过职责、边界与制约的明确化，让决策与实现可追溯、可复盘、可调整。
3. 放大生产力：让智能体从零散的提示词执行升级为系统化协作，持续产出稳定的设计、代码与测试结果。
4. 支持规模化：用分层智能体与SKILL单元化能力，适配不同规模与不同技术栈的研发项目。

## 背景

这是一个由多级智能体共同协作完成研发项目用的智能体组，每个智能体都有自己的职责和技能，共同完成项目的开发。

## 主要组件

1. 3个独立的git项目：
   - confluence: 负责管理项目的文档，包括项目背景、需求、架构、设计、代码规范、测试规范等。
   - code\_repo: 负责管理项目的代码，包括项目的代码库、版本控制、代码质量检查等。
   - jira: 负责管理项目的问题和任务，包括需求管理、缺陷管理、任务分配、进度跟踪等。
2. 3组智能体：
   智能体的层级关系为：预定义的持续活跃智能体 -> 预定义的按需加载的智能体 -> 自定义智能体
   - 预定义的持续活跃智能体：
     这些智能体会持续运行，监听项目的变化，及时响应和协调项目的进度
     - 需求分析智能体：
       - 负责对外交互，收集和分析用户需求，整理成功能点，需求规格说明书等
       - 会和技术管理智能体协调，确保需求的合理性、可实现性和可扩展性
       - 在讨论之后会给终端用户和需求提供方一个反馈，包括哪些需求可以实现，哪些需求需要调整或重新定义，以及实现这些需求的成本和风险
       - 在达成共识之后，会更新到confluence里的相关文档中
     - 技术管理智能体：
       - 负责协调项目的技术方向和实现方案，确保项目的技术质量和进度
       - 会和需求分析智能体协调，确保需求的实现符合技术方案
       - 在确定技术方案之后，会更新到confluence里的相关文档中，同时将任务进行拆解，定义接口、边界、权责等，记录在jira里
     - 项目管理智能体：
       - 负责协调项目的进度和资源，确保项目按计划进行
       - 会和需求分析智能体、技术管理智能体协调，确保项目的进度和质量
       - 在项目进度出现问题时，会及时通知相关智能体，协调解决问题
     - 文档和记忆监控智能体：
       - 负责监控项目的文档和记忆，确保文档的及时性、准确性和完整性
       - 会将智能体和人、智能体之间的沟通记录进行分析、总结和记录，然后保存在confluence里的相关文档中
   - 预定义的按需加载的智能体：
     这些智能体会在项目需要时被加载，完成项目的特定任务，自主地从jira里领取任务，分析和处理任务，完成任务后会将任务状态更新到jira里。
     如果智能体在处理任务时发现问题，会及时通知「持续活跃智能体」组，协调解决问题。
     - 视觉设计智能体：
       - 负责项目的视觉设计，包括界面布局、颜色方案、图标设计等
       - 输入为需求规格说明书、项目背景、项目目标等
       - 输出为视觉设计稿，包括界面布局、颜色方案、图标设计等，最终会保存在confluence里的相关文档中
     - 测试智能体：
       - 负责执行项目的测试用例，包括单元测试、集成测试、端到端测试等
       - 输入为需求规格说明书、项目背景、项目目标、测试用例等
       - 输出为测试结果，包括测试用例的执行结果、测试覆盖率、测试缺陷等，最终会更新在confluence里的相关文档中
     - 前端开发智能体：
       - 负责实现项目的前端界面，包括页面布局、交互逻辑、用户体验等
       - 输入为需求规格说明书、项目背景、项目目标、视觉设计稿、交互接口定义等
       - 输出为前端代码，包括HTML、CSS、JavaScript等，最终会保存在code\_repo里的相关目录中
       - 在交付任务时会将前端代码提交到code\_repo里，同时在jira里更新任务状态
     - 后端开发智能体：
       - 根据项目的具体需要来加载对应的技术能力
       - 负责实现项目的后端逻辑，包括数据库交互、业务逻辑、接口设计等
       - 输入为需求规格说明书、项目背景、项目目标、数据库设计稿、接口定义等
       - 输出为后端代码，包括Java、Python、Ruby等，最终会保存在code\_repo里的相关目录中
       - 在交付任务时会将后端代码提交到code\_repo里，同时在jira里更新任务状态
     - 运维部署智能体：
       - 负责项目的运维部署，包括服务器配置、数据库部署、应用部署等
       - 输入为需求规格说明书、项目背景、项目目标、服务器配置、数据库配置、应用配置等
       - 输出为运维部署记录，包括服务器配置、数据库配置、应用配置、部署日志等，保存在code\_repo的相关目录中，同时会更新confluence里的相关文档
     - 存储维护智能体：
       - 负责项目的存储维护，包括所有三方存储的备份、恢复、监控、优化等
       - 输入为需求规格说明书、项目背景、项目目标、三方存储配置等
       - 输出为存储维护记录，包括备份策略、恢复策略、监控策略、优化策略等，保存在code\_repo的相关目录中，同时会更新confluence里的相关文档
   - 其他自定义的智能体：
     这些智能体会根据项目的具体需要来定义和加载，负责项目的特定任务。
3. 记忆系统
   - 每个智能体实例会有一个独立的记忆体系，用于存储和管理智能体在任务执行过程中的信息
   - 记忆体系包括短期记忆、长期记忆和外部记忆三部分
   - 短期记忆用于存储当前任务执行过程中的临时信息，如任务描述、需求规格说明书、项目背景、项目目标等
   - 长期记忆用于存储智能体在任务执行过程中学习到的知识和经验，如设计模式、最佳实践、代码规范等
   - 外部记忆用于存储智能体在任务执行过程中需要访问的外部信息，如数据库设计稿、接口定义、三方存储配置等
4. SKILL体系
   - 整个系统包含一套完整的SKILL体系，每个SKILL都是独立且可配置的能力，每个都是抽象的可复用的能力unit
   - 智能体在执行自己职责的时候可能会依赖不同的SKILL，根据任务的需求和上下文，智能体会动态地加载和使用适当的SKILL来完成任务

### git项目的实现

#### git项目的结构

1. confluence: 负责管理项目的文档，包括项目背景、需求、架构、设计、代码规范、测试规范等
   - 项目背景：项目的背景介绍，包括项目的目标、受众、业务场景、价值和意义等
   - 需求：项目的需求规格说明书，包括功能需求、非功能需求、接口需求、项目不同阶段的状态及需求变更记录等
   - 架构：项目的架构设计，包括系统架构、组件架构、模块架构、系统拓扑、数据链路、系统组件等
   - 设计：项目的详细设计，包括数据库设计、接口设计、前端设计、后端设计等
   - 代码规范：项目的代码规范，包括代码风格、命名规范、注释规范等
   - 测试规范：项目的测试规范，包括测试用例设计、测试环境搭建、测试执行等
   - release信息：项目的发布信息，包括发布版本、发布日期、发布内容、当前的状态、下阶段的目标等
2. code\_repo: 负责管理项目的代码，包括项目的代码库、版本控制、代码质量检查等。
   - 项目说明文件：项目的基础说明，包含项目的名称、目标、启动方式、代码仓库地址、项目文档地址等
   - 代码目录：项目的代码目录，包含项目的所有代码文件，按照功能模块进行组织，每个模块有一个独立的目录，目录名就是模块的名称，目录下包含该模块的所有代码文件
   - 配置目录：项目的配置目录，包含项目的所有配置文件，包含系统内所有的基础配置、环境变量、数据库连接配置、日志配置、安全配置等
   - 测试目录：项目的测试目录，包含项目的所有测试代码文件，按测试种类进行组织，如单元测试、集成测试、端到端测试等，每个组织中根据测试对象进行组织，如根据功能模块、根据接口等
   - 部署文件：项目的部署文件，包含项目的所有部署配置文件，如Dockerfile、Kubernetes配置文件、环境依赖、部署脚本等
   - 附加文件：项目的其他附加文件，如项目的编译、部署脚本等
3. jira: 负责管理项目的问题和任务，包括需求管理、缺陷管理、任务分配、进度跟踪等。
   - 项目功能管理：按功能模块进行组织，包含每个功能模块的需求规格、实现功能、交互接口、使用方式等
   - 项目缺陷管理：按缺陷类型进行组织，包含每个缺陷类型的缺陷描述、影响范围、修复优先级、修复状态等
   - 项目任务管理：作为功能ticket的子模块，会按照功能模块进行拆分，包含具体的、无法被进一步细分的任务，每个任务都有一个独立的ticket，包含任务描述、任务分配、任务进度、任务完成状态、任务依赖关系等
   - ticket索引：为了方便智能体在jira里查询和管理任务，会为每个功能模块创建一个独立的ticket索引，索引里包含该功能模块下所有的任务ticket
     - 每个任务ticket都有一个独立的编号，编号格式为`功能模块名称-任务序号`，如`user_management-001`、`user_management-002`等
     - 每个任务ticket的编号都是唯一的，不会重复
     - 每个任务ticket都会注明当前状态、优先级、依赖关系方便智能体读取

#### git项目的使用流程

1. 项目初始化：
   - 创建三个独立的git项目，分别是confluence、code\_repo、jira
   - 每个项目都有一个独立的git仓库，仓库的名称就是项目的名称，如`my_project_confluence`、`my_project_code_repo`、`my_project_jira`
   - `my_project_confluence`和`my_project_jira`会以submodule的方式挂载在`my_project_code_repo`里方便智能体小组的反问和维护
   - 初始化「持续活跃智能体」小组，这个顶层小组会在整个项目的生命周期中持续保持活跃，由他们对项目进行分析和协调、任务进度追踪和项目整体状态、设计的更新和演进
2. 项目推进：
   - 顶层「持续活跃智能体」小组会根据用户需求进行分析、讨论、跟用户交互和反馈，然后进行任务的大致拆解，更新`confluence`文档和`jira`里的任务ticket
   - 然后根据系统承载能力初始化下层「持续活跃智能体」小组，每个小组负责一个功能模块的任务执行，每个小组会有一个独立的git分支，分支的名称就是功能模块的名称，如`user_management`、`order_management`等
   - 下层「持续活跃智能体」小组会递归地对`jira`里的任务ticket进行分析和拆解，按照不同的领域需求唤起「预定义的按需加载的智能体」进行任务的执行
   - 「预定义的按需加载的智能体」会根据`jira` ticket的具体需求按照自己的设定进行分析、设计、实现和测试，等验收完成后，会将实现的代码提交到`code_repo`里的对应功能模块分支，并更新`jira` ticket的状态为已完成
   - 顶层「持续活跃智能体」小组会持续监控所有`jira` ticket的状态，定时检查所有任务的状态、进度、依赖关系，根据任务的状态和进度进行协调和调整，确保项目的进度和质量
3. 项目交付和验收：
   - 当所有`jira` ticket的状态都为已完成，且所有任务的进度都已完成，项目就会进入验收阶段
   - 验收阶段会引入独立的红队对项目进行review、验收测试、质量检查，如果发现问题会创建新的`jira` ticket进行修复
   - 修复完成后，会重新进行验收测试和质量检查，直到项目通过所有的验收测试和质量检查
   - 验收阶段完成之后，会在`code_repo`里创建一个独立的项目`release`分支，包含所有的代码、配置、测试等文件，同时会更新`confluence`的`release信息`章节

## 智能体的使用流程

### 自定义智能体的定义和加载

1. 先定义智能体的基础模板，包含智能体的名称、功能描述、背景、思维框架、决策链、输入输出格式、技术能力等，保存为 `agents/agent_base_template.yml`
2. 根据说明来定义预定义的智能体模板，保存在`agents/predefined_agents/`目录下，每个智能体的文件名就是智能体的名称，文件内容就是智能体的定义
3. 在任务推进过程中，根据项目的具体需要，动态创建需要的智能体模板，保存在`agents/active_agents/`目录下，每个智能体的文件名就是智能体的名称，文件内容就是智能体的定义
4. 在任务进行过程中，会根据不同的智能体模板初始化不同的智能体实例，每个智能体实例会有独立的大模型连接及完整的生命周期，包括初始化、运行、销毁等阶段

### 智能体的运行和管理

1. 智能体的初始化：在任务推进过程中，根据需要动态创建智能体实例，初始化时会根据智能体模板加载大模型连接、设置智能体的参数、初始化智能体的状态等
2. 智能体的运行：智能体实例会在任务进行过程中持续运行，根据输入数据和任务需求，利用大模型的能力进行分析、设计、实现和测试等
3. 智能体的销毁：当任务完成或项目结束时，会销毁所有智能体实例，释放大模型连接、关闭模型会话、清除智能体的状态等

### 智能体的协作和管理

1. 顶层「持续活跃智能体」小组会负责协调和管理所有的智能体实例，包括初始化、运行、销毁等阶段
2. 下层「持续活跃智能体」小组会根据功能模块的需求，递归地对`jira`里的任务ticket进行分析和拆解，唤起相应的智能体实例进行任务的执行
3. 智能体实例之间会根据任务的依赖关系进行协调和通信，确保任务的顺序和进度
4. 每个智能体实例会有自己的独立作业空间和记忆空间，用于存储任务相关的信息、中间结果、临时数据等；同时会有项目组记忆空间和整体项目记忆空间，用于存储项目组和整体项目的相关信息、全局状态、共享资源等
5. 下一级的智能体实例会根据任务的依赖关系，从项目组记忆空间和整体项目记忆空间中获取必要的信息，进行任务的执行和协调
