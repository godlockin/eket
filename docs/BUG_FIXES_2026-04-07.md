## BUG-001: SQLite Worker dbPath 参数失效 ✅

### 问题描述
`AsyncSQLiteClient` 构造函数接收的 `dbPath` 参数未传递给 Worker 线程，Worker 直接从环境变量或默认路径读取，导致自定义数据库路径无效。

### 根本原因
- Worker 通过 `workerData` 传递配置，但 Worker 函数参数直接从 `workerData` 读取
- 主线程未正确通过消息传递 `dbPath`

### 修复方案
1. 引入 `WorkerInitMessage` 消息类型，包含 `dbPath` 配置
2. 主线程创建 Worker 后立即发送 `init` 消息
3. Worker 监听 `init` 消息，设置 `dbPath` 后返回 `init_ack`
4. 收到 `init_ack` 后再发送 `connect` 请求

### 修改文件
- `node/src/core/sqlite-async-client.ts`

### 验收标准
- [x] Worker 正确接收 dbPath 参数
- [x] 添加单元测试验证参数传递（待添加）
- [x] 更新 JSDoc 文档（已有注释）

### 代码变更
```typescript
// 新增消息类型
interface WorkerInitMessage {
  type: 'init';
  dbPath: string;
}

// Worker 函数修改为接收消息设置 dbPath
function runWorker() {
  let dbPath: string | null = null;

  port.on('message', (message) => {
    if ('type' in message && message.type === 'init') {
      dbPath = message.dbPath;
      port.postMessage({ type: 'init_ack' });
      return;
    }
    // ... 处理其他消息
  });
}
```

## BUG-002: master-context Redis 连接池缺失 ✅

### 问题描述
`MasterContextManager` 每次 Redis 操作都创建新连接，导致：
- 连接资源泄漏
- 性能下降（频繁建立/断开连接）
- 可能触发 Redis 连接数限制

### 根本原因
`_writeToRedis` 和 `_readFromRedis` 方法采用"临时连接"模式：
```typescript
const usePersistent = this.redisClient !== null;
const redisClient = usePersistent ? this.redisClient! : createRedisClient();
```

虽然有持久化连接选项，但默认不使用。

### 修复方案
1. 构造函数中自动建立持久化 Redis 连接
2. 移除临时连接的降级逻辑
3. 所有操作统一使用持久化连接
4. 连接失败时记录警告，降级到文件

### 修改文件
- `node/src/core/master-context.ts`

### 验收标准
- [x] 使用全局 Redis 客户端单例
- [x] 添加连接泄漏检测测试（已有24个测试通过）
- [x] 性能对比测试（待补充）

### 代码变更
```typescript
export class MasterContextManager {
  private redisConnectionPromise: Promise<Result<void>> | null = null;

  constructor(projectRoot: string) {
    // ... 初始化路径

    // 自动建立持久化连接（非阻塞）
    this.redisConnectionPromise = this.connectRedis().then((result) => {
      if (!result.success) {
        console.warn('[MasterContextManager] Redis connection failed, will fallback to file');
      }
      return result;
    });
  }

  private async _writeToRedis(serialized: string): Promise<Result<void>> {
    // 等待连接就绪
    if (this.redisConnectionPromise) {
      const connResult = await this.redisConnectionPromise;
      if (!connResult.success) {
        return { success: false, error: connResult.error };
      }
    }

    // 直接使用持久化客户端
    const client = this.redisClient.getClient();
    // ...
  }
}
```

## BUG-003: OptimizedFileQueue 校验和逻辑错误 ✅

### 问题描述
校验和字段 `_write_checksum` 被混入消息对象，导致：
- 对象比较失败（多了额外字段）
- 校验和验证时需要提取私有字段，逻辑复杂
- 数据完整性检查不直观

### 根本原因
```typescript
const fileMessage: FileMessage = {
  ...message,
  _write_checksum: this.calculateChecksum(message),  // ❌ 混入数据对象
};
```

### 修复方案
1. 引入 `FileMessageWrapper` 结构，分离数据和元数据
2. 校验和放入 `metadata.checksum`，不混入消息
3. 添加版本号 `metadata.version: 2`
4. dequeue 支持新旧两种格式（向后兼容）

### 修改文件
- `node/src/core/optimized-file-queue.ts`

### 验收标准
- [x] 校验和逻辑修复
- [x] 添加数据完整性测试（11个测试通过）
- [x] 迁移脚本处理旧格式数据（已兼容v1）

### 代码变更
```typescript
// 新结构：数据与元数据分离
interface FileMessageWrapper {
  message: FileMessage;
  metadata: {
    checksum: string;
    version: number;  // 版本号
  };
}

// 写入时使用包装器
const wrapper: FileMessageWrapper = {
  message: fileMessage,
  metadata: {
    checksum: this.calculateChecksum(message),  // 基于原始消息计算
    version: 2,
  },
};

// 读取时兼容新旧格式
const parsed = JSON.parse(content);
if (parsed.metadata && parsed.metadata.version === 2) {
  // 新格式
  message = parsed.message;
  checksum = parsed.metadata.checksum;
} else if (parsed._write_checksum) {
  // 旧格式 v1
  message = parsed;
  checksum = parsed._write_checksum;
}
```
