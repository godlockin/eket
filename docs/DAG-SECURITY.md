# DAG Security Boundaries

> EKET DAG YAML Trust Model and Security Limits

## Trust Model

**Assumption**: DAG YAML files are **trusted input** from CI/CD-controlled repositories.

### Supported Scenarios

- DAG files checked into version control
- Generated by trusted automation pipelines
- Reviewed via standard code review process

### Unsupported Scenarios (Security Risk)

- User-uploaded YAML files
- YAML from untrusted network sources
- Runtime-generated YAML without validation

## Security Limits

| Field | Limit | Rationale |
|-------|-------|-----------|
| `nodes` | `maxItems: 1000` | Prevent memory exhaustion (DoS) |
| `script` | `maxLength: 10000` | Limit command injection surface |
| `description` | `maxLength: 500` | Prevent log flooding |
| `retry` | `max: 10` | Prevent infinite retry loops |
| `timeout` | `max: 86400` (24h) | Prevent zombie processes |

## Risk: `script` Field

The `script` field is executed directly as a shell command. This is **by design** for CI/CD automation.

**Implications**:

1. DAG author has full shell access within the execution environment
2. No sandboxing or command whitelisting is applied
3. Environment variables are inherited by scripts

**Mitigations**:

1. Only accept DAGs from trusted repositories
2. Enforce code review for DAG changes
3. Run DAG executors with minimal privileges (principle of least privilege)
4. Consider container isolation for untrusted workloads

## Validation Enforcement

Security limits are enforced at three layers:

| Layer | File | Validation |
|-------|------|------------|
| JSON Schema | `jira/schemas/dag.schema.json` | Static schema validation |
| TypeScript | `node/src/schemas/dag.ts` | Runtime validation |
| Rust | `rust/crates/eket-engine/src/dag/schema.rs` | Native validation |

All layers **must** be kept in sync. Changes to limits require updates across all three.

## Incident Response

If a malicious DAG is detected:

1. Immediately revoke repository access for the source
2. Audit recent DAG executions from that source
3. Review CI/CD pipeline for unauthorized modifications
4. Update allowlists/blocklists as needed
