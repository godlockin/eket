## 初衷与愿景

### 初衷

在 AI 智能体技术快速发展的今天，我们发现：

1. **单点智能体能力强大但协作困难** - 单个 Claude Code 实例可以完成复杂任务，但多个实例之间缺乏可靠的协作机制
2. **研发流程缺乏标准化** - 每次新项目都需要重新定义流程、沟通方式和工作规范
3. **知识无法沉淀** - 项目经验、技术决策、最佳实践散落在对话历史中，无法有效传承
4. **规模化困难** - 随着项目复杂度增加，智能体之间的协调成本呈指数级增长

EKET 框架的诞生就是为了解决这些问题：**将多智能体协作研发变成一套可复用、可演进、可审计的工程化底座**。

### 愿景

> 让智能体从零散的提示词执行升级为系统化协作，持续产出稳定的设计、代码与测试结果。

我们期望通过 EKET 框架：

- **降低协作成本** - 把"口头约定"固化为可检索的文档与结构化流程
- **提升可控性** - 让决策与实现可追溯、可复盘、可调整
- **放大生产力** - 系统化协作而非单点执行
- **支持规模化** - 适配不同规模与不同技术栈的研发项目

## 系统目标

EKET 框架是一个**多智能体协作系统**，用于 AI 智能体（如 Claude Code）进行研发任务的自动化处理。系统通过：

- 规范化的流程
- 清晰的职责划分
- 可靠的通信机制

实现多个智能体实例的高效协作。

### 核心能力

| 能力 | 说明 |
|------|------|
| **任务自动化** | 从需求输入到代码提交的端到端自动化 |
| **多智能体协作** | 支持多个智能体实例并行工作、协调配合 |
| **状态持久化** | 通过 Git 实现任务状态、代码变更的持久化存储 |
| **人类在环** | 关键决策点引入人类确认，确保方向正确 |
| **知识沉淀** | 项目经验、技术决策自动归档，形成组织记忆 |

## 核心设计理念

### 1. 混合架构

```
┌─────────────────────────────────────────────────────────────┐
│                    顶层协调智能体小组                         │
│  ┌─────────────┬─────────────┬─────────────┬─────────────┐  │
│  │ 需求分析师   │  技术经理    │  项目经理    │  文档监控员  │  │
│  └─────────────┴─────────────┴─────────────┴─────────────┘  │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                  执行层智能体（去中心化网络）                   │
│  ┌─────────┬─────────┬──────────┬──────────┬────────┬──────┐ │
│  │ 设计师   │ 测试员   │ 前端开发  │ 后端开发   │ 运维    │ 存储 │ │
│  └─────────┴─────────┴──────────┴──────────┴────────┴──────┘ │
└─────────────────────────────────────────────────────────────┘
```

- **顶层协调小组**：4 个常驻智能体，负责任务分析、资源协调、PR 审核
- **执行层网络**：6+ 个按需智能体，负责具体任务执行

### 2. 混合通信

| 方式 | 用途 | 位置 |
|------|------|------|
| **消息队列** | 实时通信 | `shared/message_queue/` |
| **Git Commit** | 状态持久化 | 三大 Git 仓库 |
| **共享状态文件** | 跨实例同步 | `shared/.state/` |

### 3. 混合决策

| 决策类型 | 决策者 | 流程 |
|---------|-------|------|
| 常规任务 | 执行智能体自主 | 领取 → 执行 → 提交 PR |
| 技术方案 | 技术经理 | 提议 → 审批 |
| 需求变更 | 需求分析师 + 项目经理 | 分析影响 → 评估优先级 → 决策 |
| 架构变更 | 技术经理 + 人类确认 | 提案 → Review → 人类批准 |

### 4. 混合生命周期

| 类型 | 生命周期 | 示例 |
|------|---------|------|
| **协调智能体** | 常驻 | 需求分析师、技术经理、项目经理 |
| **执行智能体** | 按需 | 设计师、开发、测试、运维 |

### 5. 文件系统隔离

```
project/
├── shared/      # 共享区 - 三大 Git 仓库
├── private/     # 私有区 - 智能体工作空间
├── inbox/       # 输入区 - 人类需求
├── outbox/      # 输出区 - Review 请求
└── tasks/       # 任务区 - 任务定义
```

### 三大 Git 仓库

| 仓库 | 用途 | 内容 |
|------|------|------|
| `shared/confluence` | 文档中心 | 需求文档、架构设计、技术规范、会议记录 |
| `shared/code_repo` | 代码仓库 | 源代码、测试、部署配置、CI/CD |
| `shared/jira` | 任务管理 | 需求票、缺陷票、任务票、状态追踪 |

### 仓库关系

```
my_project_code_repo/          # 主代码仓库
├── .gitmodules                # 指向 confluence 和 jira
├── confluence/                # submodule → confluence 仓库
├── jira/                      # submodule → jira 仓库
├── src/                       # 源代码
└── ...
```

### 智能体定义

每个智能体由 YAML 配置文件定义：

```yaml
# agents/agent_base_template.yml
name: <agent_name>
type: coordinator|executor
version: 1.0.0
description: <简短描述>

capabilities:
  - <capability_tag_1>
  - <capability_tag_2>

inputs:
  - type: jira_ticket
    fields: [title, description, acceptance_criteria]

outputs:
  - type: jira_transition
    action: transition_ticket
  - type: pull_request
    format: github_pr

decision_policy:
  auto_decide: true|false
  auto_decide_scope:
    - <允许自主决策的场景>
  escalation_rules:
    - condition: <条件>
      escalate_to: <目标智能体>

lifecycle:
  mode: persistent|on_demand
  spawn_method: process|thread
```

## Workflow 设计

### 标准任务流转

```
1. 人类输入需求 → 需求分析师接收
       ↓
2. 需求分析师 → 拆解为 Jira tickets
       ↓
3. 执行智能体 → 轮询并领取任务
       ↓
4. 执行智能体 → 创建 git 分支 → 开发 → 提交 PR
       ↓
5. 执行智能体 → 通过消息队列唤醒协调智能体
       ↓
6. 协调智能体 → review PR → 合并或要求修改
       ↓
7. 任务完成 → 更新 Jira 状态 → 通知相关方
```

### 分支管理

```
分支命名规范:
- feature/{ticket-id}-{short-desc}    # 功能开发
- bugfix/{ticket-id}-{short-desc}     # 缺陷修复
- hotfix/{ticket-id}-{short-desc}     # 紧急修复
- docs/{ticket-id}-{short-desc}       # 文档更新

合并流程:
1. 执行智能体完成开发
2. 执行智能体创建 PR，指定协调智能体为 reviewer
3. 执行智能体发送 review 请求消息
4. 协调智能体接收消息 → 执行 review
5. review 通过 → 合并到主分支
6. review 不通过 → 评论反馈 → 执行智能体修改
```

### 反馈循环

```
┌──────────────┐
│  智能体执行   │
│  一轮工作     │
└──────┬───────┘
       │
       ▼
┌──────────────────────────────────┐
│ 创建状态报告                       │
│ inbox/human_feedback/            │
│ status-<task-id>-<timestamp>.md  │
└──────┬───────────────────────────┘
       │
       │ 状态：pending_confirmation
       ▼
┌──────────────┐
│  等待人类确认  │
│  (智能体暂停)  │
└──────┬───────┘
       │
       │ 人类回复
       ▼
┌──────────────────────────────────┐
│ 人类在同一个文件中回复             │
│ - 确认成果                        │
│ - 选择推荐答案                    │
│ - 提供额外指示                    │
└──────┬───────────────────────────┘
       │
       ▼
┌──────────────┐
│  智能体读取   │
│  人类反馈     │
└──────┬───────┘
       │
       │ 根据反馈
       ▼
┌──────────────────────────────────┐
│ 继续执行或修改                    │
│ - 进入下一阶段                    │
│ - 调整方案重新执行                │
└──────────────────────────────────┘
```

## Agent 宪法与行为方式

### 第一原则：安全优先

> 任何操作不得损害项目、数据或系统安全。

- 删除操作必须经过确认
- 敏感信息不得提交到 Git
- 发现安全风险立即上报

### 第二原则：人类在环

> 重大决策必须引入人类确认。

| 必须上报的场景 | 上报对象 |
|---------------|---------|
| 任务复杂度超出预估（>4 小时） | 项目经理 |
| 技术选型存在争议 | 技术经理 |
| 需要跨模块协调 | 项目经理 |
| 发现需求不明确或矛盾 | 需求分析师 |
| 遇到阻塞性问题超过 30 分钟 | 协调智能体小组 |

### 第三原则：文档即契约

> 所有决策、变更、经验必须文档化。

- 需求分析完成 → 更新 Confluence 需求文档
- 技术方案确定 → 更新 Confluence 架构文档
- 代码提交 → 关联 Jira ticket
- PR 合并 → 更新 Release Notes

### 第四原则：职责分离

> 各智能体在职责范围内自主决策，不得越权。

| 模块 | 负责智能体 | 决策权限 |
|------|----------|---------|
| 需求文档 | 需求分析师 | 完全自主 |
| 架构设计 | 技术经理 | 完全自主 |
| 前端代码 | 前端开发 | 自主实现，PR 审核 |
| 后端代码 | 后端开发 | 自主实现，PR 审核 |
| 测试代码 | 测试员 | 完全自主 |
| 部署配置 | 运维 | 自主实施，事后报告 |

### 第五原则：持续改进

> 每轮任务结束后必须总结经验，优化流程。

- 任务完成 → 生成本次任务的经验总结
- 发现问题 → 创建改进建议 ticket
- 优秀实践 → 归档到长期记忆

## 通信协议

### 消息队列

```
shared/message_queue/
├── inbox/          # 入队消息
│   ├── msg_001.json
│   └── msg_002.json
├── outbox/         # 出队消息
│   ├── msg_003.json
│   └── msg_004.json
└── broadcast/      # 广播消息
    └── announcement_001.json
```

### 消息格式

```json
{
  "id": "msg_20240115_001",
  "timestamp": "2024-01-15T10:30:00Z",
  "from": "agent_frontend_dev_001",
  "to": "agent_tech_manager",
  "type": "pr_review_request",
  "priority": "normal",
  "payload": {
    "ticket_id": "FEAT-123",
    "pr_number": 42,
    "branch": "feature/feat-123-user-auth",
    "summary": "实现用户登录功能"
  }
}
```

### 消息类型

| 类型 | 用途 | 格式 |
|------|------|------|
| `task_assigned` | 分配任务 | JSON |
| `task_claimed` | 领取任务 | JSON |
| `pr_review_request` | 请求 PR 审核 | JSON |
| `help_request` | 请求帮助 | JSON |
| `status_update` | 状态更新 | JSON |
| `blocker_alert` | 阻塞告警 | JSON |

## 记忆系统

### 三级记忆架构

| 类型 | 存储位置 | 内容 | 保留策略 |
|------|---------|------|---------|
| **短期记忆** | `runtime/memory/short_term/` | 当前任务上下文、临时变量 | 任务结束清除 |
| **长期记忆** | `runtime/memory/long_term/` | 项目知识、技术决策、经验 | 永久保留 |
| **外部记忆** | `shared/confluence/memory/` | 文档化知识、最佳实践 | 永久保留，可检索 |

### 记忆内容示例

```
短期记忆 (任务执行中):
- 任务描述和验收标准
- 临时变量和中间结果
- 当前分支和文件路径

长期记忆 (永久保存):
- 项目技术栈和架构图
- 编码规范和最佳实践
- 历史决策记录和原因

外部记忆 (共享知识):
- Confluence 文档
- API 接口定义
- 数据库 Schema
```
